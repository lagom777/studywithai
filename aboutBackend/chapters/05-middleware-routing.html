Chapter 5 — Middleware Pipeline & Routing

미들웨어 파이프라인 시뮬레이터 포함

미들웨어란?

미들웨어는 요청(Request)과 응답(Response) 사이에서 실행되는 함수입니다. 로깅, 인증 확인, 본문 파싱, CORS 처리 등을 담당하며, next()를 호출해 다음 미들웨어로 제어를 넘깁니다. next()를 호출하지 않으면 요청 처리가 거기서 멈춥니다.

미들웨어 파이프라인 시뮬레이터

요청이 미들웨어 게이트를 하나씩 통과하는 과정을 관찰하세요. "차단 모드"에서는 Auth 게이트에서 요청이 거부됩니다.

📤 요청을 보내고 미들웨어 게이트 통과를 관찰하세요

Express 미들웨어 체인

요청 → Logger → CORS → Parser → Auth → Handler
const express = require('express'); const app = express(); // 1. 로깅 미들웨어 app.use((req, res, next) => { console.log(`${req.method} ${req.url}`); next(); }); // 2. CORS app.use(cors({ origin: 'https://myapp.com' })); // 3. Body 파싱 app.use(express.json()); // 4. 인증 (특정 경로만) app.use('/api/admin', authMiddleware); // 5. 라우트 핸들러 app.get('/api/users', (req, res) => { res.json({ users: [] }); });

미들웨어 종류

📝 애플리케이션 레벨

app.use()로 등록하며 모든 요청에 실행됩니다. 로깅, 파싱 등 전역 처리에 사용합니다.

🔀 라우터 레벨

express.Router()에 연결하여 특정 경로 그룹에만 적용합니다.

🚨 에러 핸들링 미들웨어

인자를 4개(err, req, res, next) 받는 특수한 미들웨어입니다. 에러 발생 시 일반 미들웨어를 건너뛰고 여기로 점프합니다.

라우터 분리

// routes/users.js const router = express.Router(); router.get('/', listUsers); router.post('/', createUser); router.put('/:id', updateUser); router.delete('/:id', deleteUser); module.exports = router; // app.js app.use('/api/users', require('./routes/users')); app.use('/api/posts', require('./routes/posts'));
app.use((err, req, res, next) => { console.error(err.stack); res.status(err.statusCode || 500).json({ error: err.message, ...(process.env.NODE_ENV === 'development' && { stack: err.stack }) }); });

💡 탐구 과제

차단 모드를 켜고 끄면서 Auth 미들웨어의 역할을 확인해보세요. next()를 호출하지 않으면 어떻게 되는지 시각적으로 이해할 수 있습니다.

하 · 입문

한 줄로 이해하기

미들웨어는 요청과 응답 사이를 가로지르는 "통과 게이트"이고, 라우팅은 "어떤 URL을 어떤 함수에 연결할지" 정하는 지도입니다.

왜 중요한가?

로그인 검사, 로그 기록, 요청 본문 파싱, 압축, CORS, 에러 처리는 모든 API에 공통으로 필요합니다. 미들웨어로 이를 한 곳에 모아두면, 컨트롤러는 비즈니스 로직에만 집중할 수 있어 코드가 깔끔해지고 보안 누락이 사라집니다.

🌱 핵심 용어

  • 미들웨어: (req, res, next) 시그니처의 처리 함수
  • next(): 다음 미들웨어로 흐름을 넘기는 호출
  • 라우터: 메서드 + URL 패턴 → 핸들러 매핑
  • 경로 매개변수: /users/:id의 :id 부분
  • 에러 미들웨어: 4-인자 (err, req, res, next)

핵심 개념 풀어쓰기

공항 보안 검색대를 떠올려보세요. 손님(요청)이 게이트에 들어가려면 신분증 검사 → X-ray → 짐 검색을 순서대로 통과합니다. 어느 단계에서든 문제가 있으면 멈춰서 거절(응답)합니다. 통과한 손님만 다음 단계로 갑니다. 미들웨어 체인도 같은 구조입니다. 라우팅은 그 손님이 결국 어느 게이트(핸들러)로 갈지 결정하는 일입니다.

💡 학습 TIP: 미들웨어의 등록 "순서"가 곧 실행 순서입니다. cors → json 파서 → auth → router → 에러 핸들러 순으로 배치하는 게 일반적입니다.

한 페이지 요약

  • 미들웨어는 (req, res, next) 함수
  • next()를 호출해야 다음으로 진행
  • 응답을 보내거나 next(err)로 에러 전파
  • 등록 순서 = 실행 순서
  • 라우터로 URL 기반 분기 처리
중 · 응용

이론 깊이 들어가기

Express 미들웨어는 함수 체인 패턴의 단순한 구현이지만, 비동기 흐름·에러 전파·조건부 분기를 다 표현해야 해서 설계가 까다롭습니다. app.use는 전역, router.use는 라우터 범위, app.METHOD는 단일 라우트에 적용됩니다. 라우트 매칭은 path-to-regexp가 담당하며 동적 매개변수(:id), 와일드카드, 옵셔널 세그먼트를 지원합니다. 비동기 핸들러에서 throw된 예외는 Express 4.x까지는 next로 자동 전파되지 않아 express-async-errors 같은 래퍼가 필요했고, 5.x부터 표준에 포함되었습니다. CORS, helmet, compression, morgan은 거의 모든 프로젝트의 기본 셋업입니다.

핵심 코드 패턴

// 시간 측정 미들웨어
app.use((req, res, next) => {
  const t0 = Date.now();
  res.on('finish', () => console.log(req.method, req.path, Date.now()-t0,'ms'));
  next();
});

// 에러 미들웨어(반드시 마지막)
app.use((err, req, res, next) => {
  res.status(err.status||500).json({error:err.message});
});

requireRole('admin') 같은 팩토리 미들웨어를 만들어 라우트에 단계적으로 끼웁니다. 라우터 단위 또는 메서드 단위로 자유롭게 조립할 수 있습니다.

📐 예제 2: 라우터 모듈화

routes/v1/index.js에서 모든 v1 라우터를 모으고, app.use('/api/v1', v1Router)로 마운트합니다. 버전 분기와 도메인 분리를 동시에 깔끔하게 해결합니다.

응용 사례

SaaS의 멀티 테넌트 API는 미들웨어로 호스트/서브도메인을 파싱해 req.tenant를 채우고, 이후 모든 쿼리에서 자동으로 테넌트 필터를 적용합니다. 카카오·네이버 API 게이트웨이는 인증·로깅·레이트리밋·서킷브레이커를 모두 미들웨어 체인으로 표현합니다.

⚠️ 자주 하는 실수: 비동기 미들웨어에서 next()를 호출하기 전에 응답을 보내거나, 에러를 throw만 하고 next(err)로 전파하지 않아 요청이 영원히 매달리는 경우가 흔합니다.
상 · 심화

전문가 레벨 이해

대규모 API에서는 미들웨어 체인 자체가 성능 핫스팟이 됩니다. body 파싱·JSON 직렬화·로깅의 누적 비용이 ms 단위로 쌓이며, 잘못된 미들웨어 한 줄이 RPS를 절반으로 떨어뜨립니다. Fastify는 schema 컴파일 기반의 직렬화로 이를 회피합니다. Koa·NestJS는 미들웨어를 "양파(onion)" 구조로 표현해 응답 직전 후처리를 자연스럽게 표현합니다. 라우팅의 라디스 트리(prefix tree) 구현은 Trie/Radix 기반이며, 매칭 성능이 곧 처리량에 직결됩니다. API 게이트웨이(Kong, Envoy)에서는 미들웨어 개념이 "필터 체인"으로 확장되어 mTLS, JWT 검증, WAF, 추적, 캐싱을 한 곳에서 처리합니다.

최신 연구·동향

Cloudflare Workers, Vercel Edge는 미들웨어를 엣지에서 실행해 인증·AB 테스트·캐싱을 origin 호출 없이 끝냅니다. OpenTelemetry는 자동 계측 미들웨어로 모든 요청에 분산 추적을 부여하고, eBPF 기반 관측은 코드 변경 없이 미들웨어 수준 메트릭을 수집합니다.
