Chapter 6 — File Processing & Data Streaming

데이터 스트림 흐름 시각화 포함

스트리밍이란?

스트리밍은 데이터를 한꺼번에 메모리에 올리지 않고 작은 청크(chunk) 단위로 처리하는 방식입니다. 1GB 파일을 읽을 때 전체를 메모리에 올리면 서버가 터질 수 있지만, 스트림을 쓰면 64KB씩 나누어 처리할 수 있습니다.

데이터 스트림 시각화

청크 크기와 전송 속도를 조절하면서 소스에서 목적지로 데이터가 스트리밍되는 과정을 관찰하세요.

⟳ 슬라이더로 청크 크기와 전송 속도를 조절하세요

Node.js 스트림 타입

📖 Readable Stream

데이터를 읽어오는 스트림입니다. fs.createReadStream(), HTTP 요청 등이 대표적입니다.

✏️ Writable Stream

데이터를 쓰는 스트림입니다. fs.createWriteStream(), HTTP 응답 등이 해당됩니다.

🔄 Transform Stream

데이터를 변환하는 스트림입니다. 압축(zlib), 암호화 등이 Transform으로 구현됩니다.

⚠️ 백프레셔 (Backpressure)

쓰는 속도보다 읽는 속도가 빠를 때 발생합니다. pipe()가 자동으로 흐름을 조절하여 메모리 폭발을 방지합니다.

스트림 사용 예시

const fs = require('fs'); // 스트림으로 대용량 파일 복사 const readStream = fs.createReadStream('bigfile.dat'); const writeStream = fs.createWriteStream('copy.dat'); readStream.pipe(writeStream); readStream.on('data', (chunk) => { console.log(`청크 수신: ${chunk.length} bytes`); }); readStream.on('end', () => { console.log('전송 완료!'); });
app.get('/api/download/:file', (req, res) => { const filePath = path.join(uploadDir, req.params.file); const stat = fs.statSync(filePath); res.writeHead(200, { 'Content-Type': 'application/octet-stream', 'Content-Length': stat.size, 'Content-Disposition': `attachment; filename="${req.params.file}"` }); fs.createReadStream(filePath).pipe(res); });

파일 업로드 (Multer)

const multer = require('multer'); const upload = multer({ dest: 'uploads/', limits: { fileSize: 10 * 1024 * 1024 }, // 10MB fileFilter: (req, file, cb) => { const allowed = ['image/png', 'image/jpeg', 'application/pdf']; cb(null, allowed.includes(file.mimetype)); } }); app.post('/api/upload', upload.single('file'), (req, res) => { res.json({ filename: req.file.originalname, size: req.file.size, path: req.file.path }); });

💡 탐구 과제

청크 크기를 최소와 최대로 설정해보세요. 작은 청크는 메모리 효율이 좋지만 오버헤드가 크고, 큰 청크는 빠르지만 메모리 사용이 늡니다.

하 · 입문

한 줄로 이해하기

스트리밍은 "큰 파일이나 데이터를 한 번에 다 메모리에 올리지 않고, 조각조각 흘려보내며 처리"하는 기술입니다.

왜 중요한가?

유튜브가 8K 영상을 끊김 없이 보여주는 비결, 디스코드가 음성을 실시간으로 전송하는 방식, 인스타그램이 수십 MB 사진 업로드를 처리하는 메커니즘이 모두 스트리밍입니다. 메모리가 4GB뿐인 서버에서 100GB 파일을 다루는 일이 가능한 이유입니다.

🌱 핵심 용어

  • 스트림: 시간에 따라 흐르는 데이터 시퀀스
  • 청크(Chunk): 스트림을 구성하는 작은 조각
  • Pipe: 한 스트림의 출력을 다른 스트림의 입력으로 연결
  • Backpressure: 빠른 생산자/느린 소비자 사이의 흐름 제어
  • multipart/form-data: 파일 업로드용 인코딩

핵심 개념 풀어쓰기

10GB 동영상을 "다 불러온 뒤 보내" 방식으로 서비스하면 서버 메모리는 한 명만 받아도 꽉 찹니다. 대신 1MB씩 잘라 보내면 메모리는 1MB만 쓰고 수많은 사용자에게 동시에 서비스할 수 있습니다. 사용자도 다 받기를 기다릴 필요 없이 첫 청크가 도착하자마자 재생을 시작할 수 있습니다. 이게 "지연 시작(Time to First Byte)"이 빨라지는 이유입니다.

💡 학습 TIP: Node.js에서 fs.readFile vs fs.createReadStream의 메모리 사용량을 큰 파일로 비교해보세요. 차이가 100배 이상 납니다.

한 페이지 요약

  • 스트리밍 = 메모리 효율 + 빠른 응답 시작
  • 스트림 4종: Readable / Writable / Duplex / Transform
  • pipe로 스트림을 안전하게 연결
  • 업로드는 multer 같은 멀티파트 파서로 처리
  • 다운로드는 Content-Disposition과 Range로 제어
중 · 응용

이론 깊이 들어가기

Node.js 스트림은 EventEmitter 기반으로 'data', 'end', 'error', 'drain' 이벤트를 발생시킵니다. highWaterMark 옵션이 내부 버퍼 크기를 결정하며, 기본 16KB입니다. write()가 false를 반환하면 버퍼가 가득 찼다는 신호이고, drain 이벤트를 기다려야 합니다. pipe와 pipeline은 이런 backpressure를 자동으로 처리해줍니다. Transform 스트림은 입력 청크를 변환해 새 청크를 내보내며 압축(gzip), 암호화(crypto), 파싱(JSON Stream)에 쓰입니다. HTTP Range 요청을 처리하면 동영상의 임의 시점부터 전송이 가능해져 시킹(seeking)이 동작합니다.

핵심 코드 패턴

const {pipeline} = require('stream/promises');
const fs = require('fs');
const zlib = require('zlib');
await pipeline(
  fs.createReadStream('input.log'),
  zlib.createGzip(),
  fs.createWriteStream('input.log.gz')
);

응용 사례

SaaS의 CSV 익스포트 기능은 DB 쿼리 결과를 한 행씩 Transform으로 CSV 직렬화하면서 HTTP 응답으로 흘려보냅니다. 메모리는 일정 수준을 넘지 않고, 사용자는 다운로드가 즉시 시작되는 경험을 얻습니다. 음성/영상 채팅은 WebRTC가 미디어를 SCTP 스트림으로 흘리고, HLS/DASH는 영상을 청크 단위 manifest로 쪼개 HTTP로 흘립니다.

⚠️ 자주 하는 실수: stream.on('data', ...) + 동기 처리 패턴은 backpressure를 무시하기 쉽습니다. 가능한 한 pipeline()이나 async iteration(for await)으로 작성하세요.
상 · 심화

전문가 레벨 이해

현대 스트리밍 백엔드는 단일 서버 스트림 처리에서 분산 스트림 처리로 옮겨가고 있습니다. Kafka, Pulsar, Kinesis는 영속화된 분산 로그로서 스트림의 정의 자체를 확장했습니다. 백엔드는 들어오는 미디어 청크를 객체 스토리지(S3 멀티파트)에 멀티파트로 직접 업로드하면서 동시에 메타데이터 파이프라인으로 전송합니다. Range 응답 외에 HTTP/2 server push와 SSE가 부분적으로 쓰이지만, 실시간 양방향은 WebSocket·WebTransport가 표준입니다. ffmpeg 파이프라인으로 트랜스코딩을 GPU 가속하며, HLS의 fMP4 세그먼트, LL-HLS의 partial segment로 글래스 투 글래스 지연을 2초 이내로 줄입니다. CDN의 시그니처드 URL로 핫링킹을 막고 만료 시간을 강제합니다.

최신 연구·동향

WebTransport는 QUIC 기반으로 신뢰성과 비신뢰성 데이터그램을 함께 제공해 게임·실시간 협업의 새 인프라가 되고 있습니다. MoQ(Media over QUIC)는 IETF에서 표준화 중인 차세대 미디어 전송 프로토콜입니다. AV1·VVC 코덱과 함께 같은 화질을 더 작은 대역폭으로 전달하는 방향이 빠르게 진행 중입니다.
