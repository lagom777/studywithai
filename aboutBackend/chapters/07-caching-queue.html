Chapter 7 — Caching Strategies & Message Queues

캐시 Hit/Miss 시각화 포함

캐싱이란?

캐싱은 자주 요청되는 데이터를 빠른 저장소에 미리 저장해두는 전략입니다. 데이터베이스 조회는 수십~수백 ms가 걸리지만, Redis 같은 인메모리 캐시는 1ms 이내에 응답할 수 있습니다.

캐시 Hit / Miss 시뮬레이터

"Cache Hit" 버튼은 캐시에서 바로 데이터를 가져오는 빠른 경로를, "Cache Miss" 버튼은 DB까지 가야 하는 느린 경로를 보여줍니다.

🎯 Cache Hit과 Miss의 경로 차이를 비교하세요

캐싱 전략

📖 Cache-Aside (Lazy Loading)

요청 시 캐시를 먼저 확인하고, 없으면(Miss) DB에서 가져와 캐시에 저장합니다. 가장 일반적인 패턴입니다.

✏️ Write-Through

데이터를 쓸 때 캐시와 DB에 동시에 기록합니다. 일관성은 좋지만 쓰기 지연이 있습니다.

📝 Write-Back

캐시에만 먼저 쓰고, 나중에 비동기로 DB에 반영합니다. 빠르지만 캐시 장애 시 데이터 유실 위험이 있습니다.

const redis = require('redis'); const client = redis.createClient(); async function getUser(id) { // 1. 캐시 확인 const cached = await client.get(`user:${id}`); if (cached) { console.log('Cache HIT'); return JSON.parse(cached); } // 2. Cache Miss → DB 조회 console.log('Cache MISS'); const user = await db.users.findById(id); // 3. 캐시에 저장 (TTL 1시간) await client.setEx(`user:${id}`, 3600, JSON.stringify(user)); return user; }

메시지 큐

메시지 큐는 서비스 간 비동기 통신을 가능하게 합니다. 생산자(Producer)가 메시지를 큐에 넣으면, 소비자(Consumer)가 자신의 속도로 처리합니다.

Producer → Message Queue → Consumer A   Consumer B

🐰 RabbitMQ

AMQP 프로토콜 기반 메시지 브로커. Exchange, Queue, Binding으로 유연한 라우팅을 지원합니다.

📊 Apache Kafka

분산 이벤트 스트리밍 플랫폼. 높은 처리량과 데이터 영속성을 제공합니다. 로그 기반이라 메시지를 재처리할 수 있습니다.

// Publisher (이벤트 발행) const channel = await connection.createChannel(); await channel.assertQueue('order-events'); channel.sendToQueue('order-events', Buffer.from(JSON.stringify({ type: 'ORDER_CREATED', orderId: 'ORD-001', userId: 'user123' })) ); // Consumer (이벤트 소비) channel.consume('order-events', (msg) => { const event = JSON.parse(msg.content.toString()); console.log('주문 이벤트 수신:', event.type); // 이메일 전송, 재고 감소 등 처리 channel.ack(msg); });

💡 탐구 과제

Cache Hit과 Miss를 번갈아 클릭하면서 응답 경로와 속도 차이를 관찰하세요. 실제 서비스에서 캐시 적중률 95%만 달성해도 DB 부하가 크게 줄어듭니다.

한 줄로 이해하기

캐싱은 "자주 쓰는 결과를 빠른 곳에 미리 두기", 메시지 큐는 "지금 당장 처리하지 않아도 되는 일을 줄 세우기" 기술입니다.

왜 중요한가?

네이버 검색이 0.2초 안에 결과를 보여주는 비결, 쿠팡 주문 후 알림이 몇 초 뒤에 오는 이유, 인스타 게시물의 좋아요 카운트가 실시간으로 보이는 메커니즘이 모두 이 두 가지 위에 서 있습니다. 캐싱은 응답 속도를, 큐는 안정성과 확장성을 끌어올립니다.

🌱 핵심 용어

  • Cache Hit/Miss: 캐시에 있음/없음
  • TTL: 캐시 유효 시간
  • Redis: 가장 널리 쓰는 인메모리 캐시·큐
  • 큐(Queue): 먼저 들어온 메시지가 먼저 처리
  • Pub/Sub: 발행-구독 비동기 통신

핵심 개념 풀어쓰기

편의점이 매번 본사 창고에서 음료를 가져오면 손님은 한참을 기다려야 합니다. 매장 냉장고(캐시)에 자주 팔리는 음료를 미리 두면 즉시 판매가 가능합니다. 다만 신상품 입고나 가격 변경 같은 "데이터 변화"는 냉장고에도 반영해야 합니다. 메시지 큐는 손님이 결제한 뒤 영수증 이메일을 굳이 그 자리에서 보내지 않고, 나중에 보내도록 "주문서"를 작업 큐에 넣어 두는 식입니다.

💡 학습 TIP: Redis CLI로 SET, GET, EXPIRE, INCR을 직접 두드려보세요. 5분이면 캐시 감각이 잡힙니다.

한 페이지 요약

  • 캐싱 = 응답 속도 + DB 부하 절감
  • TTL과 무효화 전략이 캐시 설계의 핵심
  • 큐 = 비동기 + 부하 분산 + 재시도
  • 대표 도구: Redis, Memcached, RabbitMQ, Kafka, SQS
  • Eventual Consistency를 받아들여야 함
중 · 응용

이론 깊이 들어가기

캐시 적용 위치는 브라우저, CDN, 리버스 프록시(Varnish, Nginx), 애플리케이션 메모리, 분산 캐시(Redis), DB 결과 캐시까지 여러 층입니다. 데이터 일관성 측면에서 Cache-aside, Read-through, Write-through, Write-behind 패턴이 있고 각자 트레이드오프가 다릅니다. TTL이 짧으면 데이터 정합성이 좋지만 적중률이 떨어지고, 길면 반대입니다. 캐시 무효화는 정확한 키 삭제, 태그 기반 무효화, 버전 키 회전 등 여러 전략이 있습니다. 메시지 큐는 메시지 순서 보장(FIFO), 정확히 한 번/최소 한 번 처리, DLQ(Dead Letter Queue), 가시성 타임아웃 같은 의미를 정확히 이해해야 합니다.

핵심 코드 패턴

// Cache-aside 패턴
async function getUser(id) {
  const k = `user:${id}`;
  const cached = await redis.get(k);
  if (cached) return JSON.parse(cached);
  const u = await db.users.findById(id);
  await redis.setex(k, 300, JSON.stringify(u));
  return u;
}

대표 예제

📐 예제 1: Stampede 방어

인기 키 TTL 만료 시 한꺼번에 DB로 가는 thundering herd를 막기 위해 분산 락 또는 stale-while-revalidate를 사용합니다.

📐 예제 2: 비동기 이메일 발송

주문 완료 시 메일 전송을 큐로 위임하고, 워커가 실패 시 지수 백오프로 재시도, 최종 실패는 DLQ로 보내 운영자가 검수합니다.

응용 사례

토스의 송금 알림은 큐 기반으로 SMS·푸시·이메일을 동시 분기 처리합니다. 넷플릭스의 추천 API는 캐시 적중률이 곧 비용입니다. 인기 콘텐츠는 분 단위로 갱신, 비인기 콘텐츠는 시간 단위 TTL로 차등 설계합니다. 카카오톡 메시지 fan-out은 단체방의 수십만 구독자에 메시지를 큐로 분산 푸시합니다.

⚠️ 자주 하는 실수: 큐 소비자가 처리 도중 실패했을 때 ack를 잘못 보내면 메시지가 사라집니다. ack는 처리 성공 시점, nack/retry는 실패 시점에 명확히 분리하세요.
상 · 심화

전문가 레벨 이해

대규모 시스템의 캐싱은 다층 캐시 일관성 관리가 본질입니다. CDN(엣지) → 리버스 프록시 → 앱 메모리 → Redis → DB까지 각 층의 무효화 신호가 안전하게 전파되어야 합니다. Redis Cluster는 슬롯 기반 샤딩으로 수평 확장하고, Sentinel/Cluster로 고가용성을 확보합니다. 동시성 제어는 Lua 스크립트나 Redis WATCH/MULTI로 원자적으로 수행합니다. 메시지 큐 영역에서 Kafka는 파티션 + 컨슈머 그룹으로 순서 보장과 병렬성을 동시에 제공합니다. Exactly-once는 idempotency key + transactional outbox + idempotent consumer 조합으로 실효적 정확성을 만듭니다. 백프레셔, 카오스 테스트, 큐 깊이 알람이 운영의 핵심입니다.

최신 연구·동향

Redis 8/Valkey, KeyDB, Dragonfly 같은 차세대 인메모리 DB가 멀티스레드와 io_uring으로 처리량을 끌어올리고 있습니다. Kafka의 KRaft 모드는 Zookeeper 의존을 제거했고, Tiered Storage로 무한 보존을 저렴하게 합니다. 이벤트 스트리밍 + 머터리얼라이즈드 뷰(Materialize, RisingWave)는 캐시와 큐의 경계를 흐립니다.
