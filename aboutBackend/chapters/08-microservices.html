분산 서비스 통신 시뮬레이터 포함

모놀리식 vs 마이크로서비스

모놀리식 아키텍처는 모든 기능이 하나의 코드베이스에 있습니다. 간단하지만 규모가 커지면 배포, 확장, 유지보수가 어려워집니다. 마이크로서비스는 각 기능을 독립된 서비스로 분리하여 개별적으로 개발, 배포, 확장할 수 있게 합니다.

마이크로서비스 통신 시뮬레이터

"주문 처리" 버튼을 누르면 API Gateway를 통해 여러 서비스가 순차적으로 호출되는 과정을 관찰할 수 있습니다. "서킷 브레이커" 모드에서는 장애가 있는 서비스를 우회하는 패턴을 확인하세요.

🛒 주문 처리를 시작하고 서비스 간 통신을 관찰하세요

핵심 패턴

🚪 API Gateway

모든 외부 요청의 단일 진입점입니다. 인증, 라우팅, 속도 제한, 로깅을 중앙에서 처리합니다. Kong, AWS API Gateway, Nginx 등이 대표적입니다.

🔍 서비스 디스커버리

서비스 인스턴스의 위치(IP/포트)를 동적으로 찾습니다. Consul, etcd, Kubernetes DNS 등이 사용됩니다.

⚡ 서킷 브레이커

장애가 발생한 서비스 호출을 빠르게 실패시켜 연쇄 장애를 방지합니다. Closed → Open → Half-Open 세 상태로 동작합니다.

📜 사가 패턴 (Saga)

분산 트랜잭션을 로컬 트랜잭션의 연속으로 처리합니다. 실패 시 보상 트랜잭션(compensation)으로 롤백합니다.

서비스 간 통신

Client → API Gateway → Auth → Order → Payment → Notify
// gateway/index.js const { createProxyMiddleware } = require('http-proxy-middleware'); app.use('/api/auth', createProxyMiddleware({ target: 'http://auth-service:4001', pathRewrite: { '^/api/auth': '' } })); app.use('/api/orders', createProxyMiddleware({ target: 'http://order-service:4002', pathRewrite: { '^/api/orders': '' } })); app.use('/api/payments', createProxyMiddleware({ target: 'http://payment-service:4003', pathRewrite: { '^/api/payments': '' } }));
const CircuitBreaker = require('opossum'); const options = { timeout: 3000, // 3초 타임아웃 errorThreshold: 50, // 50% 실패 시 Open resetTimeout: 10000 // 10초 후 Half-Open }; const breaker = new CircuitBreaker(callPaymentService, options); breaker.on('open', () => console.log('서킷 OPEN — 장애 감지')); breaker.on('halfOpen', () => console.log('서킷 HALF-OPEN — 재시도 중')); breaker.on('close', () => console.log('서킷 CLOSED — 정상 복구')); // 사용 const result = await breaker.fire(orderId);

하 · 입문

한 줄로 이해하기

마이크로서비스는 "큰 앱을 하나로 만들지 않고, 작은 서비스 여러 개로 쪼개 각자 따로 배포·확장하는 구조"입니다.

왜 중요한가?

넷플릭스, 우버, 카카오, 토스 같은 대규모 서비스는 수십~수백 개의 마이크로서비스로 동작합니다. 결제팀, 추천팀, 알림팀이 서로 발목 잡지 않고 동시에 배포할 수 있게 하려면 필연적으로 도달하는 구조입니다. 다만 작은 팀에는 오히려 독이 될 수도 있어 트레이드오프를 이해해야 합니다.

🌱 핵심 용어

  • 모놀리식: 한 덩어리 애플리케이션
  • API 게이트웨이: 외부 진입점, 라우팅·인증 담당
  • 서비스 디스커버리: 어디에 어떤 서비스가 떠 있는지 찾기
  • 서킷 브레이커: 장애 전파 차단
  • 분산 추적: 요청 하나가 어떤 서비스들을 거쳤는지 추적

핵심 개념 풀어쓰기

한 가게가 음식 만들기·서빙·결제·청소를 한 사람이 다 하면 한 가지가 막혀도 가게가 멈춥니다. 큰 식당에서는 주방, 홀, 카운터, 청소가 분리되어 한 영역에 문제가 생겨도 다른 일은 계속됩니다. 다만 의사소통과 조율 비용이 듭니다. 마이크로서비스도 같습니다. 자유와 확장성을 얻는 대신 운영·관측·통신의 복잡도가 폭증합니다.

💡 학습 TIP: 모놀리식이 정말 한계인지부터 점검하세요. 팀이 5명 이하인데 마이크로서비스를 도입하면 80%는 후회합니다. "분산 모놀리스"라는 안티패턴이 가장 큰 함정입니다.

한 페이지 요약

  • 독립 배포 · 독립 데이터 · 독립 스케일
  • HTTP, gRPC, 메시지 큐로 통신
  • 관측성(로그·메트릭·트레이스)이 필수
  • 실패는 일상 — 재시도·서킷 브레이커가 기본
  • 분산 트랜잭션 대신 Saga 패턴
중 · 응용

이론 깊이 들어가기

서비스 경계는 비즈니스 도메인의 Bounded Context를 기준으로 나누는 것이 정석입니다(DDD). 한 서비스는 한 가지 책임만 갖고, 자신만의 DB를 소유하며, 다른 서비스의 DB에 직접 접근하지 않습니다. 동기 통신(HTTP/gRPC)은 일관성을 다루기 쉬우나 가용성이 결합되고, 비동기 통신(이벤트)은 결합도를 낮추지만 일관성을 별도로 설계해야 합니다. 분산 트랜잭션은 2PC 대신 Saga(Choreography 또는 Orchestration)와 보상 트랜잭션으로 표현합니다. 인증은 보통 API 게이트웨이에서 일괄 처리하고 내부 통신은 mTLS와 JWT로 보강합니다.

응용 사례

토스 송금은 송금/거래/한도/이상거래 탐지/알림 서비스가 협력해 한 건의 송금을 완성합니다. 카카오 모빌리티는 위치, 매칭, 결제, 정산 서비스를 분리해 운영합니다. 모두 분산 추적(OpenTelemetry)으로 한 송금 요청의 전체 경로를 한 화면에서 들여다봅니다.

⚠️ 자주 하는 실수: 서비스를 너무 잘게 쪼개면 한 화면을 그리는 데 수십 번 RPC가 필요해 레이턴시·운영 비용이 폭증합니다. "독립 배포가 가능한 최소 단위"가 좋은 기준입니다.
상 · 심화

전문가 레벨 이해

현대 마이크로서비스 운영의 본질은 "변하는 토폴로지에서 안정성과 관측성을 어떻게 자동화하느냐"입니다. Kubernetes는 서비스 디스커버리, 자가 치유, 롤링 업데이트를 표준화했고, Istio·Linkerd 같은 서비스 메시는 mTLS·트래픽 정책·관측성을 사이드카로 주입합니다. eBPF 기반 Cilium은 사이드카 없이 같은 일을 더 적은 오버헤드로 합니다. 분산 시스템의 한계(CAP, FLP, PACELC)는 설계 전반에 직접 영향을 주며, 데이터 영역에서는 CDC + 스트리밍 ETL로 도메인 경계를 가로지르는 분석/검색 인덱스를 일관되게 만듭니다. 카나리·블루-그린·feature flag로 배포 위험을 낮추고, SLO 기반 운영(에러 버짓)으로 변경 속도와 안정성의 균형을 잡습니다.

최신 연구·동향

"분산 모놀리스" 경험 누적 후 "모듈러 모놀리스 → 필요할 때 추출"이라는 역방향 흐름이 주류 의견으로 자리잡았습니다. Cell-based 아키텍처(AWS)는 장애 격리 단위를 셀로 정의해 폭발 반경(blast radius)을 좁힙니다. 서버리스 마이크로서비스는 Lambda/Workers + 이벤트 브리지로 인프라 운영 부담을 줄입니다.
