하 · 입문

한 줄로 이해하기

인증은 "당신이 누구인지" 확인하는 절차고, 인가는 "당신이 무엇을 할 수 있는지" 결정하는 절차입니다.

왜 배워야 하나?

네이버 로그인을 누르면 본인 확인이 일어나고, 그 후 메일이나 카페에 접근할 수 있습니다. 회사 인트라넷에서는 같은 직원이라도 인사팀만 급여 정보를 볼 수 있습니다. 인터넷 뱅킹에서는 송금 시 추가 인증을 요구합니다. 인증과 보안은 모든 서비스의 신뢰 기반이며, 잘못 설계하면 개인정보 유출과 금전 피해로 이어집니다. 백엔드 개발자라면 안전한 로그인 흐름과 권한 검증을 구현할 수 있어야 합니다.

🌱 핵심 용어

  • 인증(Authentication): 신원 확인. 비밀번호, OTP 등.
  • 인가(Authorization): 권한 확인. 어떤 자원에 접근 가능한지.
  • 세션: 서버가 로그인 상태를 기억하는 방식.
  • 토큰: 클라이언트가 들고 다니는 인증 증표(JWT 등).

개념 풀어쓰기

인증을 콘서트 입장으로 비유해보세요. 입구에서 신분증과 티켓을 확인하는 것이 인증입니다. 일단 입장한 후, VIP 라운지나 백스테이지에 들어갈 수 있는지는 또 다른 문제로 인가에 해당합니다. 비밀번호는 가장 흔한 인증 수단이지만 단독으로는 약하기 때문에 OTP나 생체 인증 같은 두 번째 요소를 추가합니다(2FA). 한번 로그인한 후 매 요청마다 신분증을 다시 보여주기는 번거로우니 서버는 세션이나 토큰으로 "이미 확인된 사람"이라고 기억합니다. 토큰은 위조되지 않도록 서버 비밀키로 서명되어 있고, 시간이 지나면 만료되어 다시 인증을 요구합니다.

💡 학습 TIP: 비밀번호는 절대 평문으로 저장하지 말고 bcrypt나 argon2로 해싱하세요. 가장 흔한 보안 사고가 평문 저장입니다.

요약

  • 인증은 신원 확인, 인가는 권한 확인이다.
  • 비밀번호는 반드시 해시 저장한다.
  • 세션과 토큰은 로그인 상태를 유지하는 두 방식이다.
  • 2FA로 보안을 한 단계 강화한다.
  • 모든 요청은 인가 검사를 거쳐야 한다.
중 · 응용

이론 깊이

비밀번호 해싱은 단순 SHA-256이 아니라 bcrypt, scrypt, argon2 같은 키 신장 알고리즘을 사용해야 합니다. 이들은 의도적으로 느리게 만들어 무차별 대입을 어렵게 합니다. 세션 기반 인증은 서버 메모리나 Redis에 세션 데이터를 두고 클라이언트는 쿠키로 세션 ID만 들고 다닙니다. JWT는 서버가 상태를 저장하지 않아도 되는 stateless 방식으로 헤더-페이로드-서명 세 부분으로 구성됩니다. OAuth 2.0은 사용자 비밀번호를 노출하지 않고 제3자 서비스에 권한 위임을 가능하게 하며, OIDC는 OAuth 2.0 위에 신원 정보를 더한 인증 프로토콜입니다. CSRF는 사용자가 의도하지 않은 요청을 다른 사이트가 트리거하는 공격으로 SameSite 쿠키, CSRF 토큰으로 방어합니다. XSS는 입력 값에 스크립트가 삽입되어 실행되는 공격으로 출력 시 이스케이프가 핵심입니다. SQL 인젝션은 쿼리에 사용자 입력이 그대로 합쳐질 때 발생하며 prepared statement로 방어합니다.

코드 패턴

// JWT 발급/검증
const jwt = require('jsonwebtoken');
function login(user) {
  return jwt.sign({ sub: user.id, role: user.role }, process.env.JWT_SECRET, { expiresIn: '15m' });
}
function authMiddleware(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch { res.status(401).json({ error: 'Unauthorized' }); }
}

💻 예제 1: 리프레시 토큰

액세스 토큰은 짧게(15분), 리프레시 토큰은 길게(2주) 두고 액세스 토큰 만료 시 리프레시로 재발급합니다. 리프레시 토큰은 DB에 저장해 회수 가능하게 합니다.

💻 예제 2: RBAC vs ABAC

RBAC는 역할(admin, editor)에 권한을 묶고 ABAC는 속성(소유자, 부서, 시간)을 조합해 동적 정책을 만듭니다.

실무 응용

금융 앱은 OAuth 2.0 + OIDC + 단말 기기 등록 + 생체 인증을 결합해 다층 방어를 합니다. SaaS 제품은 SSO(SAML, OIDC)로 기업 고객의 IdP에 통합되어 별도 비밀번호 없이 로그인합니다. 게임 서비스는 토큰 위조 방지를 위해 서명 검증과 함께 시그너처가 있는 클라이언트 메시지를 사용합니다.

⚠️ 자주 하는 실수: JWT 페이로드에 비밀번호나 민감 데이터를 넣는 경우가 있습니다. JWT는 서명되었을 뿐 암호화된 것이 아니므로 누구나 읽을 수 있습니다.
상 · 심화

전문가 레벨

대규모 인증 시스템은 ID 토큰, 액세스 토큰, 리프레시 토큰의 라이프사이클과 회수(revocation)를 정밀하게 설계합니다. JWT의 stateless 장점과 회수의 어려움을 동시에 해결하기 위해 짧은 만료와 블랙리스트, 또는 토큰 인트로스펙션 엔드포인트를 함께 사용합니다. PASETO는 JWT의 알고리즘 혼용 취약점을 해결한 대안 표준입니다. mTLS는 서비스 간 양방향 인증서 인증으로 마이크로서비스 환경에서 서비스 메시(SPIFFE/SPIRE)를 통해 자동 발급/회전됩니다. 영지식 증명(ZKP)은 비밀을 노출하지 않고 사실을 증명하는 기법으로 프라이버시 보존 인증에 활용됩니다. WebAuthn/FIDO2는 단말의 보안 칩에 키쌍을 두고 비밀번호 없는 인증을 제공하며 피싱 저항성이 매우 큽니다. OAuth 2.1은 implicit flow를 폐기하고 PKCE를 모든 공개 클라이언트에 강제합니다. DPoP는 토큰 탈취 시에도 다른 클라이언트가 사용할 수 없도록 키 바인딩을 강제합니다. 제로 트러스트 모델은 네트워크 경계가 아닌 모든 요청에 인증/인가를 적용하는 패러다임으로 BeyondCorp가 대표 사례입니다. 비밀 관리에는 Vault, AWS KMS, GCP Secret Manager를 사용하고 짧은 수명의 동적 자격 증명을 발급합니다.

고급 패턴

// PKCE 흐름(클라이언트 측 핵심)
const verifier = crypto.randomBytes(32).toString('base64url');
const challenge = crypto.createHash('sha256').update(verifier).digest('base64url');
// 1) 인가 요청 시 code_challenge 전송
// 2) 토큰 교환 시 code_verifier 동봉
// → 인가 코드 가로채기 공격 방어

최신 동향

Passkeys는 WebAuthn 위에 동기화 가능한 자격 증명을 추가해 기기 간 무비밀번호 로그인을 일상화하고 있습니다. Cloudflare Zero Trust, Tailscale 같은 ID-aware 프록시는 네트워크 자체를 인증 결과에 따라 동적으로 구성합니다. 컨테이너 환경에서는 SPIFFE 워크로드 ID로 IP 기반이 아닌 신원 기반 통신을 합니다. eBPF 기반 정책(Cilium Tetragon)은 런타임에 시스템 콜 단위로 보안 이벤트를 관찰해 침해 시도를 감지합니다.
