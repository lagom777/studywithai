Chapter 4 — Authentication, Authorization & Security

JWT 인증 흐름 시뮬레이터 포함

인증 vs 인가

인증(Authentication)은 "당신이 누구인가?"를 확인하는 과정이고, 인가(Authorization)는 "당신이 무엇을 할 수 있는가?"를 결정하는 과정입니다. 로그인이 인증이라면, 관리자 페이지 접근 권한 확인이 인가입니다.

JWT 인증 흐름 시뮬레이터

"로그인" 버튼을 눌러 클라이언트→서버→DB 인증 과정과 JWT 토큰 발급 흐름을 관찰하세요. 실패 모드에서는 인증 거부 과정을 볼 수 있습니다.

🔑 로그인 버튼을 눌러 JWT 인증 흐름을 관찰하세요

세션 기반 vs 토큰 기반

🍪 세션/쿠키 방식

서버가 세션 ID를 생성하고 쿠키에 저장합니다. 매 요청마다 쿠키를 통해 세션을 확인합니다. 서버에 상태를 저장하므로 수평 확장이 어렵습니다.

🎫 JWT (JSON Web Token) 방식

서버가 서명된 토큰을 발급하고 클라이언트가 보관합니다. 서버에 상태를 저장하지 않아(Stateless) 확장이 쉽습니다. 토큰 탈취에 대비한 만료 시간 설정이 중요합니다.

JWT 구조

Header . Payload . Signature
// Header: 알고리즘과 토큰 타입 { "alg": "HS256", "typ": "JWT" } // Payload: 사용자 정보 (Claims) { "sub": "user123", "name": "홍길동", "role": "admin", "iat": 1716000000, "exp": 1716086400 } // Signature: 위변조 방지 서명 HMACSHA256( base64(header) + "." + base64(payload), secret_key )

보안 위협과 방어

💉 SQL Injection

사용자 입력을 직접 쿼리에 넣으면 악의적인 SQL이 실행될 수 있습니다. Prepared Statement와 ORM을 사용하여 방어합니다.

🔗 XSS (Cross-Site Scripting)

악성 스크립트가 웹 페이지에 주입되어 사용자 정보를 탈취합니다. 입력값 이스케이프와 Content-Security-Policy 헤더로 방어합니다.

🔄 CSRF (Cross-Site Request Forgery)

인증된 사용자의 권한으로 의도하지 않은 요청을 보냅니다. CSRF 토큰과 SameSite 쿠키 설정으로 방어합니다.

const jwt = require('jsonwebtoken'); // 로그인 엔드포인트 app.post('/api/login', async (req, res) => { const { email, password } = req.body; const user = await User.findByEmail(email); if (!user || !await bcrypt.compare(password, user.hash)) { return res.status(401).json({ error: '인증 실패' }); } const token = jwt.sign( { sub: user.id, role: user.role }, process.env.JWT_SECRET, { expiresIn: '24h' } ); res.json({ token }); }); // 인증 미들웨어 function authMiddleware(req, res, next) { const token = req.headers.authorization?.split(' ')[1]; try { req.user = jwt.verify(token, process.env.JWT_SECRET); next(); } catch { res.status(401).json({ error: '유효하지 않은 토큰' }); } }

하 · 입문

한 줄로 이해하기

인증(Authentication)은 "당신이 누구인가"를 확인하고, 인가(Authorization)는 "그것을 할 권한이 있는가"를 확인합니다. 보안은 이 두 가지를 무너뜨리려는 모든 공격을 막는 일입니다.

왜 중요한가?

비밀번호 유출 한 번이 회사를 끝낼 수 있습니다. 2017년 Equifax 사고는 1억 4천만 명의 개인정보가 새어 나갔고, 토스나 카카오 같은 핀테크는 인증·인가 보안이 곧 신뢰의 본체입니다. 백엔드 개발자가 코드를 짤 때 가장 먼저 마음에 새겨야 할 영역입니다.

🌱 핵심 용어

  • 해시: 비밀번호를 되돌릴 수 없는 형태로 변환(bcrypt, Argon2)
  • 세션: 서버가 사용자 상태를 기억하는 방식
  • JWT: 클라이언트에 저장 가능한 서명된 토큰
  • HTTPS: TLS로 통신을 암호화한 HTTP
  • CSRF/XSS: 대표적인 웹 공격 유형

핵심 개념 풀어쓰기

회원가입에서 비밀번호를 그대로 저장하면 DB가 털릴 때 그대로 노출됩니다. 그래서 단방향 해시(bcrypt 등)에 salt와 cost를 더해 저장합니다. 로그인 후 서버는 사용자에게 "당신이 로그인했음"을 증명하는 무언가(세션 ID 또는 토큰)를 줍니다. 클라이언트는 매 요청에 이를 동봉해 신원을 증명합니다. 권한은 그 신원 위에 "어떤 행동을 허락할지" 정하는 별도의 레이어입니다.

💡 학습 TIP: OWASP Top 10을 한 번 처음부터 끝까지 읽어보세요. 사고의 90%는 거기 적힌 문제에서 옵니다.

한 페이지 요약

  • 인증 ≠ 인가, 둘 다 필요
  • 비밀번호는 절대 평문으로 저장 금지
  • HTTPS는 선택이 아닌 필수
  • 토큰/세션은 만료와 무효화 전략 필수
  • 최소 권한 원칙(Least Privilege)을 기본으로
중 · 응용

이론 깊이 들어가기

세션 방식은 서버가 sessionId ↔ 사용자 매핑을 보관하며, 무효화가 즉시 가능하지만 수평 확장 시 공유 저장소(Redis)가 필요합니다. JWT는 stateless라 확장에 유리하지만, 토큰을 즉시 무효화하기 어려워 짧은 만료 + refresh token 조합이 표준입니다. OAuth 2.0은 인증을 위임하는 프로토콜이고, OIDC는 그 위에 신원 정보를 얹은 표준입니다. 카카오/구글 로그인이 그 예입니다. 패스워드 해시는 bcrypt(cost 12 이상) 또는 Argon2id가 권장됩니다. CSRF는 SameSite 쿠키와 토큰으로, XSS는 출력 인코딩과 CSP 헤더로 방어합니다. SQL Injection은 prepared statement(파라미터 바인딩)로 사실상 완전히 막을 수 있습니다.

핵심 코드 패턴

// 안전한 비밀번호 비교
import bcrypt from 'bcrypt';
const hash = await bcrypt.hash(password, 12);
const ok   = await bcrypt.compare(input, hash);

// JWT 발급/검증
const token = jwt.sign({sub:user.id}, process.env.SECRET, {expiresIn:'15m'});

대표 예제

📐 예제 1: Refresh Token 회전

access 15분 + refresh 14일. refresh를 한 번 사용하면 새 access와 새 refresh를 발급하고 이전 refresh는 무효화합니다. 탈취 시 영향을 한 사이클로 제한합니다.

📐 예제 2: Rate Limit + 잠금

로그인 실패 5회면 IP/계정에 점진적 잠금. 분당 N회로 무차별 대입을 막고, 캡차로 보강합니다.

응용 사례

금융권은 보안 토큰이나 PKI 기반 인증서를 결합해 2FA를 강제합니다. SaaS는 SSO(SAML, OIDC)로 회사 단위 로그인을 제공해 보안과 편의성을 동시에 챙깁니다. 모바일 앱은 키체인/Keystore에 토큰을 저장하고 생체인증으로 잠금을 한 번 더 걸어 분실 시 위험을 줄입니다.

⚠️ 자주 하는 실수: JWT를 localStorage에 두면 XSS 한 방에 토큰이 새 나갑니다. HttpOnly 쿠키 + SameSite 조합이 일반적으로 더 안전합니다.
상 · 심화

전문가 레벨 이해

현대 보안은 "Zero Trust"가 표준 사고방식입니다. 내부망조차 기본 신뢰하지 않고 모든 요청을 인증·인가·검증합니다. mTLS로 서비스 간 통신을 인증하고, OPA(Open Policy Agent)·SpiceDB·Cedar로 인가 정책을 코드 외부에서 선언적으로 관리합니다. 비밀(secret) 관리는 HashiCorp Vault, AWS Secrets Manager, KMS 같은 전용 시스템으로 분리하고 정기 회전을 자동화합니다. WebAuthn/Passkey는 비밀번호 자체를 없애는 방향으로, 사용자 디바이스의 개인키로 challenge에 서명합니다. CSP·Subresource Integrity·Trusted Types로 브라우저 측 공격 표면을 좁힙니다. SBOM과 SCA로 의존성 취약점을 추적하고, secret scanning과 SAST/DAST를 CI에 통합합니다.

최신 연구·동향

Passkey가 Apple, Google, Microsoft 모두에서 표준이 되어 비밀번호의 종말을 앞당기고 있습니다. PQC(Post-Quantum Cryptography)는 NIST 표준화가 끝나면서 TLS·SSH 구현체들이 단계적으로 도입 중입니다. 컨피덴셜 컴퓨팅(SGX, SEV)은 메모리 상에서도 데이터를 암호화한 채 처리해 클라우드 운영자조차 평문에 접근하지 못하게 합니다.
