Chapter 2 — HTTP Protocol & RESTful API Design

HTTP 요청 파이프라인 시각화 포함

HTTP 프로토콜

HTTP(HyperText Transfer Protocol)는 웹에서 데이터를 주고받는 규칙입니다. 클라이언트가 요청(Request)을 보내면 서버가 응답(Response)을 돌려주는 단순한 구조이지만, 그 안에는 메서드, 헤더, 상태 코드 등 풍부한 의미가 담겨 있습니다.

HTTP 요청 파이프라인

HTTP 메서드를 선택하고 요청을 보내면 데이터가 파이프라인을 통해 흘러가는 모습을 관찰하세요.

📤 메서드를 선택하고 요청 흐름을 관찰하세요

HTTP 메서드

GET 조회

서버의 자원을 읽어옵니다. 요청 본문이 없으며, 멱등성(여러 번 호출해도 같은 결과)을 가집니다.

POST 생성

새로운 자원을 서버에 생성합니다. 요청 본문에 데이터를 담아 보냅니다.

PUT 수정   PATCH 부분 수정

PUT은 자원 전체를 대체하고, PATCH는 일부만 수정합니다. 둘 다 멱등성을 가집니다.

DELETE 삭제

서버의 자원을 삭제합니다.

HTTP 상태 코드

200 OK → 201 Created → 301 Moved → 404 Not Found → 500 Server Error

RESTful API 설계 원칙

REST(Representational State Transfer)는 자원 중심의 API 설계 패턴입니다. URL은 명사로 자원을 나타내고, HTTP 메서드가 동사(행위)를 담당합니다.

GET /api/users # 전체 사용자 조회 GET /api/users/:id # 특정 사용자 조회 POST /api/users # 새 사용자 생성 PUT /api/users/:id # 사용자 정보 수정 DELETE /api/users/:id # 사용자 삭제 GET /api/users/:id/posts # 사용자의 게시물 조회

하 · 입문

한 줄로 이해하기

HTTP는 "웹의 공용어"이고, API는 "두 프로그램이 약속한 대화 방식"입니다. 클라이언트와 서버는 HTTP로 약속된 형식의 요청·응답을 주고받습니다.

왜 중요한가?

모바일 앱 화면이 새로고침될 때, 카드 결제 승인이 일어날 때, 카카오 로그인을 누를 때 모두 뒤에서 HTTP API가 호출됩니다. API 설계가 깨지면 앱 화면이 멈추고, 외부 연동 파트너가 떠나며, 보안 사고가 납니다.

🌱 핵심 용어

  • HTTP 메서드: GET(조회), POST(생성), PUT/PATCH(수정), DELETE(삭제)
  • 상태 코드: 200 OK, 201 Created, 400/401/403/404, 500
  • 헤더: 메타데이터 — Content-Type, Authorization 등
  • 바디: 실제 데이터 — 주로 JSON
  • 엔드포인트: /api/users/42 처럼 자원을 가리키는 URL

핵심 개념 풀어쓰기

레스토랑의 메뉴판을 상상해보세요. 메뉴판(API 명세)이 있어야 손님(클라이언트)이 무엇을 어떤 형식으로 주문할 수 있는지 압니다. 잘 만든 메뉴판은 항목별 이름, 가격, 옵션이 명확합니다. API도 마찬가지로 자원(URL), 동작(메서드), 입력/출력 형식, 오류 코드가 일관되어야 좋은 API입니다. REST 스타일은 자원을 URL로 표현하고 동작을 HTTP 메서드로 표현하자는 약속입니다.

💡 학습 TIP: Postman이나 curl로 공개 API(예: GitHub API)에 직접 요청을 날려보세요. 상태 코드와 헤더를 눈으로 확인하면 감이 빨리 잡힙니다.

한 페이지 요약

  • 요청 = 메서드 + URL + 헤더 + 바디
  • 응답 = 상태 코드 + 헤더 + 바디
  • REST는 자원 + 메서드 조합으로 표현
  • JSON이 사실상 데이터 형식 표준
  • 좋은 API는 일관성·예측 가능성이 핵심
중 · 응용

이론 깊이 들어가기

HTTP는 무상태(stateless) 프로토콜입니다. 매 요청은 독립적이며 서버는 이전 요청을 기억하지 않습니다. 상태 유지가 필요하면 쿠키, 세션, 토큰을 사용합니다. HTTP/1.1은 keep-alive로 연결을 재사용하고, HTTP/2는 멀티플렉싱과 헤더 압축(HPACK)으로 효율을 높였습니다. HTTP/3는 QUIC 위에서 동작하여 패킷 손실 회복이 빠릅니다. REST 외에 GraphQL은 클라이언트가 필요한 필드만 골라 받을 수 있고, gRPC는 Protocol Buffers와 HTTP/2를 결합해 마이크로서비스 내부 통신에 강합니다. 멱등성(idempotency)은 같은 요청을 여러 번 보내도 결과가 같아야 하는 성질로, PUT/DELETE에 필수입니다.

핵심 응답 패턴

{
  "data": [...],
  "meta": { "page": 1, "total": 120 },
  "error": null
}

📐 예제 1: 페이지네이션

GET /api/posts?page=2&size=20 형태로 offset·limit 또는 cursor 기반 페이징을 제공합니다. 대용량은 cursor 방식이 성능에 유리합니다.

/api/v1/users → /api/v2/users 처럼 URL에 버전을 박거나 Accept 헤더로 협상합니다. 호환을 깨는 변경은 반드시 새 버전으로 분리합니다.

응용 사례

토스, 카카오뱅크 같은 핀테크 앱은 외부 결제·송금 API에 강한 멱등성을 요구합니다. 클라이언트가 같은 결제 요청을 재시도해도 중복 결제가 일어나지 않도록 Idempotency-Key 헤더를 활용합니다. 공공 데이터 포털은 OpenAPI 명세로 API 카탈로그를 제공해 외부 개발자가 자동으로 SDK를 생성할 수 있게 합니다.

⚠️ 자주 하는 실수: 모든 오류를 200 OK로 내려주고 바디에 success:false를 넣는 방식은 캐시·모니터링·로깅에 치명적입니다. 상태 코드를 정확히 사용하세요.
상 · 심화

전문가 레벨 이해

API 설계의 진짜 어려움은 일관성·진화 가능성·하위 호환의 균형입니다. Roy Fielding의 REST 논문은 HATEOAS(Hypermedia as the Engine of Application State)를 강조하지만, 실무에서는 거의 채택되지 않고 Pragmatic REST가 우세합니다. 백워드 호환을 위해 필드 추가는 허용하되 제거·의미 변경은 금기입니다. 분산 환경에서 정확히 한 번 처리(exactly-once delivery)는 실제로 불가능하므로 "최소 한 번 + 멱등성"을 결합해 실효적 정확성을 만듭니다. 비동기 API에는 webhook, 서버 전송 이벤트(SSE), WebSocket이 있고, 각자 신뢰성·복잡도·인프라 요구가 다릅니다. OpenAPI/AsyncAPI 같은 명세 우선(spec-first) 워크플로는 명세에서 코드와 테스트, 문서를 자동 생성하여 표류를 막습니다.

최신 연구·동향

gRPC-Web과 Connect 프로토콜은 브라우저에서도 RPC를 자연스럽게 쓸 수 있게 하며, tRPC는 TypeScript 환경에서 타입을 공유하면서 RPC를 표현합니다. GraphQL Federation은 여러 마이크로서비스의 스키마를 하나의 그래프로 묶어 클라이언트가 단일 엔드포인트로 질의하게 합니다. HTTP/3 + QUIC의 보급은 모바일 환경에서 핸드오버 시 연결 끊김을 줄입니다.
