하 · 입문

한 줄로 이해하기

HTTP는 클라이언트와 서버가 약속된 형식으로 대화하는 규칙이고, API는 서버 기능을 외부에서 호출할 수 있게 만든 창구입니다.

왜 배워야 하나?

모바일 앱에서 날씨를 확인하면 앱이 날씨 API를 호출해 서버로부터 데이터를 받아옵니다. 카카오 로그인 버튼을 누르면 카카오 인증 API가 동작합니다. 인터넷 뱅킹의 송금 버튼도 결국 은행 서버의 API를 부르는 행위입니다. HTTP/API를 이해하면 어떤 외부 서비스든 연결할 수 있고, 본인의 서비스도 다른 시스템과 연동되도록 만들 수 있어 백엔드 개발의 의사소통 능력을 갖추게 됩니다.

🌱 핵심 용어

  • HTTP: 웹에서 요청/응답을 주고받는 표준 프로토콜.
  • 메서드: GET(조회), POST(생성), PUT(수정), DELETE(삭제).
  • 상태 코드: 200(성공), 404(없음), 500(서버 오류) 등.
  • JSON: 서버와 클라이언트가 주고받는 가장 흔한 데이터 형식.
  • 엔드포인트: API에서 호출할 수 있는 URL 경로.

개념 풀어쓰기

HTTP를 편지에 비유해보세요. 편지에는 보내는 사람, 받는 사람, 제목, 본문이 있습니다. HTTP 요청도 마찬가지로 메서드(어떤 일을 할지), URL(어디로 갈지), 헤더(부가 정보), 본문(실제 데이터)이라는 구조가 있습니다. 답장이 오면 그 안에 처리 결과(상태 코드)와 응답 본문이 담겨 있습니다. API는 식당의 메뉴판과 같습니다. 손님이 음식을 직접 만들 수 없으니 메뉴판에 적힌 항목을 주문해서 받습니다. 마찬가지로 외부 시스템은 서버 내부 코드를 바로 호출할 수 없으니 미리 약속한 엔드포인트로 요청을 보냅니다. RESTful API는 자원(URL)과 행동(메서드)을 명확히 분리해 의미를 직관적으로 표현합니다.

💡 학습 TIP: Postman이나 curl로 실제 공개 API(예: jsonplaceholder.typicode.com)에 요청을 보내보세요. 손으로 한 번 해보면 이해가 빠릅니다.

요약

  • HTTP는 요청과 응답으로 동작하는 무상태 프로토콜이다.
  • 메서드와 URL 조합으로 자원의 행동을 표현한다.
  • 상태 코드로 결과를 분류한다.
  • JSON은 가장 널리 쓰이는 페이로드 형식이다.
  • API는 시스템 간 약속된 인터페이스다.
중 · 응용

이론 깊이

HTTP/1.1은 지속 연결(keep-alive)과 파이프라이닝을 도입했지만 헤드 오브 라인 블로킹 문제가 있습니다. HTTP/2는 단일 TCP 연결 위에서 여러 스트림을 멀티플렉싱하고 헤더 압축(HPACK)을 적용해 성능을 높였습니다. HTTP/3는 QUIC(UDP 기반) 위에서 동작하여 연결 지연을 줄이고 패킷 손실 회복을 빠르게 합니다. RESTful 설계는 자원 중심으로 URL을 설계하고 HTTP 메서드의 의미를 따릅니다. GraphQL은 클라이언트가 필요한 필드만 요청하는 쿼리 언어로, 모바일 같이 네트워크가 제한된 환경에서 페이로드를 줄여줍니다. gRPC는 Protocol Buffers로 직렬화한 바이너리 메시지를 HTTP/2 위에서 주고받는 RPC 프레임워크로 마이크로서비스 간 통신에 적합합니다. CORS는 다른 출처에서의 호출을 제한하는 브라우저 보안 정책으로 서버에서 Access-Control-Allow-Origin 헤더로 허용 정책을 설정합니다.

코드 패턴

// Express REST 라우트
app.get('/api/users/:id', async (req, res) => {
  const user = await db.findUser(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not Found' });
  res.json(user);
});
app.post('/api/users', async (req, res) => {
  const created = await db.createUser(req.body);
  res.status(201).json(created);
});

💻 예제 1: REST vs GraphQL

REST는 /users/123, /users/123/posts처럼 여러 엔드포인트를 호출해야 할 때 GraphQL은 단일 쿼리로 user와 posts를 한 번에 가져옵니다.

💻 예제 2: 페이지네이션

오프셋 방식(?page=2&size=20)은 단순하지만 데이터가 변하면 중복/누락이 발생합니다. 커서 방식(?cursor=abc)은 안정적이며 무한 스크롤에 적합합니다.

실무 응용

인스타그램 같은 피드 서비스는 GraphQL로 게시물, 좋아요 수, 댓글 일부, 작성자 정보를 한 쿼리로 받아옵니다. 토스페이먼츠 같은 결제 API는 RESTful 표준을 엄격히 따라 멱등성 키를 헤더로 받아 중복 결제를 방지합니다. 슬랙은 이벤트 발생 시 등록된 URL로 POST 요청을 보내는 웹훅 패턴으로 외부 서비스와 연동합니다.

⚠️ 자주 하는 실수: 모든 응답을 200 OK로 내려주고 본문에서만 에러를 표시하면 클라이언트의 자동 재시도/모니터링이 깨집니다. 의미에 맞는 상태 코드를 쓰세요.
상 · 심화

전문가 레벨

대규모 API 게이트웨이 환경에서는 라우팅, 인증, 속도 제한, 사이드카 트레이싱이 모두 한 곳에서 처리됩니다. Envoy 기반 게이트웨이는 동적 설정 갱신(xDS), 회로 차단기, 재시도 정책을 표준화합니다. HTTP 캐싱은 ETag/Last-Modified와 Cache-Control max-age, stale-while-revalidate, stale-if-error로 세밀하게 제어합니다. CDN과 함께 사용할 때는 Vary 헤더와 캐시 키 정규화가 중요합니다. API 버전 관리는 URL 버전(/v1/), 헤더 버전, 콘텐츠 협상 방식이 있고 운영 비용이 다릅니다. OpenAPI 스펙으로 정의를 단일 출처로 두면 서버 스텁, 클라이언트 SDK, 문서, 계약 테스트(Pact)까지 자동화됩니다. GraphQL의 N+1 문제는 DataLoader로 배치/캐싱하고, 쿼리 깊이/복잡도 제한과 영속화 쿼리(persisted queries)로 악의적 쿼리를 방어합니다. gRPC 양방향 스트리밍은 채팅, 실시간 위치 추적, 텔레메트리 같은 고빈도 통신을 효율적으로 처리합니다. 멱등성 토큰, 분산 트랜잭션의 사가 패턴, 동시성 제어를 위한 ETag 기반 낙관적 잠금은 신뢰성 있는 API의 필수 요소입니다.

고급 패턴

// GraphQL DataLoader 배치
const userLoader = new DataLoader(async (ids) => {
  const users = await db.findUsersByIds(ids);
  return ids.map(id => users.find(u => u.id === id));
});
const resolvers = {
  Post: { author: (post) => userLoader.load(post.authorId) }
};

최신 동향

HTTP/3와 QUIC은 모바일 환경에서 네트워크 전환 시 연결을 유지해 끊김을 줄입니다. gRPC-Web과 Connect는 브라우저에서 gRPC 호출을 가능하게 합니다. tRPC는 TypeScript 풀스택 프로젝트에서 스키마를 타입으로 공유하며 코드 생성 없이 RPC를 제공합니다. 엣지에서 동작하는 API(Cloudflare Workers, Vercel Edge)는 콜드 스타트 ms 단위, 글로벌 레이턴시 수십 ms를 달성합니다. AsyncAPI 스펙은 메시지 기반 비동기 API를 표준 문서로 표현해 OpenAPI를 보완합니다.
