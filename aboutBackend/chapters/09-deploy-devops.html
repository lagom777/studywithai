Chapter 9 — Deployment, Docker & CI/CD Pipeline

CI/CD 파이프라인 시뮬레이터 포함

DevOps란?

DevOps는 개발(Development)과 운영(Operations)을 하나로 통합하는 문화이자 방법론입니다. 코드를 작성하고, 테스트하고, 빌드하고, 배포하는 전 과정을 자동화하여 빠르고 안정적인 소프트웨어 딜리버리를 추구합니다.

CI/CD 파이프라인 시뮬레이터

"파이프라인 실행" 버튼을 누르면 코드 커밋이 Build → Test → Stage → Deploy 단계를 거치는 과정을 관찰할 수 있습니다. "테스트 실패" 모드에서는 Test 단계에서 파이프라인이 중단됩니다.

🚀 파이프라인을 실행하고 CI/CD 흐름을 관찰하세요

Docker 컨테이너

Docker는 애플리케이션과 의존성을 하나의 컨테이너로 패키징합니다. "내 컴퓨터에서는 되는데?"를 해결해주는 핵심 도구입니다.

FROM node:20-alpine WORKDIR /app # 의존성 먼저 설치 (캐시 활용) COPY package*.json ./ RUN npm ci --only=production # 소스 코드 복사 COPY . . EXPOSE 3000 CMD ["node", "server.js"]
docker build -t my-backend:1.0 . [+] Building 12.3s (10/10) FINISHED docker run -d -p 3000:3000 my-backend:1.0 a7b3c4d5e6f... docker ps CONTAINER ID IMAGE STATUS PORTS a7b3c4d5e6f my-backend:1.0 Up 5 sec 0.0.0.0:3000→3000

🐳 Docker 핵심 개념

Image(설계도), Container(실행 인스턴스), Volume(데이터 영속화), Network(컨테이너 간 통신). docker-compose로 여러 컨테이너를 한 번에 관리합니다.

Kubernetes

☸️ K8s 핵심 오브젝트

Pod(컨테이너 그룹), Service(네트워크 엔드포인트), Deployment(선언적 배포), Ingress(외부 트래픽 라우팅). 자동 스케일링, 자가 치유, 롤링 업데이트를 제공합니다.

apiVersion: apps/v1 kind: Deployment metadata: name: backend-app spec: replicas: 3 selector: matchLabels: app: backend template: metadata: labels: app: backend spec: containers: - name: backend image: my-backend:1.0 ports: - containerPort: 3000 resources: limits: cpu: "500m" memory: "256Mi"

CI/CD 파이프라인

Code Push → Build → Test → Stage → Deploy
# .github/workflows/deploy.yml name: CI/CD Pipeline on: push: branches: [main] jobs: build-and-deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' - run: npm ci - run: npm test - run: docker build -t app:${{ github.sha }} . - run: docker push registry/app:${{ github.sha }} - run: kubectl set image deployment/app app=registry/app:${{ github.sha }}

배포 전략

🔵🟢 블루-그린 배포

두 환경(블루=현재, 그린=신규)을 유지하고 트래픽을 한 번에 전환합니다. 롤백이 즉각적이지만 리소스가 2배 필요합니다.

🐦 카나리 배포

신규 버전을 소수의 사용자에게만 먼저 배포합니다. 문제가 없으면 점진적으로 확대합니다.

🔄 롤링 업데이트

인스턴스를 하나씩 새 버전으로 교체합니다. Kubernetes의 기본 배포 전략입니다.

하 · 입문

한 줄로 이해하기

배포는 "코드를 사용자가 쓸 수 있는 상태로 올리는 일", DevOps는 "그 과정을 자동화하고 운영까지 함께 책임지는 문화"입니다.

왜 중요한가?

코드를 잘 짜도 배포가 두렵고 어려우면 새 기능이 한 달에 한 번만 나갑니다. 토스가 하루 수십 번 배포하는 비결, 넷플릭스가 카오스 몽키로 일부러 장애를 내는 자신감의 근원이 모두 DevOps에 있습니다. 빠른 피드백, 작은 변경, 자동화된 안전망이 핵심입니다.

🌱 핵심 용어

  • CI: 코드 통합 시 자동 빌드·테스트
  • CD: 자동 배포(Continuous Deployment/Delivery)
  • 컨테이너: 앱과 환경을 하나로 패키징(Docker)
  • 오케스트레이션: 컨테이너 자동 운영(Kubernetes)
  • IaC: 인프라를 코드로 정의(Terraform)

핵심 개념 풀어쓰기

"내 컴퓨터에선 잘 돌아가는데..."는 컨테이너로 사실상 사라집니다. Docker 이미지는 OS·런타임·코드·설정을 한 덩어리로 묶어 어디서든 똑같이 실행되게 합니다. CI는 PR이 올라올 때마다 자동으로 빌드와 테스트를 돌려 "깨진 코드"가 main에 들어가지 못하게 막는 안전망입니다. CD는 그 뒤로 자동으로 스테이징·프로덕션까지 배포해 사람의 손을 최소화합니다.

💡 학습 TIP: GitHub Actions로 간단한 워크플로(테스트 → 빌드 → 도커 푸시)를 직접 만들어보세요. 1시간이면 첫 파이프라인을 갖출 수 있습니다.

한 페이지 요약

  • 작게, 자주 배포하라(small batches)
  • 모든 환경은 코드로 재현 가능해야
  • CI는 안전망, CD는 가속 페달
  • 롤백을 배포만큼 자주 연습하라
  • 관측성 없이는 자동화도 위험
중 · 응용

이론 깊이 들어가기

현대 배포의 표준 단위는 컨테이너 이미지입니다. 멀티스테이지 Dockerfile로 빌드와 런타임을 분리해 이미지 크기와 공격 표면을 줄입니다. Kubernetes는 선언적 manifest와 컨트롤 루프 패턴으로 desired state를 유지합니다. 배포 전략은 롤링·블루-그린·카나리·다크 런치가 있고, 각자 리스크/롤백 속도/트래픽 통제 정도가 다릅니다. 시크릿은 코드/이미지에 절대 박지 않고 Vault, AWS Secrets Manager, Kubernetes Secret(암호화 백엔드)로 분리합니다. 관측성 3대 축은 메트릭(Prometheus), 로그(Loki/ELK), 트레이스(Tempo/Jaeger)이고 SLO·에러 버짓으로 배포 속도를 조절합니다.

핵심 Dockerfile 패턴

FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
USER node
CMD ["node","dist/server.js"]

대표 예제

📐 예제 1: 카나리 배포

새 버전에 트래픽 5% → 25% → 100%를 단계적으로 보내면서 에러율과 레이턴시를 자동 모니터링하고, 기준을 넘으면 자동 롤백.

📐 예제 2: GitOps

운영 상태를 Git 저장소에 manifest로 두고 ArgoCD가 자동으로 desired state ↔ 실제 클러스터를 동기화합니다. 변경 이력이 곧 git log입니다.

응용 사례

토스의 배포는 사내 PaaS로 추상화되어 개발자가 PR 머지만 하면 카나리 → 단계 승격이 자동입니다. 카카오뱅크는 트래픽 피크 직전 오토스케일을 미리 워밍업해 콜드 스타트를 피하고, 장애 시뮬레이션을 정기적으로 돌립니다. 인디 개발자는 Render/Fly.io/Vercel로 인프라 운영 부담 없이 production-grade 배포를 누립니다.

⚠️ 자주 하는 실수: "한 번에 큰 변경"을 배포하는 습관은 장애를 키우고 롤백을 어렵게 합니다. 작은 변경을 자주 배포하는 쪽이 통계적으로 훨씬 안전합니다.
상 · 심화

전문가 레벨 이해

대규모 운영에서는 DORA 4대 지표(배포 빈도, 변경 리드 타임, MTTR, 변경 실패율)가 조직 건강의 핵심 KPI가 됩니다. 플랫폼 엔지니어링 팀은 IDP(Internal Developer Platform)를 구축해 골든 패스(golden path)와 셀프 서비스 환경을 제공합니다. 이미지·아티팩트 공급망 보안(SLSA, Sigstore, in-toto)으로 빌드 출처와 무결성을 증명하고, SBOM으로 의존성 취약점을 추적합니다. 카오스 엔지니어링(Chaos Mesh, Litmus)으로 의도적 장애를 주입해 시스템 회복력을 검증합니다. 멀티 리전·멀티 클러스터·셀 기반 아키텍처는 장애 격리와 점진적 배포를 결합합니다.

최신 연구·동향

eBPF는 사이드카 없이 런타임 보안과 관측성을 가능하게 하고, Wasm은 컨테이너보다 가벼운 차세대 배포 단위로 부상 중입니다(KubeWarden, SpinKube). FinOps가 표준이 되어 비용 가시화와 자동 최적화가 배포 파이프라인 안에 들어옵니다. AI 도입으로 incident summary와 root-cause 분석, runbook 실행이 점차 자동화됩니다.
