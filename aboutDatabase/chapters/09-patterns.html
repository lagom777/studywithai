9장. 실전 설계 패턴

1. 이벤트 소싱(Event Sourcing)

쉬운 설명

당신의 은행 통장을 생각해보세요. 통장에는 현재 잔액 하나가 아니라, 모든 거래 내역(입금, 출금)이 남아있습니다.

개념: 데이터의 "현재 상태"를 저장하는 대신, "모든 변경 기록"을 저장합니다.

계산: 현재 잔액 = 처음 잔액 + 모든 거래 내역의 합

장점:

  • 감사 추적(Audit Trail)이 자동으로 생깁니다
  • 과거 어느 시점의 상태도 복구할 수 있습니다
  • 버그를 발견했을 때 과거 상태로 돌아갈 수 있습니다

학술 설명

이벤트 소싱 아키텍처:

  • 이벤트 스토어(Event Store): 모든 상태 변화를 이벤트로 저장. append-only 로그.
  • 이벤트 리플레이(Event Replay): 이벤트를 처음부터 다시 실행해 특정 시점의 상태 복원.
  • 현재 상태 조회(Current State Query): 모든 이벤트를 합쳐 현재 상태 계산.

구현 전략:

  • 이벤트 타입 정의: UserCreated, MoneyDeposited, MoneyWithdrawn 등
  • 스냅샷(Snapshot): 이벤트가 너무 많으면 중간 지점의 상태를 저장해 성능 개선
  • 프로젝션(Projection): 이벤트로부터 읽기 최적화된 뷰 생성

데이터 모델: { eventId, aggregateId, eventType, timestamp, data, version }

도구: Event Store, Axon Framework, Apache Kafka

2. CQRS (Command Query Responsibility Segregation)

쉬운 설명

개념: 읽기와 쓰기를 완전히 분리합니다.

식당 비유: 손님의 주문을 받는 사람과 요리하는 사람이 다릅니다. 주문받는 사람은 주문만 받고 기록하고, 요리사는 그 기록을 보고 요리합니다.

실제 예시:

  • 쓰기(Command): 상품 주문, 잔액 이체 등. 느리지만 정확해야 함.
  • 읽기(Query): 주문 목록 보기, 잔액 조회 등. 매우 빨라야 함.

장점: 읽기는 빠르고, 쓰기는 정확합니다.

학술 설명

CQRS 패턴의 구조:

  • 명령 모델(Command Model): 쓰기 최적화. 정규화된 스키마, ACID 보장.
  • 쿼리 모델(Query Model): 읽기 최적화. 반정규화된 뷰, NoSQL 가능.

동기화 메커니즘:

  • 동기식: 쓰기 후 즉시 읽기 모델 갱신. 복잡하고 느림.
  • 비동기식: 이벤트 기반으로 읽기 모델 갱신. 최종 일관성 보장.

최종 일관성(Eventual Consistency): 쓰기 후 읽기 모델이 약간 뒤처질 수 있지만, 결국 일치합니다.

예시: 명령 모델은 PostgreSQL(쓰기), 쿼리 모델은 Elasticsearch(읽기)

3. 마이그레이션과 버전 관리

쉬운 설명

앱이 업데이트되듯이, DB도 버전 관리가 필요합니다.

시나리오: 처음에는 users 테이블이 name 컬럼만 있었는데, 나중에 email 컬럼을 추가해야 합니다.

방법:

  • 마이그레이션 파일: "001_create_users_table.sql", "002_add_email_to_users.sql"
  • 버전 추적: "현재 DB는 v5입니다" 라고 기록
  • 자동 실행: 앱 시작 시 현재 버전에서 최신 버전까지 마이그레이션 자동 실행

무중단 마이그레이션: 서비스 중단 없이 DB 변경

학술 설명

스키마 마이그레이션 전략:

  • Blue-Green 배포: 이전 스키마(Blue)와 새 스키마(Green)를 병행 운영 후 전환
  • 점진적 마이그레이션: 신규 데이터는 새 스키마, 기존 데이터는 호환 레이어
  • 롤백 계획: 마이그레이션 실패 시 복구 전략 수립

무중단 마이그레이션(Zero-Downtime Migration):

  1. 새 컬럼 추가 (NOT NULL 제약 없이)
  2. 트래픽 일부를 새 스키마로 시작
  3. 기존 데이터 배경에서 마이그레이션
  4. 트래픽 100% 전환

도구: Flyway, Liquibase, Alembic (Python)

4. 실전 안티패턴과 모범 사례

쉬운 설명

안티패턴 (나쁜 패턴) vs 모범 사례 (좋은 패턴):

❌ 안티패턴 ✅ 모범 사례
God Table: 모든 걸 한 테이블에 (user_id, product_id, order_id, comment...) 관심사 분리: Users, Products, Orders 테이블 분리
NULL 남용: 선택 사항을 NULL로 처리 기본값 설정 또는 별도 테이블 사용
N+1 쿼리: 루프 안에서 쿼리 실행 조인이나 배치 쿼리 사용
하드딜리트: 데이터 영구 삭제 소프트 딜리트: deleted_at 컬럼 추가
스키마 버전 없음: DB 변경 기록 부재 마이그레이션 파일 관리

EAV 안티패턴: Entity-Attribute-Value 패턴은 확장성은 좋지만 조회가 복잡합니다.

다형성 연관: 한 컬럼이 여러 테이블을 참조하지 않기. 외래키 제약이 필요합니다.

학술 설명

주요 안티패턴 분석:

1. EAV (Entity-Attribute-Value) 안티패턴

속성이 동적으로 변할 때 사용하는 패턴. 하지만 조인이 많고 인덱싱이 어렵습니다.

해결책: JSON 컬럼 (PostgreSQL JSONB), NoSQL 사용

2. 다형성 미사용

예: comments 테이블이 user, product, order 모두를 참조. 외래키 제약 불가능.

해결책: 각 엔티티별 댓글 테이블, 또는 단일 comments 테이블 + 타입 컬럼

3. N+1 쿼리 문제

1개 쿼리 + N개 반복 쿼리 = N+1. 대규모 데이터에서 성능 저하.

해결책: 1) 조인 사용, 2) 배치 쿼리, 3) ORM의 eager loading

4. Soft Delete vs Hard Delete

Soft Delete (deleted_at 컬럼): 감사 추적 가능, 복구 가능. 쿼리 복잡화.

Hard Delete: 간단하지만 데이터 손실, 감사 추적 불가.

모범 사례: 둘 다 사용. 중요한 데이터는 soft delete, 로그는 hard delete

5. Temporal 데이터 처리

시간이 지나면서 데이터 변경 추적. 버전 컬럼이나 별도 히스토리 테이블 사용.

표준: SQL 2011 TEMPORAL 테이블 (MySQL 5.7.6+, PostgreSQL)

한 줄로 이해하기

실전 설계 패턴은 자주 마주치는 데이터 문제를 잘 푸는 검증된 방법들의 모음집입니다.

왜 중요한가?

패턴을 알면 바퀴를 다시 발명하지 않아도 됩니다. 첫째, 캐시 패턴 하나만 잘 써도 응답 속도가 10배 빨라집니다. 둘째, 메시지 큐와 Outbox 패턴은 분산 환경의 데이터 일관성을 지켜 줍니다. 셋째, 읽기/쓰기 분리 패턴은 트래픽 폭주 시에도 시스템을 안정적으로 유지합니다.

🌱 핵심 용어

  • 캐시(Cache): 자주 쓰는 데이터를 빠른 저장소에 복사.
  • 큐(Queue): 작업을 줄 세워 처리하는 통로.
  • Outbox: DB 변경과 메시지를 한 트랜잭션으로 묶는 패턴.
  • 읽기 복제본: 조회 부하를 분산하는 사본 노드.
  • 아이덴포턴시(멱등성): 같은 요청이 여러 번 와도 결과는 한 번.

개념 풀어쓰기

설계 패턴은 요리 레시피와 같습니다. 처음에는 자유롭게 시도해도 좋지만, 검증된 레시피를 따라 하면 실패 확률이 훨씬 낮습니다. 캐시 패턴은 단골 메뉴를 미리 만들어 두는 일과 같아 손님이 오면 바로 내놓을 수 있습니다. 큐 패턴은 주방에 주문서를 차곡차곡 쌓는 것과 같아 한꺼번에 손님이 몰려도 차분히 처리할 수 있게 해 줍니다. 읽기/쓰기 분리는 메인 주방과 셀프 서비스 코너를 따로 두는 격이라 주문 처리와 자주 가져가는 음료가 서로 방해하지 않습니다. Outbox 패턴은 영수증과 주방 주문서를 동시에 끊어 둘 다 빠짐없이 전달되도록 하는 안전장치입니다. 멱등성은 손님이 같은 카드로 두 번 결제 버튼을 눌러도 한 번만 청구되도록 하는 안전망입니다. 이런 패턴들을 알고 적용하면, 갑작스러운 부하나 장애 상황에도 데이터가 깨지거나 사라지지 않게 시스템을 운영할 수 있습니다.

💡 학습 TIP: 처음에는 캐시-어사이드 패턴 하나만이라도 본인 프로젝트에 적용해 보세요. Redis 한 번 도입한 경험이 다른 패턴을 빠르게 이해하게 해 줍니다.

요약

  • 패턴은 검증된 해결책 모음입니다.
  • 캐시로 응답 속도 향상.
  • 큐로 부하 평준화와 비동기 처리.
  • Outbox로 분산 일관성 보장.
  • 멱등성으로 중복 요청 안전 처리.
중 · 응용

이론 깊이

캐시 전략에는 Cache-Aside, Read-Through, Write-Through, Write-Behind, Refresh-Ahead 다섯 가지가 대표적입니다. Cache-Aside는 애플리케이션이 캐시 미스 시 DB를 조회해 채워 넣는 가장 흔한 패턴이고, Write-Through는 일관성에 강하지만 쓰기 지연이 늘어나는 트레이드오프가 있습니다. TTL 설정과 캐시 무효화는 Phil Karlton의 명언처럼 컴퓨터 과학에서 가장 어려운 문제 중 하나입니다. Saga 패턴은 분산 트랜잭션의 대안으로 보상 이벤트 시퀀스를 정의하며, 오케스트레이션 방식과 코레오그래피 방식이 있습니다. CQRS는 읽기 모델과 쓰기 모델을 분리해 각각의 요구에 맞춘 최적화를 가능케 합니다. 이벤트 소싱은 상태 대신 이벤트 로그를 저장하고, 스냅샷과 함께 운영합니다. 멱등키(Idempotency Key)는 결제·주문 시스템에서 필수이며, 클라이언트가 발급한 UUID를 서버가 일정 기간 캐시해 중복 요청을 무시합니다. 분산 락은 Redlock 알고리즘이나 Zookeeper의 임시 노드, ZAB 기반 임차(lease)로 구현됩니다.

실전 SQL/쿼리

-- Outbox 패턴
BEGIN;
INSERT INTO orders (order_id, customer_id, amount, status)
VALUES ('ord-001', 1001, 50000, 'CREATED');

INSERT INTO outbox (event_id, aggregate, type, payload, created_at)
VALUES (gen_random_uuid(),
        'orders', 'OrderCreated',
        '{"order_id":"ord-001","amount":50000}',
        NOW());
COMMIT;
-- 별도 워커가 outbox를 polling/CDC로 읽어 Kafka로 발행

-- 멱등키로 중복 요청 차단
INSERT INTO payment (idempotency_key, order_id, amount, status)
VALUES ('idem-abc-001', 'ord-001', 50000, 'PAID')
ON CONFLICT (idempotency_key) DO NOTHING
RETURNING payment_id;

-- 키셋 페이지네이션
SELECT order_id, created_at, amount
FROM   orders
WHERE  member_id = 1001
  AND  (created_at, order_id) < (?, ?)
ORDER  BY created_at DESC, order_id DESC
LIMIT  20;

예제

💻 예제 1: 좋아요 카운터

Redis INCR로 좋아요를 즉시 반영하고, 1분에 한 번 배치로 PostgreSQL에 동기화합니다. 정확도가 약간 양보되지만 성능이 수십 배 향상됩니다.

💻 예제 2: 주문-결제 사가

주문 생성 → 결제 시도 → 재고 차감 → 배송 예약 시퀀스에서 어느 단계가 실패하면 보상 이벤트(주문 취소, 환불, 재고 복구)를 순서대로 발행합니다. Saga 오케스트레이터가 진행 상태를 관리합니다.

실무 응용

대규모 이커머스는 카탈로그 캐시(Redis), 검색 인덱스(Elasticsearch), 주문 DB(PostgreSQL), 분석 웨어하우스(Snowflake)로 구성된 폴리글랏 아키텍처를 운영합니다. 또한 결제 PG는 멱등키, 사가, Outbox를 모두 결합한 안전 패턴으로 운영되며, 게임 백엔드는 핫 데이터를 인메모리 그리드에 두고 주기적으로 영속화하는 패턴을 사용합니다.

⚠️ 자주 하는 실수: 캐시와 DB 사이의 무효화 순서를 잘못 두면 stale 데이터가 영구적으로 남습니다. 또 Outbox 없이 "DB 저장 후 큐 발행"을 분리하면 일관성이 깨집니다.
상 · 심화

전문가 레벨

Outbox 패턴의 진화형으로 Transactional Outbox + CDC가 사실상 표준입니다. Debezium이 PostgreSQL 논리 복제 슬롯을 구독해 outbox 테이블의 INSERT를 Kafka 토픽으로 자동 발행하며, exactly-once에 가까운 의미를 제공합니다. CQRS 시스템에서는 Projection의 멱등 재구성과 진행 위치(체크포인트) 관리가 중요하며, 이벤트 스트림에 대한 점프와 재처리 전략을 설계해야 합니다. Saga의 코레오그래피는 결합도를 낮추지만 흐름 추적이 어렵고, 오케스트레이션은 명시적이지만 단일 장애점이 될 수 있어 워크플로 엔진(Temporal, Cadence, AWS Step Functions)이 보조 도구로 자주 사용됩니다. 분산 락은 Redlock의 안전성에 대한 논쟁이 있으며, 실무에서는 펜싱 토큰(monotonic increasing token)으로 잠금이 만료된 후의 작업을 거부하는 패턴이 권장됩니다. 멀티리전 ACID 결제 시스템에서는 글로벌 ID(Snowflake/ULID), 시간 동기화, idempotency, 재시도 백오프, 회로차단기를 결합한 안정성 패턴이 표준입니다. 데이터 메시 환경에서는 도메인 이벤트 스키마와 호환성(스키마 레지스트리, Avro/Protobuf)을 강제해 변경에 강하게 만듭니다.

고급 패턴

-- Outbox + CDC + Avro 발행 (Debezium 핵심 부분)
-- outbox 테이블 정의 (debezium outbox-event-router 컨벤션)
CREATE TABLE outbox (
  id            UUID PRIMARY KEY,
  aggregatetype VARCHAR(50),
  aggregateid   VARCHAR(50),
  type          VARCHAR(50),
  payload       JSONB,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 펜싱 토큰을 결합한 분산 락
INSERT INTO dist_lock (key, owner, fence_token, expires_at)
VALUES ('settle-2025-05-07', 'worker-3', nextval('fence_seq'),
        NOW() + INTERVAL '60 seconds')
ON CONFLICT (key) DO UPDATE
SET owner = EXCLUDED.owner,
    fence_token = nextval('fence_seq'),
    expires_at = EXCLUDED.expires_at
WHERE dist_lock.expires_at < NOW()
RETURNING fence_token;

-- 이벤트 소싱 + 스냅샷 결합
SELECT *
FROM   account_events
WHERE  account_id = ?
  AND  version > (
    SELECT version
    FROM   account_snapshot
    WHERE  account_id = ?
    ORDER  BY version DESC
    LIMIT  1
  )
ORDER  BY version;

최신 동향

Temporal과 Restate 같은 워크플로 런타임은 영속 함수(durable execution)를 통해 보상 트랜잭션과 재시도를 코드 레벨로 자연스럽게 표현합니다. 또한 zero-ETL(AWS Aurora→Redshift, Snowflake Iceberg)은 Outbox/CDC 부담을 인프라가 흡수하는 흐름을 보입니다. WebAssembly 기반 엣지 데이터베이스(Turso, Cloudflare D1)는 글로벌 분산 캐시 패턴을 단순화합니다. LLM Agent 시스템은 멱등성, 보상, 재현성 패턴을 더 강하게 요구하면서 트랜잭셔널 메모리 모델을 재조명하고 있습니다.
