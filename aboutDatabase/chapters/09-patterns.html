9장. 실전 설계 패턴

1. 이벤트 소싱(Event Sourcing)

쉬운 설명

당신의 은행 통장을 생각해보세요. 통장에는 현재 잔액 하나가 아니라, 모든 거래 내역(입금, 출금)이 남아있습니다.

개념: 데이터의 "현재 상태"를 저장하는 대신, "모든 변경 기록"을 저장합니다.

계산: 현재 잔액 = 처음 잔액 + 모든 거래 내역의 합

장점:

  • 감사 추적(Audit Trail)이 자동으로 생깁니다
  • 과거 어느 시점의 상태도 복구할 수 있습니다
  • 버그를 발견했을 때 과거 상태로 돌아갈 수 있습니다

학술 설명

이벤트 소싱 아키텍처:

  • 이벤트 스토어(Event Store): 모든 상태 변화를 이벤트로 저장. append-only 로그.
  • 이벤트 리플레이(Event Replay): 이벤트를 처음부터 다시 실행해 특정 시점의 상태 복원.
  • 현재 상태 조회(Current State Query): 모든 이벤트를 합쳐 현재 상태 계산.

구현 전략:

  • 이벤트 타입 정의: UserCreated, MoneyDeposited, MoneyWithdrawn 등
  • 스냅샷(Snapshot): 이벤트가 너무 많으면 중간 지점의 상태를 저장해 성능 개선
  • 프로젝션(Projection): 이벤트로부터 읽기 최적화된 뷰 생성

데이터 모델: { eventId, aggregateId, eventType, timestamp, data, version }

도구: Event Store, Axon Framework, Apache Kafka

2. CQRS (Command Query Responsibility Segregation)

쉬운 설명

개념: 읽기와 쓰기를 완전히 분리합니다.

식당 비유: 손님의 주문을 받는 사람과 요리하는 사람이 다릅니다. 주문받는 사람은 주문만 받고 기록하고, 요리사는 그 기록을 보고 요리합니다.

실제 예시:

  • 쓰기(Command): 상품 주문, 잔액 이체 등. 느리지만 정확해야 함.
  • 읽기(Query): 주문 목록 보기, 잔액 조회 등. 매우 빨라야 함.

장점: 읽기는 빠르고, 쓰기는 정확합니다.

학술 설명

CQRS 패턴의 구조:

  • 명령 모델(Command Model): 쓰기 최적화. 정규화된 스키마, ACID 보장.
  • 쿼리 모델(Query Model): 읽기 최적화. 반정규화된 뷰, NoSQL 가능.

동기화 메커니즘:

  • 동기식: 쓰기 후 즉시 읽기 모델 갱신. 복잡하고 느림.
  • 비동기식: 이벤트 기반으로 읽기 모델 갱신. 최종 일관성 보장.

최종 일관성(Eventual Consistency): 쓰기 후 읽기 모델이 약간 뒤처질 수 있지만, 결국 일치합니다.

예시: 명령 모델은 PostgreSQL(쓰기), 쿼리 모델은 Elasticsearch(읽기)

3. 마이그레이션과 버전 관리

쉬운 설명

앱이 업데이트되듯이, DB도 버전 관리가 필요합니다.

시나리오: 처음에는 users 테이블이 name 컬럼만 있었는데, 나중에 email 컬럼을 추가해야 합니다.

방법:

  • 마이그레이션 파일: "001_create_users_table.sql", "002_add_email_to_users.sql"
  • 버전 추적: "현재 DB는 v5입니다" 라고 기록
  • 자동 실행: 앱 시작 시 현재 버전에서 최신 버전까지 마이그레이션 자동 실행

무중단 마이그레이션: 서비스 중단 없이 DB 변경

학술 설명

스키마 마이그레이션 전략:

  • Blue-Green 배포: 이전 스키마(Blue)와 새 스키마(Green)를 병행 운영 후 전환
  • 점진적 마이그레이션: 신규 데이터는 새 스키마, 기존 데이터는 호환 레이어
  • 롤백 계획: 마이그레이션 실패 시 복구 전략 수립

무중단 마이그레이션(Zero-Downtime Migration):

  1. 새 컬럼 추가 (NOT NULL 제약 없이)
  2. 트래픽 일부를 새 스키마로 시작
  3. 기존 데이터 배경에서 마이그레이션
  4. 트래픽 100% 전환

도구: Flyway, Liquibase, Alembic (Python)

4. 실전 안티패턴과 모범 사례

쉬운 설명

안티패턴 (나쁜 패턴) vs 모범 사례 (좋은 패턴):

❌ 안티패턴 ✅ 모범 사례
God Table: 모든 걸 한 테이블에 (user_id, product_id, order_id, comment...) 관심사 분리: Users, Products, Orders 테이블 분리
NULL 남용: 선택 사항을 NULL로 처리 기본값 설정 또는 별도 테이블 사용
N+1 쿼리: 루프 안에서 쿼리 실행 조인이나 배치 쿼리 사용
하드딜리트: 데이터 영구 삭제 소프트 딜리트: deleted_at 컬럼 추가
스키마 버전 없음: DB 변경 기록 부재 마이그레이션 파일 관리

EAV 안티패턴: Entity-Attribute-Value 패턴은 확장성은 좋지만 조회가 복잡합니다.

다형성 연관: 한 컬럼이 여러 테이블을 참조하지 않기. 외래키 제약이 필요합니다.

학술 설명

주요 안티패턴 분석:

1. EAV (Entity-Attribute-Value) 안티패턴

속성이 동적으로 변할 때 사용하는 패턴. 하지만 조인이 많고 인덱싱이 어렵습니다.

해결책: JSON 컬럼 (PostgreSQL JSONB), NoSQL 사용

2. 다형성 미사용

예: comments 테이블이 user, product, order 모두를 참조. 외래키 제약 불가능.

해결책: 각 엔티티별 댓글 테이블, 또는 단일 comments 테이블 + 타입 컬럼

3. N+1 쿼리 문제

1개 쿼리 + N개 반복 쿼리 = N+1. 대규모 데이터에서 성능 저하.

해결책: 1) 조인 사용, 2) 배치 쿼리, 3) ORM의 eager loading

4. Soft Delete vs Hard Delete

Soft Delete (deleted_at 컬럼): 감사 추적 가능, 복구 가능. 쿼리 복잡화.

Hard Delete: 간단하지만 데이터 손실, 감사 추적 불가.

모범 사례: 둘 다 사용. 중요한 데이터는 soft delete, 로그는 hard delete

5. Temporal 데이터 처리

시간이 지나면서 데이터 변경 추적. 버전 컬럼이나 별도 히스토리 테이블 사용.

표준: SQL 2011 TEMPORAL 테이블 (MySQL 5.7.6+, PostgreSQL)
