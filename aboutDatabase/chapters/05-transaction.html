5장. 트랜잭션과 동시성 제어

1. 트랜잭션이란?

쉬운 설명

은행에서 A 계좌에서 돈을 빼고 B 계좌에 넣는 송금 과정을 생각해봅시다.

  • A에서 100만원 빼기
  • B에 100만원 더하기

만약 첫 번째 작업은 완료되었는데 컴퓨터가 꺼진다면? 돈이 사라집니다!

트랜잭션: 이 두 작업을 하나의 묶음으로 취급해서 "둘 다 성공" 또는 "둘 다 실패"만 가능하게 만듭니다.

논문 수준 설명

정의: 트랜잭션(Transaction)은 데이터베이스의 논리적 단위로, 여러 개의 SQL 문들이 모두 성공하거나 모두 실패해야 하는 작업의 집합입니다.

명령어:

  • BEGIN 또는 START TRANSACTION: 트랜잭션 시작
  • COMMIT: 모든 변경 사항을 영구 저장
  • ROLLBACK: 트랜잭션 시작 전으로 되돌리기

Atomicity Guarantee: 트랜잭션 내의 모든 작업이 원자적(atomic)으로 처리되어야 함

2. ACID 속성

쉬운 설명

트랜잭션이 안전하려면 4가지 속성을 가져야 합니다:

  • 원자성(Atomicity): 다 되거나 다 안 되거나. 반쪽짜리는 없음.
  • 일관성(Consistency): 데이터 규칙을 위반할 수 없음. (예: 나이는 음수 불가)
  • 격리성(Isolation): 여러 트랜잭션이 동시에 실행되어도 서로 방해하지 않음.
  • 지속성(Durability): 한번 저장되면 영구적. 전기가 나가도 복구됨.

논문 수준 설명

원자성: All or Nothing. 트랜잭션의 모든 연산이 수행되거나 하나도 수행되지 않음.

일관성: 트랜잭션 시작 전과 후의 무결성 제약 조건이 모두 만족되어야 함.

격리성: 동시 실행 중인 트랜잭션들 간의 상호 간섭이 없어야 함. Serializability 보장.

지속성: 커밋된 트랜잭션의 데이터는 시스템 장애 후에도 복구 가능해야 함.

WAL (Write-Ahead Logging): 디스크에 쓰기 전에 로그에 먼저 기록. ARIES 복구 알고리즘 사용.

3. 격리 수준

쉬운 설명

도서관에서 같은 책을 여러 명이 동시에 볼 때의 규칙처럼, DB도 격리 수준을 정합니다:

  • READ UNCOMMITTED: 누구나 마음대로 접근 (가장 빠름, 가장 위험)
  • READ COMMITTED: 완료된 책만 봄 (적당함)
  • REPEATABLE READ: 같은 책은 같은 사람과만 공유 (엄격함)
  • SERIALIZABLE: 한 명씩 순서대로 (가장 안전, 가장 느림)

논문 수준 설명

READ UNCOMMITTED: 커밋되지 않은 데이터 읽기 가능. Dirty Read 허용.

READ COMMITTED: 커밋된 데이터만 읽기. Non-repeatable Read 발생 가능.

REPEATABLE READ: 트랜잭션 중에 읽은 데이터는 항상 같음. Phantom Read 발생 가능.

SERIALIZABLE: 모든 격리 수준의 이상현상(Dirty Read, Non-repeatable Read, Phantom Read) 방지.

이상 현상:

  • Dirty Read: 커밋되지 않은 데이터를 다른 트랜잭션이 읽음
  • Non-repeatable Read: 같은 데이터를 읽을 때마다 값이 달라짐
  • Phantom Read: 범위 검색 중 새 행이 추가됨

4. 잠금과 동시성

쉬운 설명

화장실 문에 잠금장치가 있는 이유와 같습니다:

  • 공유 잠금(Shared Lock): 여러 명이 함께 책을 읽을 수 있음 (쓰기는 불가)
  • 배타 잠금(Exclusive Lock): 한 명만 독점해서 쓸 수 있음

잠금을 잘못 관리하면 데드락이 발생합니다. (A가 B를 기다리고, B가 A를 기다리는 상황)

논문 수준 설명

2PL (Two-Phase Locking): 확장 단계(Growing Phase)에서만 잠금 획득, 축소 단계(Shrinking Phase)에서만 해제.

데드락 탐지 및 예방:

  • Wait-for Graph를 이용한 사이클 탐지
  • Timeout 기반 해제
  • 타임스탬프 기반 우선순위 지정

MVCC (Multi-Version Concurrency Control): 데이터의 여러 버전을 유지하여 읽기와 쓰기를 동시에 진행. PostgreSQL, MySQL InnoDB에서 사용.
