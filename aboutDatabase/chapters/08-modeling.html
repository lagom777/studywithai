8장. 데이터 모델링

1. 데이터 모델링이란?

쉬운 설명

건축가가 건물을 짓기 전에 설계도를 그리듯이, 데이터베이스를 만들기 전에 먼저 구조를 그리는 과정입니다.

순서: 생각 → 종이에 그리기 → 테이블 생성 → 데이터 입력

왜 필요한가? 설계도 없이 지으면 나중에 고치기 비쌉니다. DB도 마찬가지입니다.

예: 온라인 쇼핑몰을 만들려면, 고객 정보, 상품, 주문, 배송 정보 등을 어떻게 저장할지 미리 정합니다.

학술 설명

데이터 모델링의 3단계:

  • 개념적 모델링: 비즈니스 요구사항 분석. 엔티티, 속성, 관계 추출.
  • 논리적 모델링: DBMS 독립적인 구조 설계. ER 다이어그램, 정규화 적용.
  • 물리적 모델링: 실제 DBMS에 맞게 구현. 인덱스, 파티셔닝, 저장 공간 고려.

핵심 산출물: ER 다이어그램, 정규화된 스키마, 물리적 설계 문서

표준: IDEF1X, UML, Crow's Foot 표기법

2. ER 다이어그램

쉬운 설명

데이터의 구조를 그림으로 표현하는 방법입니다.

기본 요소:

  • 네모(엔티티): 사람, 상품, 주문 등 데이터의 대상
  • 다이아몬드(관계): 두 엔티티 간의 관계. "구매한다", "소속된다" 등
  • 선: 엔티티와 관계를 연결

예: 고객이 상품을 구매하면, 고객 엔티티에서 구매라는 다이아몬드를 거쳐 상품 엔티티로 연결됩니다.

카디널리티: 1명의 고객이 여러 상품을 살 수 있다. (1:N 관계)

학술 설명

ER 모델의 구성요소:

  • 엔티티(Entity): 데이터의 단위. 약한 엔티티(Weak Entity)는 부모 엔티티에 존재 의존.
  • 속성(Attribute): 엔티티의 특성. 단순/복합, 저장/유도 속성.
  • 관계(Relationship): 엔티티 간의 연관. 재귀 관계, 다항 관계 가능.

표기법:

  • Chen 표기법: 원, 다이아몬드, 선으로 표현. 학술용.
  • Crow's Foot: 까마귀 발처럼 그림. 실무 표준.

카디널리티(Cardinality): 1:1, 1:N, M:N. 최소/최대 카디널리티 지정.

참여 제약(Participation Constraint): 전체 vs 부분 참여

3. 스키마 설계 원칙

쉬운 설명

좋은 테이블을 만들기 위한 5가지 규칙:

  1. 중복 최소화: 같은 데이터를 여러 곳에 저장하지 않기
  2. 명확한 이름: "name"보다 "customer_name", "product_name" 처럼 구체적으로
  3. 적절한 타입: 나이는 INTEGER, 설명은 TEXT, 가격은 DECIMAL
  4. NULL 관리: NULL이 꼭 필요한 경우에만 허용. 기본값 설정
  5. 관계 명시: 어떤 테이블이 어떤 테이블과 관련되는지 FK로 명시

효과: 이 규칙을 따르면 데이터 오류가 줄고, 쿼리가 빨라집니다.

학술 설명

정규화 기반 설계: 1NF → 2NF → 3NF → BCNF로 단계적 정규화

반정규화(Denormalization): 성능을 위해 의도적으로 정규화 규칙 완화

  • 계산된 속성 저장 (합, 평균)
  • 중복된 속성 추가 (빈번한 조인 회피)
  • 컬럼 분할/통합

물리 설계 고려사항:

  • 인덱스 전략: 조회가 많은 컬럼에 인덱스
  • 파티셔닝: 대용량 테이블을 범위/해시로 분할
  • 저장 포맷: ROW vs COLUMN 스토리지

4. 실무 모델링 프로세스

쉬운 설명

4단계 프로세스:

  1. 요구사항 수집: "우리는 뭘 저장해야 할까?" 고객, 상품, 주문...
  2. ER 다이어그램 작성: 위의 2번 섹션처럼 그리기
  3. 테이블 생성: CREATE TABLE 쿼리로 실제 테이블 만들기
  4. 검증: 정말 필요한 데이터를 모두 저장할 수 있는가? 테스트.

팁: 처음엔 최대한 정규화하고, 나중에 성능 이유로 반정규화합니다.

학술 설명

상세 모델링 프로세스:

  1. 요구사항 분석: 비즈니스 프로세스 이해, 데이터 플로우 분석
  2. 도메인 분석: 각 도메인(고객, 상품 등)의 속성, 제약 조건 파악
  3. 개념 모델링: 고수준 ER 다이어그램
  4. 비즈니스 규칙 매핑: "고객은 최대 1명의 판매 담당자를 가진다" 등
  5. 논리 모델링: 정규화, 테이블 추출, 속성 정의
  6. 물리 설계: 인덱스, 파티셔닝, 뷰, 저장 프로시저

검증 방법: 데이터 업데이트 이상(Anomaly) 검사, 쿼리 성능 테스트

하 · 입문

한 줄로 이해하기

데이터 모델링은 비즈니스 요구사항을 데이터로 어떻게 담아낼지 설계하는 일입니다.

왜 중요한가?

잘못된 모델링은 평생 고생의 시작입니다. 첫째, 한 번 출시한 서비스의 테이블 구조를 바꾸려면 데이터 마이그레이션과 다운타임이 필요합니다. 둘째, 모델이 비즈니스 개념과 어긋나면 모든 쿼리가 복잡해집니다. 셋째, 잘 설계된 모델은 새 기능 추가가 쉽고 성능도 좋습니다.

🌱 핵심 용어

  • 엔티티(Entity): 시스템이 다루는 주체. 회원, 상품, 주문 등.
  • 속성(Attribute): 엔티티의 특징. 이름, 가격, 일자.
  • 관계(Relationship): 엔티티 사이의 연결. 회원이 주문한다.
  • ERD: 엔티티-관계 다이어그램, 모델의 그림.
  • 카디널리티: 관계의 수량. 1:1, 1:N, M:N.

개념 풀어쓰기

데이터 모델링은 집을 짓기 전에 설계도를 그리는 일과 비슷합니다. 어디에 방을 두고, 화장실은 몇 개로 하고, 현관은 어디로 낼지를 미리 정해야 짓고 나서 후회하지 않습니다. 데이터 모델링도 마찬가지로, 시스템이 다룰 정보들을 미리 종이에 그려 보면서 무엇이 중요한 엔티티인지, 그것들 사이에 어떤 관계가 있는지를 정리합니다. 일반적으로 개념 모델 → 논리 모델 → 물리 모델 세 단계로 진행됩니다. 개념 모델은 비즈니스 사람과 이야기하기 위한 큰 그림, 논리 모델은 정규화와 무결성을 반영한 설계도, 물리 모델은 특정 DB 제품에 맞춰 인덱스와 자료형까지 결정한 최종 도면입니다. ERD라는 다이어그램으로 표현하면 한눈에 전체 구조가 보여 협업이 쉬워집니다. 가장 중요한 마음가짐은 "데이터는 오래 산다"는 사실입니다. 코드는 자주 갈아엎어도, 데이터는 회사가 쌓아 온 자산이라 함부로 못 바꿉니다. 그래서 모델링은 신중하게, 그리고 비즈니스 도메인 전문가와 함께 해야 합니다.

💡 학습 TIP: dbdiagram.io나 drawio로 ERD를 직접 그려 보세요. 머릿속에만 있는 그림은 협업이 안 됩니다.

요약

  • 모델링은 비즈니스를 데이터로 표현하는 작업.
  • 엔티티, 속성, 관계가 기본 요소.
  • 개념→논리→물리 세 단계로 진행.
  • ERD로 시각화해 협업.
  • 나중에 바꾸기 어려우니 신중하게.
중 · 응용

이론 깊이

데이터 모델링은 OLTP와 OLAP에서 접근법이 다릅니다. OLTP는 정규화 기반의 ER 모델로 트랜잭션 무결성을 우선하고, OLAP은 차원 모델링(Dimensional Modeling)으로 분석 쿼리 성능을 우선합니다. Kimball의 스타 스키마는 사실 테이블(Fact)과 차원 테이블(Dimension)로 구성되며, 스노우플레이크 스키마는 차원을 다시 정규화한 변형입니다. SCD(Slowly Changing Dimension)는 차원 변경을 추적하는 방식으로 Type 1(덮어쓰기), Type 2(이력 보존), Type 3(이전값 컬럼) 등이 있습니다. 도메인 주도 설계(DDD)의 애그리게이트는 일관성 경계를 정해 트랜잭션 단위와 NoSQL 도큐먼트 경계를 도출합니다. 식별자 전략은 자연키 vs 대리키, 정수 vs UUID, ULID vs Snowflake ID 등 다양하며, 분산 환경에서는 노드 충돌을 피하기 위해 시간+노드 비트 조합이 표준입니다. 시간성 데이터(이력)는 valid_from/valid_to 컬럼이나 별도 이력 테이블, 양방향 시간(bitemporal) 모델로 표현합니다.

실전 SQL/쿼리

-- 스타 스키마 예시
CREATE TABLE dim_customer (
  customer_sk   BIGINT PRIMARY KEY,  -- 대리키
  customer_id   VARCHAR(20),         -- 자연키
  name          VARCHAR(100),
  region        VARCHAR(50),
  valid_from    DATE,
  valid_to      DATE,
  is_current    BOOLEAN
);

CREATE TABLE dim_product (
  product_sk    BIGINT PRIMARY KEY,
  product_id    VARCHAR(30),
  category      VARCHAR(50)
);

CREATE TABLE fact_sales (
  sale_id       BIGINT PRIMARY KEY,
  customer_sk   BIGINT REFERENCES dim_customer(customer_sk),
  product_sk    BIGINT REFERENCES dim_product(product_sk),
  date_sk       INT,
  quantity      INT,
  amount        DECIMAL(12,2)
);

-- 분석 쿼리
SELECT  d.region, p.category, SUM(f.amount) AS revenue
FROM    fact_sales f
JOIN    dim_customer d ON f.customer_sk = d.customer_sk
JOIN    dim_product  p ON f.product_sk  = p.product_sk
WHERE   d.is_current
GROUP   BY d.region, p.category;

예제

💻 예제 1: 이커머스 도메인

회원·상품·주문·결제·배송을 핵심 애그리게이트로 분리합니다. 주문은 결제와 배송의 식별자를 참조하지만 트랜잭션 경계는 분리해 마이크로서비스로 발전 가능하게 둡니다.

💻 예제 2: SaaS 멀티테넌시

tenant_id를 거의 모든 테이블의 첫 컬럼으로 두고, 인덱스와 RLS(Row Level Security)로 격리합니다. 큰 테넌트는 별도 스키마로 분리하는 하이브리드 모델이 자주 쓰입니다.

실무 응용

대기업 데이터 웨어하우스는 Inmon식 정규화 모델 위에 Kimball식 데이터마트를 두는 하이브리드가 일반적입니다. 또한 데이터 볼트(Data Vault) 모델링은 변경 추적과 감사 가능성에 강점이 있어 금융권에서 채택됩니다. 도메인 주도 설계는 마이크로서비스 분해와 함께 사용되어 서비스별 자체 DB를 갖는 데이터 분산 패턴으로 이어집니다.

⚠️ 자주 하는 실수: 모든 곳에 UUID를 박으면 인덱스가 비대해지고 클러스터드 인덱스(InnoDB) 환경에서 페이지 분할이 잦아집니다. 또 SCD 없이 차원에 덮어쓰기만 하면 과거 매출 분석이 왜곡됩니다.
상 · 심화

전문가 레벨

데이터 볼트 2.0은 Hub(비즈니스 키), Link(관계), Satellite(속성·이력)로 이력과 무결성을 분리해 모델링합니다. 변경 데이터 캡처(CDC)는 WAL이나 binlog를 읽어 운영 DB의 변화를 분석 시스템으로 전파하며, Debezium·Maxwell·AWS DMS가 대표 도구입니다. 이벤트 소싱은 상태 대신 사건의 로그를 1차 모델로 두고 읽기 모델을 투영(Projection)으로 구성하며, CQRS와 결합해 읽기·쓰기 모델을 명확히 분리합니다. 시간성 모델은 transaction time과 valid time을 동시에 추적하는 bitemporal 형태로 발전하며, 회계와 보험에서 표준입니다. NoSQL 모델링에서는 액세스 패턴 우선 설계(특히 DynamoDB의 single-table design)가 핵심이며, GSI/LSI와 sparse index, denormalization 패턴이 결합됩니다. 그래프 모델링은 노드와 관계의 라벨, 속성을 어떻게 분포시키느냐가 쿼리 성능과 직결되며, 추천·사기 탐지·지식 그래프에서 핵심입니다. 또한 LLM RAG에는 정규화된 사실 테이블 위에 임베딩 컬럼이나 벡터 인덱스를 결합한 하이브리드 모델이 자리 잡고 있습니다.

-- 데이터 볼트 핵심 구조 예시
CREATE TABLE hub_customer (
  customer_hk   CHAR(64) PRIMARY KEY,   -- 해시 키
  customer_bk   VARCHAR(50) NOT NULL,   -- 비즈니스 키
  load_dts      TIMESTAMP NOT NULL,
  record_source VARCHAR(50)
);

CREATE TABLE sat_customer_profile (
  customer_hk   CHAR(64),
  load_dts      TIMESTAMP,
  hash_diff     CHAR(64),
  name          VARCHAR(100),
  email         VARCHAR(100),
  PRIMARY KEY (customer_hk, load_dts)
);

UPDATE contract_history SET txn_to = NOW()
  WHERE contract_id = ? AND txn_to = 'infinity'
    AND (valid_from, valid_to) OVERLAPS (?, ?);

최신 동향

데이터 메시 아키텍처는 도메인 팀이 데이터 제품을 소유하고 표준 인터페이스로 발행하는 분산 데이터 운영 모델을 제시합니다. 또한 dbt와 Iceberg/Delta는 SQL 중심의 모델링을 코드화·버전관리하는 표준이 되었으며, 시맨틱 레이어(Cube, Looker)는 비즈니스 메트릭을 코드로 정의해 일관성을 보장합니다. 그래프와 벡터, 시계열을 통합 처리하는 멀티모달 DB가 새로운 흐름입니다.
