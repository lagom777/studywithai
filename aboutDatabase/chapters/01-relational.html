1장. 관계형 데이터베이스 기초

1.1 관계형 모델이란?

🎯 쉽게 이해하기

관계형 데이터베이스는 데이터를 표(테이블)로 관리하는 방식입니다. 마치 엑셀 스프레드시트처럼 행과 열로 구성되어 있어요.

예시: 학교의 학생 명부

학번 이름 학년 전공
001 김철수 2학년 컴퓨터공학
002 이영희 3학년 경영학

이런 식으로 데이터를 구조화하여 효율적으로 관리합니다.

📚 학술 정의

관계형 데이터 모델은 E.F. Codd가 1970년 "A Relational Model of Data for Large Shared Data Banks"에서 제안했습니다.

핵심 개념:

  • 릴레이션(Relation): 수학의 집합론에 기반한 2차원 표. R ⊆ D₁ × D₂ × ... × Dₙ
  • 튜플(Tuple): 표의 한 행(row). 속성값들의 순서쌍 집합
  • 속성(Attribute): 표의 한 열(column). 특정 도메인의 값을 가짐
  • 도메인(Domain): 속성이 취할 수 있는 값들의 집합 (예: 학번 도메인 = {001, 002, ...})
  • 카디날리티(Cardinality): 릴레이션의 행(튜플) 개수
  • 디그리(Degree): 릴레이션의 열(속성) 개수

관계형 모델의 강점은 수학적 엄밀성과 쿼리 최적화에 있습니다.

1.2 핵심 개념: 테이블, 행, 열

🎯 쉽게 이해하기

테이블은 행(row)과 열(column)로 이루어진 2차원 구조입니다.

  • 열(Column): 특정 속성을 나타냅니다. 예) "이름" 열, "학번" 열
  • 행(Row): 하나의 객체에 대한 정보입니다. 예) 학생 한 명의 모든 정보
  • 셀: 행과 열이 만나는 지점의 하나의 값

학생 테이블 예시:

학번 이름 학년 전공
001 김철수 2 컴퓨터공학
002 이영희 3 경영학
003 박민준 1 수학

📚 형식적 정의

관계 스키마(Relation Schema): R(A₁:D₁, A₂:D₂, ..., Aₙ:Dₙ) 여기서 각 Aᵢ는 속성명, Dᵢ는 도메인입니다.

관계 인스턴스(Relation Instance): 스키마에 대한 실제 데이터 튜플들의 집합.

관계대수(Relational Algebra) 기본 연산자:

  • 선택(Selection, σ): σ_{condition}(R) - 조건을 만족하는 행 선택. 예) σ_{grade>2}(Students)
  • 프로젝션(Projection, π): π_{A₁,A₂,...}(R) - 특정 속성만 선택. 예) π_{name,dept}(Students)
  • 자연 조인(Natural Join, ⋈): R₁ ⋈ R₂ - 공통 속성을 기준으로 두 릴레이션 결합
  • 카티션 곱(Cartesian Product, ×): R₁ × R₂ - 모든 조합의 곱집합
  • 합집합(Union, ∪): R₁ ∪ R₂ - 두 릴레이션의 모든 행
  • 차집합(Difference, −): R₁ − R₂ - R₁에는 있고 R₂에 없는 행
  • 교집합(Intersection, ∩): R₁ ∩ R₂ = R₁ − (R₁ − R₂)

1.3 키(Key)의 종류

🎯 쉽게 이해하기

학생들을 구분하기 위해 학번이 필요하죠. 이것이 기본 키(Primary Key)입니다.

키의 종류를 친구관계로 비유하면:

  • 기본 키(PK): 각 사람의 주민등록번호처럼 유일하게 구분해주는 것
  • 외래 키(FK): 친구 목록에서 친구가 누구인지 가리키는 포인터. 다른 테이블과 연결하는 것

예시:

Students 테이블 - 학번이 PK (각 학생을 유일하게 식별)

Enrollments 테이블 - 학번과 과목코드가 FK (학생과 강좌를 연결)

📚 형식적 정의

K를 릴레이션 R의 속성들의 부분집합이라 하자.

  • 슈퍼키(Superkey): K의 어떤 두 튜플도 모든 속성에서 동일한 값을 가지지 않음. ∀t₁,t₂ ∈ r, t₁ ≠ t₂ ⟹ t₁[K] ≠ t₂[K]
  • 후보 키(Candidate Key): 최소 슈퍼키. 어떤 진부분집합도 슈퍼키가 아님
  • 기본 키(Primary Key): DBA가 선택한 후보 키. NULL이 될 수 없음(NOT NULL, UNIQUE)
  • 대체 키(Alternate Key): 선택되지 않은 후보 키들
  • 외래 키(Foreign Key): 릴레이션 R₁의 속성 FK는 다른 릴레이션 R₂의 기본 키를 참조. ∀t ∈ R₁, t[FK] ∈ {t'[PK] : t' ∈ R₂} ∪ {NULL}

무결성 제약조건(Integrity Constraints):

  • 엔티티 무결성(Entity Integrity): 기본 키는 NULL일 수 없음
  • 참조 무결성(Referential Integrity): 외래 키는 참조 대상의 기본 키 값이거나 NULL
  • 도메인 무결성(Domain Integrity): 각 속성은 도메인 내의 값만 가짐

1.4 관계 연산

🎯 쉽게 이해하기

데이터베이스에서 데이터를 꺼낼 때 사용하는 기본 동작들입니다.

  • 필터링(σ - Selection): "학년이 2학년인 학생만 보여줘" → 특정 행만 추출
  • 특정 열만 보기(π - Projection): "학번과 이름만 보여줘" → 특정 열만 추출
  • 두 표 합치기(⋈ - Join): 학생 정보와 수강 정보를 학번으로 묶어서 보여주기
  • 합치기(∪ - Union): 컴퓨터공학 학생들과 경영학 학생들 합치기

실제 예시:

학생 테이블에서 "2학년 학생의 이름과 전공만 보고 싶다" →

π_{name,dept}(σ_{grade=2}(Students))

📚 관계대수 상세

기본 집합 연산:

  • 합집합(∪): r₁ ∪ r₂ = {t | t ∈ r₁ ∨ t ∈ r₂}. 호환성: 같은 스키마 필요
  • 교집합(∩): r₁ ∩ r₂ = r₁ − (r₁ − r₂)
  • 차집합(−): r₁ − r₂ = {t | t ∈ r₁ ∧ t ∉ r₂}
  • 카티션 곱(×): r₁ × r₂ = {(t₁, t₂) | t₁ ∈ r₁ ∧ t₂ ∈ r₂}. 결과 도수 = |r₁| × |r₂|

관계식 연산:

  • 선택(σ): σ_F(r) = {t | t ∈ r ∧ F(t) = true}. 여기서 F는 선택 조건식
  • 프로젝션(π): π_{A₁,...,Aₖ}(r) = {[t[A₁],...,t[Aₖ]] | t ∈ r}. 중복 제거됨
  • 조인(⋈): r₁ ⋈_F r₂ = σ_F(r₁ × r₂). 특수한 경우가 내부 조인, 외부 조인
  • 나누기(÷): r₁ ÷ r₂ = {t | ∀t' ∈ r₂, (t ⊕ t') ∈ r₁}. 복잡한 쿼리 표현에 사용
  • 이름변경(ρ): ρ_{S(B₁,...,Bₖ)}(r) - 릴레이션 이름이나 속성명 변경

동치 변환 규칙(Equivalence Rules):

  • σ_{F₁∧F₂}(r) ≡ σ_{F₁}(σ_{F₂}(r)) (선택의 분해)
  • π_{A}(σ_F(r)) ≡ σ_F(π_{A}(r)) (선택과 프로젝션의 순서교환은 조건부)
  • σ_{F₁}(r₁ ⋈ r₂) ≡ σ_{F₁}(r₁) ⋈ r₂ (선택 연산 푸시다운)

한 줄로 이해하기

관계형 데이터베이스는 표(테이블) 형태로 데이터를 정리하고, 표끼리 연결해서 의미 있는 정보를 꺼내쓰는 도구입니다.

왜 중요한가?

일상에서 관계형 DB가 쓰이지 않는 곳을 찾기가 더 어렵습니다. 첫째, 은행 계좌 시스템은 고객 표, 계좌 표, 거래 표를 연결해 잔액을 정확히 계산합니다. 둘째, 온라인 쇼핑몰은 회원 표와 주문 표, 상품 표를 묶어 "내가 산 물건"을 보여줍니다. 셋째, 학교 성적 시스템은 학생, 과목, 성적 표를 관계로 묶어 통지표를 생성합니다.

🌱 핵심 용어

  • 테이블(Table): 행과 열로 구성된 표. 엑셀 시트와 비슷합니다.
  • 행(Row/Record): 하나의 데이터 묶음. 예: 한 명의 학생 정보.
  • 열(Column/Attribute): 같은 종류의 값을 모은 칸. 예: 이름, 나이.
  • 기본키(Primary Key): 각 행을 유일하게 구분하는 값. 학번, 주민번호 같은 것.
  • 외래키(Foreign Key): 다른 표의 기본키를 가리키는 연결고리.

개념 풀어쓰기

관계형 데이터베이스를 도서관에 비유해 봅시다. 도서관에는 책 목록, 회원 목록, 대출 기록이라는 세 개의 장부가 있습니다. 각각의 장부는 잘 정리된 표입니다. 책 장부에는 책마다 고유한 청구기호가 있고, 회원 장부에는 회원증 번호가 있죠. 대출 기록에는 누가 어떤 책을 빌렸는지가 적혀 있는데, 회원증 번호와 청구기호로 책과 회원을 연결합니다. 만약 누군가 "홍길동이 빌린 책을 알려줘"라고 하면 회원 장부에서 홍길동의 번호를 찾고, 그 번호로 대출 기록을 뒤져 청구기호를 얻은 뒤, 마지막으로 책 장부에서 제목을 찾아냅니다. 이런 과정을 SQL이라는 언어로 한 줄에 끝낼 수 있게 해 주는 것이 바로 관계형 데이터베이스입니다. 모든 데이터가 표 안에 들어 있고, 표와 표 사이의 연결고리(외래키)가 있어서 흩어진 정보를 모아 답을 만들어 냅니다. 무엇보다 중요한 점은, 같은 정보를 두 번 적지 않는다는 약속입니다. 회원 이름이 바뀌어도 회원 장부 한 곳만 고치면 모든 대출 기록에 반영됩니다.

💡 학습 TIP: 처음 공부할 때는 무조건 종이에 표를 그려 보세요. 학생 표, 과목 표, 수강 표 세 개를 그리고 화살표로 연결하면 외래키 개념이 쉽게 이해됩니다.

요약

  • 관계형 DB는 데이터를 표(테이블)로 저장합니다.
  • 각 행은 기본키로 구분됩니다.
  • 외래키로 표와 표를 연결해 흩어진 정보를 묶습니다.
  • 같은 데이터를 중복 저장하지 않는 것이 원칙입니다.
  • SQL이라는 언어로 데이터를 다룹니다.
중 · 응용

이론 깊이

관계형 모델은 1970년 E.F. Codd가 발표한 수학적 집합 이론에 기반합니다. 각 테이블은 하나의 릴레이션(관계)이며, 행은 튜플, 열은 속성으로 불립니다. 관계대수와 관계해석이라는 두 가지 형식 언어가 SQL의 이론적 기반이 됩니다. 무결성 제약은 크게 개체 무결성(기본키는 NULL이 될 수 없음), 참조 무결성(외래키가 가리키는 값이 실제 존재해야 함), 도메인 무결성(컬럼 값이 정의된 자료형 범위 안에 있어야 함)으로 나뉩니다. RDBMS는 이러한 무결성을 자동으로 검사하여 잘못된 데이터를 막아 줍니다. 또한 ACID 특성을 통해 트랜잭션이 안전하게 처리되도록 보장합니다. 카디널리티(1:1, 1:N, M:N)에 따라 테이블 간 관계가 정의되며, M:N 관계는 별도의 연결 테이블로 풀어 표현하는 것이 표준 방식입니다.

실전 SQL/쿼리

-- 학생, 과목, 수강 테이블 설계 후 조회
CREATE TABLE student (
  student_id   INT PRIMARY KEY,
  name         VARCHAR(50) NOT NULL,
  enrolled_at  DATE
);

CREATE TABLE course (
  course_id    INT PRIMARY KEY,
  title        VARCHAR(100) NOT NULL,
  credits      INT CHECK (credits BETWEEN 1 AND 4)
);

CREATE TABLE enrollment (
  student_id   INT,
  course_id    INT,
  grade        CHAR(1),
  PRIMARY KEY (student_id, course_id),
  FOREIGN KEY (student_id) REFERENCES student(student_id),
  FOREIGN KEY (course_id)  REFERENCES course(course_id)
);

-- 특정 학생이 수강한 과목과 학점 합계
SELECT s.name, SUM(c.credits) AS total_credits
FROM   student s
JOIN   enrollment e ON s.student_id = e.student_id
JOIN   course     c ON e.course_id  = c.course_id
WHERE  s.student_id = 1001
GROUP  BY s.name;

예제

💻 예제 1: 도서관 시스템

회원 테이블, 도서 테이블, 대출 테이블을 만들고 외래키로 연결합니다. 한 회원이 여러 책을 빌릴 수 있는 1:N 관계를 대출 테이블로 표현하며, 반납일 NULL은 "아직 반납 안 함"을 의미합니다.

💻 예제 2: 쇼핑몰 주문

회원, 상품, 주문, 주문상세 네 테이블로 M:N 관계를 풀어 표현합니다. 주문상세는 주문 번호와 상품 번호의 복합키를 갖고, 수량과 단가를 별도 컬럼으로 둡니다. 가격 변동에도 과거 주문의 단가가 보존되도록 설계합니다.

실무 응용

대형 ERP에서는 수백 개 테이블이 외래키로 얽혀 있으며, 카탈로그·재고·결제·배송 도메인이 각각의 스키마를 갖습니다. 또한 은행 코어뱅킹 시스템은 계좌·고객·거래 세 큰 테이블을 중심으로 관계가 짜여 있고, 일별 마감 배치가 트랜잭션 무결성을 점검합니다. SaaS 플랫폼은 멀티 테넌시를 위해 tenant_id 컬럼을 거의 모든 테이블에 두는 패턴이 일반적입니다.

⚠️ 자주 하는 실수: 외래키 제약을 빠뜨리고 운영하다 데이터 정합성이 깨지는 경우가 많습니다. 또 NULL이 허용되는 컬럼을 기본키로 쓰려다 오류를 보는 사례도 흔합니다.
상 · 심화

전문가 레벨

관계형 데이터베이스의 내부는 스토리지 엔진, 쿼리 옵티마이저, 트랜잭션 매니저, 락 매니저, 로그 매니저로 구성됩니다. InnoDB 같은 스토리지 엔진은 클러스터드 인덱스 구조를 사용해 기본키 순서로 데이터 페이지를 물리적으로 정렬하며, 페이지는 보통 16KB 단위로 관리됩니다. 옵티마이저는 통계(히스토그램, 카디널리티)를 기반으로 비용 모델을 세워 가장 저렴한 실행계획을 선택합니다. 트랜잭션은 WAL(Write-Ahead Logging)을 통해 ACID를 구현하며, REDO/UNDO 로그가 충돌 복구의 핵심입니다. 동시성 제어는 락 기반(2PL) 또는 MVCC(다중 버전 동시성 제어)로 구현되며, PostgreSQL과 Oracle은 MVCC를 표준으로 채택합니다. 분산 환경에서는 행 수준 락 외에도 글로벌 트랜잭션과 2PC가 필요하지만, 성능 손실 때문에 대부분 보상 트랜잭션 패턴으로 대체합니다. 또한 파티셔닝과 샤딩으로 단일 노드의 한계를 넘어서며, 파티션 프루닝과 파티션-와이즈 조인 같은 고급 기법이 옵티마이저에 의해 자동 적용됩니다.

고급 패턴

-- 실행계획으로 인덱스 사용 여부 확인
EXPLAIN (ANALYZE, BUFFERS)
SELECT s.name, COUNT(*) AS cnt
FROM   student s
JOIN   enrollment e ON s.student_id = e.student_id
WHERE  s.enrolled_at >= '2024-01-01'
GROUP  BY s.name
HAVING COUNT(*) >= 5;

-- 윈도우 함수로 학생별 누적 학점 추적
SELECT student_id, course_id,
       SUM(credits) OVER (
         PARTITION BY student_id
         ORDER BY taken_at
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS cumulative_credits
FROM   enrollment_log;

최신 동향

NewSQL(CockroachDB, Spanner, TiDB)은 관계형 모델과 ACID를 유지하면서 수평 확장을 가능하게 만든 분산 RDBMS 흐름입니다. 또한 PostgreSQL은 pgvector 확장으로 벡터 검색을 통합해 LLM RAG 파이프라인의 기반 저장소가 되었습니다. Time-series DB(TimescaleDB)는 PostgreSQL 위에서 시계열 최적화를 더해 IoT·관측 영역을 흡수하고 있으며, HTAP(Hybrid Transactional/Analytical) 아키텍처는 OLTP와 OLAP 경계를 허물고 있습니다.

관련 분야

  • 분산 시스템: 합의 알고리즘(Raft, Paxos)이 복제와 리더 선출의 기반이 됩니다.
  • 스토리지 엔진: LSM 트리(RocksDB)와 B+ 트리의 트레이드오프 분석.
  • 쿼리 옵티마이저: 비용 기반 최적화, 조인 순서 결정 알고리즘.
