4장. 인덱스와 최적화

1. 인덱스란?

쉬운 설명

인덱스는 책의 목차나 색인과 같습니다. 전체 페이지를 하나하나 넘기지 않고, 원하는 내용을 바로 찾을 수 있게 해줍니다.

예를 들어, 1000명의 학생 정보에서 "이름이 김철수인 학생"을 찾을 때:

  • 인덱스 없음: 1000명을 처음부터 끝까지 하나씩 확인 → 평균 500번
  • 인덱스 있음: 알파벳 목차로 "김"을 찾으면 → 약 10번

논문 수준 설명

정의: 인덱스(Index)는 데이터 테이블의 특정 컬럼에 대해 별도의 데이터 구조(주로 B-Tree)를 유지하여 검색 성능을 개선하는 메커니즘입니다.

시간복잡도:

  • Full Scan: O(n) - n은 테이블의 행 수
  • B-Tree 인덱스: O(log n)

비용: 인덱스 유지에 따른 쓰기 작업 오버헤드, 저장 공간 증가

2. B-Tree와 B+Tree

쉬운 설명

B-Tree는 도서관의 분류 체계와 같습니다:

  • 1층: 큰 카테고리 (문학, 과학, 역사...)
  • 2층: 작은 카테고리 (한국 문학, 외국 문학...)
  • 3층: 원하는 책을 찾음

B+Tree는 B-Tree의 개선판으로, 리프 노드들이 서로 연결되어 있어서 범위 검색이 더 빠릅니다.

논문 수준 설명

B-Tree 특성:

  • 차수(Order) m인 B-Tree는 최대 m개의 자식을 가짐
  • 모든 리프 노드는 같은 깊이(균형 유지)
  • 삽입/삭제 시 분할(split)과 병합(merge) 연산 수행

B+Tree vs B-Tree: B+Tree는 내부 노드에 데이터를 저장하지 않고 인덱스만 저장, 리프 노드에만 실제 데이터 저장. 리프 노드를 연결 리스트로 연결하여 범위 검색 시 O(log n) 후 선형 순회.

3. 쿼리 실행 계획(EXPLAIN)

쉬운 설명

내비게이션이 목적지로 가는 최적 경로를 미리 알려주듯이, 데이터베이스도 SQL 쿼리를 실행하기 전에 최적 경로를 계획합니다.

이 계획을 확인하려면 SQL 앞에 EXPLAIN을 붙입니다:

EXPLAIN SELECT * FROM users WHERE age > 30;

이 명령은 DB가 어떤 경로로 데이터를 찾을 것인지 보여줍니다.

논문 수준 설명

쿼리 옵티마이저: 가능한 모든 실행 계획의 비용을 추정하고 최소 비용 계획을 선택합니다.

비용 기반 최적화(Cost-Based Optimization):

  • Full Scan: I/O 비용 = 전체 페이지 수
  • Index Scan: I/O 비용 = 인덱스 높이 + 결과 페이지 수
  • Index Seek: 가장 효율적 (인덱스로 범위를 좁힌 후 검색)

통계 정보: 카디널리티(Cardinality), 페이지 수, 히스토그램을 기반으로 계산

4. 인덱스 설계 전략

쉬운 설명

어디에 인덱스를 걸어야 빨리 찾을 수 있을까?

  • WHERE 절: 가장 자주 검색하는 컬럼
  • JOIN 절: 두 테이블을 연결하는 컬럼
  • ORDER BY 절: 정렬하는 컬럼

예: "users 테이블에서 나이 > 30인 사람을 찾는다" → age 컬럼에 인덱스!

논문 수준 설명

복합 인덱스(Composite Index): 여러 컬럼을 함께 인덱싱. 순서가 중요 (WHERE a AND b vs WHERE b AND a에서 성능 차이)

커버링 인덱스(Covering Index): 인덱스만으로 쿼리가 완성되어 테이블 접근이 불필요

선택도(Selectivity): 컬럼의 고유값 비율. 선택도가 높을수록 인덱싱 효과 큼

카디널리티(Cardinality): 컬럼의 서로 다른 값의 개수. 낮은 카디널리티(성별 등)는 인덱싱 효율 낮음
