7장. 분산 데이터베이스

1. 분산 DB란?

쉬운 설명

하나의 데이터베이스를 여러 컴퓨터에 나눠 저장하는 것입니다. 예를 들어, 당신이 운영하는 온라인 쇼핑몰이 매우 커져서 하나의 서버로는 감당할 수 없다면, 데이터를 여러 서버에 분산시킵니다.

장점: 한 서버가 망가져도 다른 서버가 서비스를 유지합니다. 동시에 더 많은 사람들을 처리할 수 있습니다.

비유: 하나의 도서관이 너무 커져서 여러 지점을 열어 책을 분산 보관하는 것과 같습니다.

학술 설명

분산 데이터베이스 시스템(DDBMS)은 물리적으로 떨어진 여러 노드에 데이터가 분산되어 저장되며, 네트워크를 통해 통신하는 시스템입니다.

확장성 전략:

  • 수평 확장(Scale-out): 더 많은 서버를 추가. 분산 DB의 목표.
  • 수직 확장(Scale-up): 기존 서버의 성능을 높임. 비용이 높음.

핵심 과제: 분산된 데이터의 일관성, 가용성, 신뢰성 유지

2. CAP 정리

쉬운 설명

분산 DB는 3가지 중 최대 2가지만 완벽히 보장할 수 있습니다:

  • 일관성(Consistency): 모든 노드가 같은 데이터를 본다
  • 가용성(Availability): 언제든 응답한다
  • 분할내성(Partition Tolerance): 네트워크 끊김을 견딘다

삼각형 비유: 삼각형 중에 2개의 꼭짓점만 잡을 수 있다면, 하나는 버려야 한다는 뜻입니다.

실제: 은행(일관성 중심) vs SNS(가용성 중심)

학술 설명

Brewer의 CAP 정리(2000): 분산 시스템은 C, A, P 중 최대 2개만 동시에 만족할 수 없다.

Gilbert & Lynch의 증명(2002): 네트워크 분할 상황에서 일관성과 가용성을 동시에 보장 불가능함을 증명.

실무 분류:

  • CA: 전통적 RDBMS (네트워크 분할 가정하지 않음)
  • AP: Dynamo, Cassandra (최종 일관성)
  • CP: MongoDB, HBase (가용성 포기)

PACELC 확장: 네트워크가 정상일 때 Latency-Consistency 트레이드오프도 고려

3. 샤딩(Sharding)

쉬운 설명

데이터를 규칙에 따라 여러 서버로 분산하는 방법입니다.

도서관 비유: 도서관 분관을 만들 때, A~M 성씨 책은 강남지점에, N~ㅎ 성씨 책은 강북지점에 보관합니다.

장점: 각 서버가 작은 데이터만 관리하므로 빠릅니다.

문제: 한 서버에 데이터가 몰릴 수 있습니다 (핫스팟).

학술 설명

샤딩 전략:

  • 해시 샤딩: hash(key) % N으로 샤드 결정. 균등 분산.
  • 레인지 샤딩: 범위로 분할. 범위 쿼리에 유리하지만 핫스팟 발생 가능.
  • 디렉토리 샤딩: 매핑 테이블 사용. 유연하지만 조회 비용 증가.

컨시스턴트 해싱: 노드 추가/제거 시 대부분의 데이터 이동을 피함.

핫스팟 문제: 일부 샤드에 데이터가 집중. 해결책: 샤드 재분배, 마이크로 샤딩

4. 복제(Replication)

쉬운 설명

같은 데이터를 여러 서버에 복사해 저장하는 방법입니다.

도서관 비유: 인기 있는 책은 여러 도서관에 비치합니다. 하나가 불타도 다른 곳에 책이 남아있습니다.

장점: 서버 장애에 강합니다. 읽기 성능도 향상됩니다.

문제: 모든 복제본을 최신으로 유지해야 합니다.

학술 설명

복제 토폴로지:

  • 마스터-슬레이브: 주 노드가 쓰기, 부 노드가 읽기. 단순하지만 주 노드가 병목.
  • 멀티마스터: 모든 노드가 쓰기 가능. 충돌 해결이 복잡.
  • 피어-투-피어: 모든 노드가 동등. 가장 탄력적.

일관성 모델:

  • 강한 일관성: 모든 읽기가 최신 데이터 보장. 느림.
  • 최종 일관성(Eventual Consistency): 결국 일관성 도달. 빠르지만 임시 불일치 발생.

쿼럼(Quorum) 기반 복제: N개 복제본 중 W개 쓰기, R개 읽기로 일관성 조정

하 · 입문

한 줄로 이해하기

분산 데이터베이스는 데이터를 여러 컴퓨터(노드)에 나눠 두고 함께 일하게 만드는 시스템입니다.

왜 중요한가?

한 대의 서버가 감당하기 어려운 규모를 다루려면 분산이 필수입니다. 첫째, 글로벌 SNS는 전 세계 사용자에게 빠른 응답을 주려고 데이터센터를 여러 대륙에 둡니다. 둘째, 한 대가 고장 나도 다른 노드가 대신 응답해 무중단 운영이 가능합니다. 셋째, 사용량이 늘면 노드를 추가하는 식으로 수평 확장할 수 있습니다.

🌱 핵심 용어

  • 샤딩(Sharding): 데이터를 조각으로 나눠 여러 노드에 분산.
  • 복제(Replication): 같은 데이터를 여러 노드에 복사.
  • 리더/팔로워: 쓰기를 받는 노드와 그것을 따라가는 노드.
  • 일관성: 어디서 읽어도 같은 값이 보이는 정도.
  • 가용성: 시스템이 요청을 응답할 수 있는 능력.

개념 풀어쓰기

분산 데이터베이스를 이해하려면 도서관 체인을 떠올리면 됩니다. 한 도시에만 도서관이 있으면 멀리 사는 사람은 책을 빌리기 어렵습니다. 그래서 같은 책을 여러 도서관에 복사해 두면(복제) 가까운 도서관에서 빌릴 수 있습니다. 또 책이 너무 많아 한 건물에 다 못 넣으면 분야별로 나눠 다른 건물에 분산해 둡니다(샤딩). 문제는 한 도서관에서 책 내용을 수정하면 다른 도서관에도 그 변경이 전파되어야 한다는 점입니다. 즉시 전파하면 정확하지만 느려지고, 천천히 전파하면 빠르지만 잠깐 다른 값이 보일 수 있습니다. 이 트레이드오프를 일관성 모델이라고 합니다. 또한 도서관 한 곳이 정전으로 닫혀도 다른 도서관이 대신 봉사해야 하는데, 이를 가용성이라고 합니다. 분산 DB는 이런 일관성·가용성·분할허용성 셋을 균형 있게 다루는 기술의 총합이며, 큰 회사일수록 자체적인 분산 DB 운영 노하우가 핵심 자산이 됩니다.

💡 학습 TIP: 처음에는 MySQL 마스터-슬레이브 복제를 직접 구성해 보세요. 마스터에 INSERT한 데이터가 슬레이브에 반영되는 시간을 측정하면 복제 지연이라는 개념이 손에 잡힙니다.

요약

  • 분산 DB는 여러 노드로 데이터를 처리합니다.
  • 샤딩은 분할, 복제는 사본을 만듭니다.
  • 일관성과 가용성은 종종 트레이드오프 관계.
  • 장애 시 자동 복구가 큰 장점.
  • 운영 복잡도가 단일 노드보다 훨씬 높습니다.
중 · 응용

이론 깊이

CAP 정리(Brewer)는 네트워크 분할 상황에서 일관성과 가용성 중 하나를 포기해야 한다는 명제입니다. PACELC는 분할이 없을 때도 지연(Latency)과 일관성 중 하나를 골라야 한다는 확장 모델입니다. 일관성 모델은 강한 일관성, 선형성(Linearizability), 순차 일관성, 인과 일관성, 결과적 일관성으로 단계가 있습니다. 복제 토폴로지는 단일 리더(Primary-Replica), 다중 리더(Multi-Master), 리더리스(Dynamo 스타일)로 나뉘며, 각각 충돌 해소 전략이 다릅니다. 샤딩은 범위 기반(Range), 해시 기반(Hash), 디렉토리 기반(Lookup)으로 나뉘며, 핫스팟을 피하고 리밸런싱 비용을 줄이는 설계가 핵심입니다. 합의 알고리즘 Raft와 Paxos는 분산 시스템에서 단일 진실의 원천을 정하는 데 사용되며, etcd, Consul, CockroachDB가 Raft 기반입니다. 분산 트랜잭션은 2PC와 Saga가 대표적이며, 현대 NewSQL은 외부 일관성을 제공하기 위해 정밀한 시계 동기화나 HLC를 활용합니다.

실전 SQL/쿼리

-- MySQL 비동기 복제 설정 핵심 부분
-- (Primary)
CHANGE REPLICATION SOURCE TO
  SOURCE_HOST = 'primary.example.com',
  SOURCE_USER = 'repl_user',
  SOURCE_AUTO_POSITION = 1;
START REPLICA;
SHOW REPLICA STATUS\G

-- 복제 지연 모니터링
SELECT TIME_TO_SEC(TIMEDIFF(NOW(), Last_Update))
FROM   performance_schema.replication_applier_status_by_worker;

-- PostgreSQL 논리 복제 (CDC 기반)
CREATE PUBLICATION pub_orders FOR TABLE orders;
-- 구독자 측
CREATE SUBSCRIPTION sub_orders
  CONNECTION 'host=primary dbname=app user=repl'
  PUBLICATION pub_orders;

예제

💻 예제 1: 글로벌 사용자 분산

아시아·유럽·미주 세 리전에 클러스터를 두고, 사용자 ID 해시로 1차 라우팅, 사용자 거주 리전 메타데이터로 2차 라우팅합니다. 쓰기 지연을 줄이기 위해 거주 리전을 홈 샤드로 지정합니다.

💻 예제 2: 읽기 부하 분산

리더에 쓰기를, 다수의 팔로워에 읽기를 라우팅합니다. 복제 지연이 민감한 쿼리(예: 결제 완료 직후 주문 내역 조회)는 리더로 강제하거나 read-after-write 토큰을 사용합니다.

실무 응용

유튜브와 인스타그램은 사용자 ID 기반 샤딩을 핵심 전략으로 사용합니다. 또한 Discord는 메시지를 Cassandra와 ScyllaDB에 채널 ID로 샤딩해 처리량을 확보합니다. AWS Aurora는 스토리지 계층을 분산 로그로 재설계해 컴퓨트와 스토리지를 분리했고, Google Spanner는 글로벌 ACID 트랜잭션을 TrueTime API로 실현했습니다.

⚠️ 자주 하는 실수: 샤딩 키를 잘못 고르면 핫스팟이 생겨 한 노드만 죽도록 일합니다. 또 복제 지연을 무시하고 팔로워에서 즉시 조회하면 "방금 저장한 게 안 보여요" 사건이 벌어집니다.
상 · 심화

전문가 레벨

합의 알고리즘 Paxos는 안전성과 활성을 모두 갖춘 메시지 패싱 모델로, 단일 합의(Single-Decree)와 다중 합의(Multi-Paxos)로 확장됩니다. Raft는 강한 리더 모델과 로그 복제 구조로 구현 난도를 낮췄으며, etcd·Consul·TiKV가 대표 사용처입니다. ZAB은 Zookeeper의 합의 프로토콜로 원자 브로드캐스트에 특화되어 있습니다. 분산 트랜잭션의 외부 일관성을 위해 Spanner는 GPS와 원자시계로 정밀 시간 동기화(TrueTime)를 보장하고, CockroachDB는 HLC로 비교적 저렴하게 인과 순서를 부여합니다. 멀티 리전 ACID는 커밋 대기(commit wait)나 폐쇄형 타임스탬프 같은 기법이 결합됩니다. CRDT(Conflict-free Replicated Data Type)는 카운터·세트·맵·시퀀스를 수학적으로 합치 가능하게 정의해 다중 리더 환경에서도 충돌 없이 수렴합니다. Yjs와 Automerge는 협업 편집기에, Riak DT는 일반 분산 응용에 활용됩니다. 분산 옵티마이저는 데이터 지역성을 고려해 push-down 조인과 분산 윈도우 함수를 실행하며, 데이터 스큐와 네트워크 셔플 비용이 핵심 병목입니다.

고급 패턴

-- CockroachDB: 멀티 리전 테이블 정의
ALTER DATABASE app PRIMARY REGION "asia-northeast1";
ALTER DATABASE app ADD REGION "europe-west1";
ALTER DATABASE app ADD REGION "us-east4";

CREATE TABLE users (
  id UUID PRIMARY KEY,
  name TEXT,
  home_region crdb_internal_region NOT NULL
)
LOCALITY REGIONAL BY ROW AS home_region;

-- Spanner: 외부 일관성 보장 SELECT
SELECT * FROM Orders
  WHERE OrderId = @id
  AS OF TIMESTAMP '2024-06-01T00:00:00Z';

-- Vitess: 샤딩 키 기반 라우팅 힌트
/*+ vt+ shard_session=true */
SELECT * FROM users WHERE user_id = 12345;

최신 동향

NewSQL(Spanner, CockroachDB, TiDB, YugabyteDB)은 ACID와 수평 확장의 양립을 추구하며 글로벌 서비스의 새로운 기본값이 되어 가고 있습니다. 또한 서버리스 분산 DB(Aurora Serverless v2, Neon, PlanetScale)는 자동 스케일링과 분기 기반 개발 워크플로를 제공합니다. 데이터 메시(Data Mesh) 아키텍처는 도메인 단위로 데이터를 분산 소유하는 조직적 접근법으로 떠오르고 있습니다.
