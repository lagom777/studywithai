3장. 정규화

3.1 정규화란?

🎯 쉽게 이해하기

정규화는 데이터 중복을 없애는 과정입니다.

왜 필요한가?

학생들의 전공 정보가 여러 테이블에 중복되어 있다면:

  • 전공이 변경될 때 모든 테이블을 수정해야 함 (수정 이상)
  • 실수로 다른 정보를 입력할 수 있음 (데이터 불일치)
  • 저장 공간 낭비 (데이터 중복)

정규화의 목표:

같은 정보는 한 곳에만 저장하고, 필요할 때 테이블을 연결해서 사용하기

📚 학술 정의

정규화는 E.F. Codd가 1971년 제안한 개념으로, 데이터 이상(Anomaly)을 제거하고 데이터 무결성을 보장하는 과정입니다.

함수 종속성(Functional Dependency, FD):

릴레이션 R에서 속성 X, Y에 대해, X의 값이 Y의 값을 유일하게 결정할 때, Y는 X에 함수 종속이라고 하며 X → Y로 표기합니다.

  • 완전 함수 종속(Full FD): X의 진부분집합이 Y를 함수 종속시키지 않음
  • 부분 함수 종속(Partial FD): X의 진부분집합이 Y를 함수 종속시킴
  • 이행 함수 종속(Transitive FD): X → Y이고 Y → Z일 때, X → Z

Armstrong's Axioms (함수 종속성의 공리):

  • 반사성(Reflexivity): Y ⊆ X이면 X → Y
  • 증대성(Augmentation): X → Y이면 XZ → YZ
  • 이행성(Transitivity): X → Y, Y → Z이면 X → Z

데이터 이상(Anomaly):

  • 삽입 이상(Insertion Anomaly): 새 데이터를 삽입할 때 불필요한 정보도 함께 삽입되어야 함
  • 수정 이상(Update Anomaly): 데이터 수정 시 여러 곳을 수정해야 함
  • 삭제 이상(Deletion Anomaly): 데이터 삭제 시 원치 않는 정보까지 삭제됨

3.2 제1정규형 (1NF)

🎯 쉽게 이해하기

1NF = 한 칸에 하나의 값만 저장하기

문제 상황:

학번 이름 전화번호 (문제!)
001 김철수 010-1234-5678, 02-1111-2222

해결 방법: 전화번호를 여러 행으로 나누기

학번 이름 전화번호
001 김철수 010-1234-5678
001 김철수 02-1111-2222

📚 형식적 정의

릴레이션 R이 1NF에 있다 ⟺ 모든 속성의 도메인이 원자값(Atomic Value)만 포함

원자값(Atomic Value): 더 이상 분해될 수 없는 기본 값. 즉, 반복 그룹(Repeating Group)이 없어야 함.

1NF 달성 방법:

  1. 반복 그룹 식별 (예: {전화번호1, 전화번호2, ...})
  2. 반복 그룹을 별도 릴레이션으로 분해
  3. 원래 릴레이션의 PK를 새 릴레이션에 FK로 추가

예시:

STUDENT_PHONE (학번, 전화번호) ← 1NF 관계

3.3 제2정규형 (2NF)

🎯 쉽게 이해하기

2NF = 부분 종속 제거

문제 상황:

선수 테이블에 선수번호, 팀, 팀_감독이 있다고 하면:

선수번호 팀 팀_감독
1 FC 서울 박항서

문제: "팀_감독"은 선수번호만으로 결정되지 않고, "팀"으로도 결정됨 → 부분 종속!

해결 방법: 팀 정보를 별도 테이블로 분리

선수 테이블 팀 테이블

선수(선수번호, 팀) + 팀(팀, 감독)

📚 형식적 정의

릴레이션 R이 2NF에 있다 ⟺ R이 1NF이고, 모든 비주요 속성(non-key attribute)이 모든 후보 키에 대해 완전 함수 종속(full functional dependency)

완전 함수 종속: 부분 함수 종속이 아님. 즉, 후보 키의 진부분집합이 비주요 속성을 함수 종속시키지 않음.

2NF 달성 방법 (분해 알고리즘):

  1. 부분 함수 종속 식별 (예: (A,B) → C인데, A → C도 성립)
  2. 부분 함수 종속으로 인한 속성들을 새 릴레이션으로 분리
  3. 새 릴레이션의 결정자를 PK로 설정

예시:

분해 전: STUDENT_COURSE (student_id, course_id, student_name)
          PK = (student_id, course_id)
          부분 FD: student_id → student_name

분해 후: STUDENT (student_id, student_name)
        ENROLLMENT (student_id, course_id)

3.4 제3정규형 (3NF) & BCNF

🎯 쉽게 이해하기

3NF = 이행 종속 제거

문제 상황:

학생(학번, 학과, 학과장)이라는 테이블이 있다면:

학번 → 학과 → 학과장 (이행 종속!)

학번 학과 학과장
001 컴퓨터공학 김교수

해결 방법: 학과 정보를 별도 테이블로 분리

학생(학번, 학과) + 학과(학과, 학과장)

BCNF: 더 엄격한 정규형. 모든 결정자가 후보 키여야 함.

📚 형식적 정의

릴레이션 R이 3NF에 있다 ⟺ R이 2NF이고, 어떤 비주요 속성도 후보 키에 이행 종속되지 않음

3NF 조건: ∀ FD X → A인 경우, 다음 중 하나를 만족:

  1. X가 슈퍼키이거나
  2. A가 비주요 속성이거나
  3. 또는 둘 다

BCNF (Boyce-Codd Normal Form):

릴레이션 R이 BCNF에 있다 ⟺ R의 모든 FD X → A에 대해, X가 슈퍼키

즉, 3NF보다 더 엄격함. 모든 결정자가 후보 키여야 함.

무손실 분해(Lossless Decomposition):

분해 후 자연 조인으로 원래 릴레이션을 복원할 수 있으면 무손실. 

R ⊆ R₁ ⋈ R₂ (항상 참)
R ⊇ R₁ ⋈ R₂ (무손실이면 참)
따라서 R = R₁ ⋈ R₂ (무손실 분해)

종속성 보존(Dependency Preservation):

분해 후 모든 FD가 보존되어야 함. 분해 후 릴레이션들의 FD 집합 ⊇ 원래 FD 집합

한 줄로 이해하기

정규화는 표 안에 같은 정보를 두 번 적지 않게 깔끔히 정리하는 규칙입니다.

왜 중요한가?

정규화를 모르면 데이터가 뒤죽박죽됩니다. 첫째, 회원 이름이 바뀌었는데 한 곳만 고쳐서 다른 표에는 옛 이름이 남는 사고가 흔합니다. 둘째, 한 행에 같은 값이 반복되면 저장공간이 낭비됩니다. 셋째, 잘못 정규화된 표는 통계 집계 결과까지 틀어집니다.

🌱 핵심 용어

  • 중복(Redundancy): 같은 값을 여러 곳에 반복 저장하는 것.
  • 1NF: 한 칸에 한 값만(쪼갤 수 없게).
  • 2NF: 부분 종속 제거(복합키의 일부에만 의존하는 컬럼 분리).
  • 3NF: 이행 종속 제거(키→A→B 같은 사슬을 끊기).
  • 이상 현상(Anomaly): 삽입·삭제·갱신 시 발생하는 오류.

개념 풀어쓰기

정규화는 책상 정리법과 같습니다. 책상 위에 펜, 노트, 영수증, 명함이 뒤섞여 있다면 무엇을 찾기도 어렵고, 같은 영수증을 두세 장 갖고 있어도 모를 수 있습니다. 정규화는 "펜은 펜꽂이에, 영수증은 영수증 파일에, 명함은 명함첩에"라고 카테고리별로 분리하는 일입니다. 데이터베이스에서는 한 표에 너무 많은 정보를 욱여넣지 말고, 주제별로 표를 나눈 뒤 외래키로 연결하는 것이 정규화입니다. 처음 듣기엔 표가 늘어나서 복잡해 보이지만, 한번 분리해 두면 회원 이름을 바꿀 때 회원 표 한 줄만 수정하면 모든 주문에 반영됩니다. 만약 주문 표 안에 회원 이름을 통째로 적어 뒀다면 천 건의 주문을 모두 고쳐야 했겠죠. 정규화는 "변경에 강한 데이터 구조"를 만드는 일이며, 그 보상으로 디스크 공간 절약과 데이터 정합성을 얻습니다. 다만 너무 잘게 쪼개면 조인이 많아져 느려질 수 있어, 실무에서는 3NF까지를 표준으로 삼습니다.

💡 학습 TIP: 일단 엉망인 표(예: 주문번호, 회원이름, 상품들이 한 칸에 콤마로 나열된 표)를 직접 그려 본 뒤 1NF→2NF→3NF로 단계별 정리해 보세요. 손으로 그릴 때 가장 잘 이해됩니다.

요약

  • 정규화는 중복 제거가 목적입니다.
  • 1NF: 칸에 값 하나만.
  • 2NF: 부분 종속 제거.
  • 3NF: 이행 종속 제거.
  • 실무 기준은 보통 3NF까지입니다.
중 · 응용

이론 깊이

정규화는 함수 종속(Functional Dependency, FD)에 기반합니다. X→Y는 "X 값이 같으면 Y 값도 같다"는 뜻이며, 모든 정규형은 FD를 어떻게 분해해 이상 현상을 없애느냐의 문제입니다. 1NF는 원자값 보장, 2NF는 모든 비키 속성이 후보키 전체에 종속(완전 함수 종속)되어야 한다는 규칙, 3NF는 비키 속성이 다른 비키 속성에 종속되면 안 된다는 규칙입니다. BCNF(Boyce-Codd Normal Form)는 모든 결정자가 후보키여야 한다는 더 엄격한 조건이며, 4NF는 다치 종속(Multi-Valued Dependency), 5NF는 조인 종속까지 다룹니다. 분해 시 무손실 분해(Lossless Decomposition)와 종속 보존(Dependency Preservation)을 동시에 만족해야 의미가 보존됩니다. 실무에서는 3NF/BCNF를 목표로 하되, 읽기 성능이 중요한 OLAP 환경에서는 의도적으로 반정규화(Denormalization)하여 조인을 줄입니다.

실전 SQL/쿼리

-- 1NF 위반: 한 칸에 여러 값
-- order_id | products
-- 1001     | 사과,바나나,포도

-- 1NF 적용 후: 행을 분리
CREATE TABLE order_item (
  order_id   INT,
  product    VARCHAR(50),
  quantity   INT,
  PRIMARY KEY (order_id, product)
);

-- 3NF 위반 예: zip_code → city, state (이행 종속)
-- member(member_id, name, zip_code, city, state)

-- 3NF 적용
CREATE TABLE zipcode (
  zip_code  CHAR(5) PRIMARY KEY,
  city      VARCHAR(50),
  state     VARCHAR(50)
);
CREATE TABLE member (
  member_id INT PRIMARY KEY,
  name      VARCHAR(50),
  zip_code  CHAR(5) REFERENCES zipcode(zip_code)
);

예제

💻 예제 1: 강의 수강 모델

(학번, 과목코드, 과목명, 교수명, 교수실)이 한 표에 있다면 과목명이 학번에 부분 종속됩니다. 과목 정보를 별도 표로 분리하고, 교수실은 교수에 종속이므로 교수 테이블도 따로 만듭니다.

💻 예제 2: 의도적 반정규화

대시보드 응답 속도가 중요하면 일별 매출 집계 테이블을 미리 만들어 둡니다. 원본 트랜잭션 테이블은 정규화된 채 두고, 집계 테이블은 반정규화 형태로 운영합니다. 야간 배치로 동기화합니다.

실무 응용

실제 ERP는 마스터 데이터(회사, 부서, 직원, 코드)를 강하게 정규화해 두고, 트랜잭션 데이터(전표, 주문)는 발생 시점의 값을 스냅샷으로 일부 복사해 둡니다. 이렇게 하면 마스터 변경에도 과거 거래의 의미가 보존됩니다. 또한 데이터 웨어하우스에서는 스타 스키마와 스노우플레이크 스키마가 정규화 정도의 차이를 보여 주며, 분석 쿼리 성능과 ETL 단순성을 두고 트레이드오프를 조율합니다.

⚠️ 자주 하는 실수: 일단 모두 3NF로 만들었는데 조회 성능이 안 나와서 뒤늦게 컬럼을 합치는 일이 많습니다. 처음부터 읽기/쓰기 비율을 고려해 의도적 반정규화를 설계하는 것이 좋습니다.
상 · 심화

전문가 레벨

BCNF는 3NF보다 엄격해서 일부 종속 보존을 희생할 수 있는 정규형입니다. 예컨대 (학생, 과목, 교수)에서 교수→과목, (학생, 과목)→교수 관계가 동시에 있을 때 BCNF로 분해하면 종속 (학생, 과목)→교수가 보존되지 않습니다. 4NF는 다치 종속을, 5NF(PJNF)는 조인 종속을 다룹니다. 분해의 핵심 정리로 Heath의 정리와 무손실 결합 보장 알고리즘이 있으며, 정규화 결정은 후보키 식별과 닫힘 계산(attribute closure)으로 형식화됩니다. 6NF는 시간성(Temporal) 데이터 모델링에서 중요한 역할을 하며, 모든 비키 속성을 단일 컬럼 테이블로 분리하여 시간 변화를 정확히 추적합니다. NoSQL 등장 이후에는 도메인 주도 설계(DDD)의 애그리게이트 단위로 비정규화를 의도적으로 수용하는 사례가 많아졌습니다. 또한 그래프 데이터베이스에서는 정규화 개념이 노드와 관계의 분리로 자연스럽게 구현됩니다. 분산 환경에서는 정규화된 스키마의 조인 비용이 매우 비싸서, 단일 키 조회로 끝낼 수 있도록 의도적으로 비정규화한 데이터 모델을 채택하는 것이 일반적입니다.

고급 패턴

-- 6NF 풍의 시간성 모델: 속성별 유효기간 분리
CREATE TABLE employee (
  employee_id INT PRIMARY KEY
);

CREATE TABLE employee_name (
  employee_id INT,
  full_name   VARCHAR(100),
  valid_from  DATE,
  valid_to    DATE,
  PRIMARY KEY (employee_id, valid_from)
);

CREATE TABLE employee_salary (
  employee_id INT,
  salary      DECIMAL(12,2),
  valid_from  DATE,
  valid_to    DATE,
  PRIMARY KEY (employee_id, valid_from)
);

-- 특정 시점 스냅샷 재구성
SELECT e.employee_id, n.full_name, s.salary
FROM   employee e
JOIN   employee_name   n ON e.employee_id = n.employee_id
JOIN   employee_salary s ON e.employee_id = s.employee_id
WHERE  DATE '2024-06-01' BETWEEN n.valid_from AND n.valid_to
  AND  DATE '2024-06-01' BETWEEN s.valid_from AND s.valid_to;

최신 동향

이벤트 소싱(Event Sourcing) 아키텍처는 정규화 대신 이벤트 로그 자체를 진실의 원천으로 삼고, 읽기 모델은 비정규화된 형태로 따로 구성합니다(CQRS). 또한 데이터 레이크하우스(Iceberg, Delta Lake)는 시간여행(Time Travel) 기능으로 6NF 모델링을 자연스럽게 지원합니다. 그래프 모델은 노드와 관계 단위 정규화로 다대다 관계를 직관적으로 표현해 추천 시스템과 사기 탐지에 활용됩니다.
