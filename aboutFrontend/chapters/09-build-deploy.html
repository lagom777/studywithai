9장. 빌드도구와 배포

개발 코드를 프로덕션 애플리케이션으로 만들기

9.1 빌드 도구의 필요성

현대적인 웹 개발에서 빌드 도구는 필수 요소입니다. 빌드 도구는 다음 작업을 자동화합니다:

1. 번들링 (Bundling)

  • 여러 파일을 하나 또는 소수의 번들로 묶기
  • 모듈 간 의존성 해결
  • 파일 크기 최소화

2. 트랜스파일링 (Transpiling)

  • 최신 JavaScript를 구형 브라우저가 이해할 수 있는 코드로 변환
  • TypeScript를 JavaScript로 컴파일
  • JSX를 JavaScript로 변환

3. 최적화 (Optimization)

  • 코드 최소화(minification)
  • 미사용 코드 제거(tree shaking)
  • 이미지 최적화
  • 소스맵 생성
역사: 초기에는 Grunt, Gulp 등의 Task Runner가 주류였으나, 현재는 Webpack, Vite, esbuild 같은 번들러가 표준입니다.

9.2 Vite vs Webpack: 현대 번들러 비교

프론트엔드 개발 환경에서 가장 인기 있는 두 가지 번들러를 비교합니다.

항목 Webpack Vite
출시 2014년 (역사 깊음) 2020년 (최신 기술)
개발 서버 속도 느림 (전체 번들 필요) 매우 빠름 (ES 모듈 사용)
HMR (Hot Module Replacement) 느림 즉시 반영
빌드 속도 중간 매우 빠름
커스터마이제이션 뛰어남 간단함
플러그인 생태계 매우 풍부 성장 중
학습 곡선 가파름 완만함
추천 용도 복잡한 프로젝트 신규 프로젝트
/* Vite 설정 - 간단함 */ // vite.config.js import { defineConfig } from 'vite' import react from '@vitejs/plugin-react' export default defineConfig({ plugins: [react()], server: { port: 3000, hmr: true } }) /* Webpack 설정 - 복잡함 */ // webpack.config.js const path = require('path'); module.exports = { mode: 'production', entry: './src/index.js', output: { path: path.resolve(__dirname, 'dist'), filename: '[name].[contenthash].js' }, module: { rules: [ { test: /\.jsx?$/, exclude: /node_modules/, use: { loader: 'babel-loader' } } ] } };

9.3 Babel과 TypeScript

Babel: 최신 JavaScript 코드를 이전 표준으로 변환합니다.

// .babelrc 설정 { "presets": [ ["@babel/preset-env", { "targets": { "browsers": ["> 1%", "last 2 versions", "not dead"] } }], "@babel/preset-react" ], "plugins": [ "@babel/plugin-proposal-class-properties" ] } // 변환 전: ES2020 화살표 함수 + optional chaining const user = users?.find(u => u.id === 1); const email = user?.email ?? 'no-email'; // Babel 변환 후: ES5 호환 코드 var user = users != null ? users.find(function(u) { return u.id === 1; }) : null; var email = user != null && user.email != null ? user.email : 'no-email';

TypeScript: JavaScript에 정적 타입 시스템을 추가합니다.

// TypeScript 코드 interface User { id: number; name: string; email: string; } function getUser(id: number): User | null { const user = users.find(u => u.id === id); return user || null; } // 컴파일 후 JavaScript function getUser(id) { var user = users.find(function(u) { return u.id === id; }); return user || null; }
TypeScript의 장점: 개발 중 타입 오류를 미리 발견하고, IDE의 자동완성과 리팩토링이 뛰어납니다.

9.4 패키지 관리자: npm vs yarn vs pnpm

JavaScript 패키지(라이브러리)를 설치하고 관리하는 도구입니다. 모두 동일한 npm 레지스트리를 사용하지만 성능과 전략이 다릅니다.

항목 npm yarn pnpm
출시 2010년 (기본값) 2016년 (Facebook) 2016년 (경량)
설치 속도 중간 빠름 매우 빠름
디스크 사용량 높음 높음 낮음 (심링크 사용)
Lock 파일 package-lock.json yarn.lock pnpm-lock.yaml
의존성 관리 플랫 엄격함 가장 엄격함
권장 용도 일반적 프로젝트 대규모 프로젝트 모노레포
// 설치 명령어 비교 npm install lodash yarn add lodash pnpm add lodash // 개발용 의존성 추가 npm install --save-dev webpack yarn add --dev webpack pnpm add -D webpack

9.5 CI/CD 파이프라인

CI (Continuous Integration): 코드 변경을 자동으로 테스트하고 빌드합니다.

CD (Continuous Deployment/Delivery): 테스트를 통과한 코드를 자동으로 배포합니다.

가장 인기 있는 CI/CD 도구는 GitHub Actions입니다.

// .github/workflows/deploy.yml name: Deploy on: push: branches: [main] pull_request: branches: [main] jobs: build-and-test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Setup Node.js uses: actions/setup-node@v3 with: node-version: '18' - name: Install dependencies run: npm ci - name: Run tests run: npm test - name: Build run: npm run build - name: Deploy to production if: github.ref == 'refs/heads/main' run: npm run deploy

9.6 배포 플랫폼

현대적인 웹 애플리케이션은 다양한 플랫폼에 배포할 수 있습니다. 각각의 장단점을 비교해봅시다.

Vercel (Next.js 제작사): 가장 편리한 배포 경험. GitHub 연동으로 자동 배포.

Netlify: JAMstack 전문. 빌드와 배포가 매우 간단.

Cloudflare Pages: 전 세계 CDN 기반. 매우 빠른 속도.

AWS / Google Cloud / Azure: 강력하고 유연하지만 설정이 복잡.

/* Vercel 배포 (vercel.json) */ { "buildCommand": "npm run build", "outputDirectory": "dist", "env": { "REACT_APP_API_URL": "@api-url" } } /* 환경변수 설정 */ // Vercel Dashboard에서 설정하거나: // .env.production REACT_APP_API_URL=https://api.example.com REACT_APP_VERSION=1.0.0

하 · 입문

한 줄로 이해하기

빌드는 개발자가 짠 코드를 브라우저가 빠르게 실행할 수 있는 형태로 묶고 압축하는 과정이고, 배포는 그 결과물을 사용자에게 전달하는 일입니다.

왜 배워야 하나?

실무 사례로 (1) npm run build 한 번으로 수십 개 파일이 자동 최적화·번들링, (2) GitHub에 푸시하면 1분 내 배포가 끝나는 자동화 파이프라인 구축, (3) 새 기능을 일부 사용자에게만 먼저 공개하는 Canary 배포 등이 있습니다. 빌드/배포를 모르면 "내 컴퓨터에선 되는데" 문제로 동료를 괴롭히기 쉽습니다.

🌱 핵심 용어

  • 번들러: 여러 JS 파일을 하나로 합치는 도구(Webpack, Vite)
  • 트랜스파일러: 최신 문법을 구형 브라우저용으로 변환(Babel, SWC)
  • 미니파이: 코드 압축으로 크기 축소
  • CI/CD: 자동 테스트·자동 배포 파이프라인
  • 호스팅: 배포된 사이트가 살고 있는 서버

개념 풀어쓰기

빌드와 배포를 이삿짐 회사에 비유해봅시다. 빌드는 흩어진 살림을 박스에 차곡차곡 넣고 라벨을 붙이는 일이고, 배포는 그 박스를 새 집(서버)에 가져다 놓는 일입니다. 좋은 이삿짐 회사일수록 (1) 짐을 적게 부피로 줄이고(미니파이), (2) 트럭 한 대에 효율적으로 싣고(번들링), (3) 자주 쓰는 짐만 먼저 풀고 나머지는 나중에(코드 스플리팅) 처리합니다. 요즘은 Vite 같은 도구가 개발 중에는 빠른 미리보기를 제공하고, vite build로 명령 한 줄에 최적화된 산출물을 만들어 줍니다. Vercel·Netlify·Cloudflare Pages는 깃허브와 연결만 하면 푸시할 때마다 자동으로 빌드·배포해주므로 초보자도 무료로 사이트를 띄울 수 있습니다.

💡 학습 TIP: 작은 프로젝트로 GitHub Pages·Vercel·Cloudflare Pages를 차례로 배포해보면 차이가 한눈에 들어옵니다. 도메인 연결까지 한 번 해보세요.

한 페이지 요약

  • 빌드 = 코드 변환 + 번들 + 압축
  • Vite는 ESM 기반으로 개발 시작이 빠름
  • 배포는 정적 호스팅 + CDN이 가장 단순
  • GitHub Actions로 푸시 → 빌드 → 배포 자동화
  • 환경변수(.env)로 설정값을 코드와 분리
중 · 응용

이론 깊이

모던 번들러는 모듈 그래프를 만들고, 진입점부터 의존성을 따라가며 최종 산출물을 생성합니다. 이때 트리 셰이킹은 ESM의 정적 분석성을 활용해 사용되지 않는 export를 제거합니다. 코드 스플리팅은 동적 import()·라우트 단위로 청크를 분리해 초기 로드를 줄이고, 청크는 콘텐츠 해시 파일명으로 장기 캐싱이 가능합니다. CSS는 PostCSS 파이프라인으로 자동 prefix·중첩 처리를 거쳐 산출되며, 사용 안 된 클래스는 PurgeCSS로 제거합니다. CI/CD는 (1) 의존성 설치 캐싱, (2) 병렬 테스트, (3) 산출물 캐시, (4) 환경별 배포(스테이징·프로덕션) 단계로 구성되며, GitHub Actions·CircleCI·GitLab CI가 표준입니다. 배포 전략으로는 Blue-Green, Canary, Feature Flag(LaunchDarkly·Unleash)가 안전망 역할을 합니다.

핵심 코드 패턴

# .github/workflows/deploy.yml
name: Deploy
on:
  push: { branches: [main] }

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'pnpm' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm test --run
      - run: pnpm build
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CF_API_TOKEN }}
          command: pages deploy dist --project-name=my-app

상 · 심화

전문가 레벨

고도화된 빌드 시스템은 의존성 그래프 분석, 영속 캐시(persistent cache), 분산 빌드를 활용합니다. esbuild·SWC는 Go·Rust로 작성돼 Babel 대비 10–100배 빠르고, Turbopack은 함수 단위 증분 컴파일로 변경 부분만 다시 빌드합니다. ESM·CJS 듀얼 패키지를 정확히 출력하려면 exports 필드의 condition(import/require/types)을 정밀 설정해야 하며, "subpath patterns"로 트리 셰이킹 단위를 잘게 쪼갤 수 있습니다. 배포 측면에선 Edge SSR(Cloudflare Workers·Vercel Edge·Deno Deploy)이 PoP에서 페이지를 즉시 생성해 TTFB 50ms 이하를 달성합니다. 보안은 SRI(Subresource Integrity), CSP(Content Security Policy), Trusted Types로 XSS 표면을 줄이고, Sigstore·SLSA로 빌드 공급망 무결성을 보증하는 것이 표준화되고 있습니다.

고급 패턴

// vite.config.ts — 라이브러리 모드 + ESM/CJS 듀얼 출력
import { defineConfig } from 'vite';
import dts from 'vite-plugin-dts';

export default defineConfig({
  plugins: [dts({ rollupTypes: true })],
  build: {
    lib: {
      entry: { index: 'src/index.ts', server: 'src/server.ts' },
      formats: ['es', 'cjs']
    },
    rollupOptions: {
      external: ['react', 'react-dom'],
      output: { exports: 'named' }
    },
    sourcemap: 'hidden',
    minify: 'esbuild'
  }
});

// package.json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./server": { "import": "./dist/server.js" }
  }
}
