8장. 성능 최적화

빠르고 반응형이 좋은 웹사이트의 비결

8.1 웹 성능이 중요한 이유

웹 성능은 단순한 기술 문제를 넘어 비즈니스에 직접적인 영향을 미칩니다.

로딩 시간과 사용자 행동:

  • 페이지 로딩 시간 1초 증가 → 사용자 만족도 7% 감소
  • 로딩 3초 이상 → 모바일 사용자 40% 이탈
  • 성능 개선 → 전환율(수익) 증가

검색 엔진 최적화(SEO): Google은 Core Web Vitals을 검색 순위의 핵심 요소로 반영합니다.

💡 핵심: 성능 최적화는 기술뿐만 아니라 사용자 경험과 수익성을 향상시키는 전략입니다.

8.2 Core Web Vitals

Google이 정의한 웹 성능의 3가지 핵심 지표입니다.

1. LCP (Largest Contentful Paint)

  • 주요 콘텐츠가 화면에 나타나는 시간
  • 목표: 2.5초 이내

2. FID / INP (First Input Delay / Interaction to Next Paint)

  • 사용자가 클릭했을 때 응답 시간
  • 목표: 100ms 이내

3. CLS (Cumulative Layout Shift)

  • 페이지 요소가 얼마나 움직이는가
  • 목표: 0.1 이하
/* Core Web Vitals 측정 코드 */ web.vitals.onLCP((metric) => { console.log('LCP:', metric.value); }); web.vitals.onFID((metric) => { console.log('FID:', metric.value); }); web.vitals.onCLS((metric) => { console.log('CLS:', metric.value); });

8.3 번들 최적화

JavaScript와 CSS 번들 크기를 줄여 로딩 시간을 단축합니다.

Tree Shaking: 사용하지 않는 코드를 빌드 과정에서 제거합니다.

/* ES6 모듈 문법 사용 (Tree shaking 가능) */ export function usedFunction() { } export function unusedFunction() { } // 사용하는 것만 import import { usedFunction } from './utils.js'; /* Webpack에서 tree shaking 활성화 */ // webpack.config.js module.exports = { mode: 'production', optimization: { usedExports: true, sideEffects: false } }; /* Code Splitting */ import { lazy, Suspense } from 'react'; const HeavyComponent = lazy(() => import('./HeavyComponent')); export function App() { return ( <Suspense fallback={<div>로딩 중...</div>}> <HeavyComponent /> </Suspense> ); }

Dynamic Import: 필요할 때만 코드를 로드합니다.

// 사용자 클릭 시에만 모듈 로드 button.addEventListener('click', async () => { const module = await import('./heavy-module.js'); module.doSomething(); });

8.4 이미지 최적화

이미지는 대부분의 웹사이트에서 가장 큰 파일입니다. 효율적인 이미지 관리가 성능 개선의 핵심입니다.

포맷 최적화:

  • WebP: JPEG보다 25-35% 작은 파일 크기
  • AVIF: 가장 최신 포맷, WebP보다도 더 효율적
  • Fallback: 지원하지 않는 브라우저를 위한 PNG/JPEG
<picture> <source srcset="image.avif" type="image/avif"> <source srcset="image.webp" type="image/webp"> <source srcset="image.jpg" type="image/jpeg"> <img src="image.jpg" alt="최적화된 이미지"> </picture> /* Lazy Loading - 스크롤했을 때만 로드 */ <img src="placeholder.jpg" loading="lazy" alt="지연 로드 이미지"> /* Responsive Images with srcset */ <img srcset="small.jpg 480w, medium.jpg 768w, large.jpg 1200w" sizes="(max-width: 480px) 100vw, (max-width: 768px) 50vw, 33vw" src="large.jpg" alt="반응형 이미지">

8.5 렌더링 최적화

Critical Rendering Path (CRP)는 브라우저가 HTML, CSS, JavaScript를 처리하여 화면에 그리는 순서입니다.

성능 개선 전략:

  • 중요 CSS는 inline으로, 나머지는 async/defer로 로드
  • JavaScript 실행을 지연시켜 렌더링 블로킹 제거
  • Reflow/Repaint 최소화 (DOM 변경 배치 처리)
/* 렌더링 블로킹 제거 */ <!-- CSS는 head에서 로드 (렌더링 차단) --> <link rel="stylesheet" href="style.css"> <!-- 중요한 CSS만 inline으로 --> <style> /* 초기 화면에 필요한 스타일 */ body { font-family: sans-serif; } </style> <!-- JavaScript는 body 끝에서 로드 --> <script src="app.js" defer></script> /* Reflow/Repaint 최소화 */ // ❌ 나쁜 예: 반복마다 DOM에 접근 for (let i = 0; i < 100; i++) { element.style.width = (i * 10) + 'px'; // 매번 reflow! } // ✅ 좋은 예: 배치 처리 const widths = Array.from({length: 100}, (_, i) => (i * 10) + 'px'); element.style.width = widths[widths.length - 1];

8.6 캐싱 전략

캐싱은 같은 데이터를 반복 요청하지 않아 로딩 시간을 획기적으로 단축합니다.

3단계 캐싱:

  1. 브라우저 캐시: 로컬 저장소에 저장된 파일
  2. Service Worker: 오프라인에서도 작동하는 네트워크 캐시
  3. CDN: 전 세계 서버에 콘텐츠 복제
/* HTTP 캐시 헤더 설정 */ // HTML: 매번 확인 Cache-Control: no-cache, must-revalidate // CSS/JS: 1년 캐시 (해시 기반 버저닝 사용) Cache-Control: public, max-age=31536000, immutable // 이미지: 30일 캐시 Cache-Control: public, max-age=2592000 /* Service Worker 캐싱 */ self.addEventListener('install', (event) => { event.waitUntil( caches.open('v1').then((cache) => { return cache.addAll([ '/', '/styles.css', '/app.js' ]); }) ); });

한 줄로 이해하기

웹 성능 최적화는 사이트가 빠르게 보이고 매끄럽게 반응하도록 만드는 모든 작업이며, 이탈률·매출에 직접 영향을 미칩니다.

왜 배워야 하나?

실무 사례로 (1) 첫 화면 로딩이 1초 늘어날 때마다 이커머스 전환율이 약 7% 떨어진다는 Akamai 보고, (2) 구글이 Core Web Vitals를 검색 순위에 반영하면서 SEO와 직결, (3) 모바일 데이터·저사양 기기 사용자를 잃지 않으려면 200KB 이하 초기 번들이 목표인 경우가 늘었습니다.

🌱 핵심 용어

  • LCP(Largest Contentful Paint): 가장 큰 콘텐츠가 보이는 시간
  • INP(Interaction to Next Paint): 인터랙션 응답성
  • CLS(Cumulative Layout Shift): 레이아웃 흔들림 정도
  • TTFB: 서버가 첫 바이트를 내려보내는 시간
  • 번들 크기: 사용자가 받아야 하는 JS·CSS의 합

개념 풀어쓰기

웹 성능을 음식점에 비유해보면, TTFB는 주문하고 첫 음식이 나오기까지 걸린 시간, LCP는 메인 요리가 식탁에 놓인 시점, INP는 손님이 종을 누르고 직원이 응답하기까지의 속도, CLS는 직원이 음식을 놓는 동안 식탁이 갑자기 흔들리지 않는 안정성에 해당합니다. 빠른 사이트의 비결은 (1) 보낼 짐(JS·이미지)을 가볍게, (2) 보낼 길(네트워크)을 짧게, (3) 받은 짐을 빨리 풀어 보여주는 것 — 이 3가지입니다. 이미지를 적절한 포맷·크기로 압축하고, 큰 JS는 필요할 때만 로드하고, 첫 화면에 꼭 필요한 CSS는 인라인으로 보내는 식입니다. 측정 없이는 최적화도 없으므로 Lighthouse·PageSpeed Insights부터 돌려보는 습관이 중요합니다.

💡 학습 TIP: Chrome DevTools → Lighthouse 탭에서 1분 만에 점수와 개선 권고를 받을 수 있고, "Performance" 탭의 녹화 기능으로 실제 프레임을 분석할 수 있습니다.

한 페이지 요약

  • 측정 → 가설 → 적용 → 재측정의 사이클
  • Core Web Vitals 3대 지표(LCP/INP/CLS) 우선
  • 이미지·JS 다이어트가 가장 효과적
  • 지연 로딩(lazy)·코드 스플리팅으로 초기 페이로드 축소
  • 캐시·CDN으로 두 번째 방문은 즉시
중 · 응용

이론 깊이

브라우저 렌더링은 Critical Rendering Path를 따릅니다. HTML 파싱 → DOM·CSSOM 생성 → Render Tree → Layout → Paint → Composite. 이 경로의 어느 단계가 늦어도 LCP는 미뤄집니다. 가장 흔한 병목은 (1) 렌더 차단 CSS·JS, (2) 큰 동기 JS의 메인 스레드 점유, (3) 거대한 이미지의 디코드 시간, (4) 폰트 로딩 지연으로 인한 FOIT/FOUT입니다. 해결책으로는 defer/async·type="module" 사용, rel="preload"·preconnect" 힌트, fetchpriority="high"로 LCP 이미지 우선순위 부여, AVIF/WebP 포맷 + responsive image, font-display: swap 등이 있습니다. JS 측면에선 코드 스플리팅(route-based, component-based), 트리 셰이킹, lazy import가 핵심이며, INP는 Long Task(50ms 초과) 분할이 관건입니다.

핵심 코드 패턴

// 코드 스플리팅 + Suspense
const Editor = lazy(() => import('./Editor'));

function App() {
  return (
    <Suspense fallback={<Skeleton />}>
      <Editor />
    </Suspense>
  );
}

// LCP 이미지 우선순위 + 반응형
<img src="/hero-1280.avif"
     srcset="/hero-640.avif 640w, /hero-1280.avif 1280w, /hero-1920.avif 1920w"
     sizes="100vw"
     fetchpriority="high"
     decoding="async"
     alt="..." />

대표 예제

💻 예제 1: Long Task 분할

1만 개 항목 처리 작업을 scheduler.yield() 또는 setTimeout(.., 0)으로 분할하면 INP가 200ms → 50ms로 떨어져 입력 응답성이 크게 개선됩니다.

💻 예제 2: 가상 스크롤

react-window·@tanstack/virtual로 화면에 보이는 50개만 렌더해 1만 행 테이블도 60fps 유지가 가능합니다.

실무 응용

실무에선 (1) RUM(Real User Monitoring)으로 실사용자 지표를 SpeedCurve·Calibre·Sentry로 수집, (2) PR마다 Lighthouse CI로 회귀 자동 차단, (3) WebP/AVIF 자동 변환 CDN(Cloudflare Images, Vercel Image), (4) Brotli 압축·HTTP/3·Edge SSR 활용 등이 표준입니다.

⚠️ 자주 하는 실수: 합성 점수만 좋게 만들고 실사용자 지표 외면, useMemo 남발로 메모리만 늘리기, 거대한 폰트 패밀리 통째 로딩, 이미지 width/height 미지정으로 CLS 발생.
상 · 심화

전문가 레벨

고급 성능 최적화는 측정 정확도부터 시작합니다. PerformanceObserver로 LCP·INP·CLS를 RUM으로 수집하고, web-vitals 라이브러리를 통해 분포(p75)를 모니터링합니다. 메인 스레드 부담을 줄이려면 React Concurrent Renderer의 useTransition·startTransition으로 우선순위가 낮은 업데이트를 분리하고, Web Worker로 무거운 계산을 오프로드합니다. 네트워크 측면에선 HTTP/3 + 0-RTT, Early Hints(103), Server Timing 헤더로 서버 단계별 시간을 노출시켜 디버깅합니다. 이미지 최적화의 끝판은 BlurHash/LQIP 플레이스홀더 + content-visibility: auto + contain-intrinsic-size 조합으로 화면 밖 콘텐츠를 사실상 렌더하지 않는 것입니다. JS 측면에선 V8의 Hidden Class 안정화, 인라인 캐싱 친화적 객체 형태, ArrayBuffer/SharedArrayBuffer 활용한 zero-copy 통신이 미세 최적화 영역입니다.

고급 패턴

// Web Worker + Comlink로 무거운 계산 오프로드
// worker.js
import * as Comlink from 'comlink';
Comlink.expose({
  heavyCompute(input) {
    // CPU 집약 연산 (이미지 분석, ML 추론 등)
    return result;
  }
});

// main.js
const worker = new Worker('./worker.js', { type: 'module' });
const api = Comlink.wrap(worker);
const result = await api.heavyCompute(input);  // 메인 스레드 블록 X

// LCP 이미지를 CSS background로 사용 시 preload + image-set + AVIF
<link rel="preload" as="image"
      imagesrcset="/h-640.avif 640w, /h-1280.avif 1280w"
      imagesizes="100vw" fetchpriority="high">

최신 트렌드

(1) Speculation Rules API: 다음 페이지를 미리 prerender해 즉시 전환, (2) Partytown: 서드파티 스크립트(GA·픽셀)를 Web Worker로 격리, (3) Islands Architecture(Astro·Qwik): hydrate를 인터랙티브 영역으로만 한정해 JS 페이로드 90% 감소.

관련 분야

  • 네트워크 프로토콜(HTTP/3, QUIC, BBR 혼잡제어)
  • 엣지 컴퓨팅(Cloudflare Workers, Vercel Edge)
  • WebAssembly로 핵심 로직 가속
