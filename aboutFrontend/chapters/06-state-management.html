6장. 상태관리

React 애플리케이션의 상태를 효율적으로 관리하는 방법들

6.1 상태란?

상태(State)는 애플리케이션이 어느 순간의 조건을 나타내는 데이터입니다. 시간에 따라 변할 수 있으며, 이 변화가 UI 업데이트를 트리거합니다.

UI 상태 vs 서버 상태

UI 상태

• 모달 열림/닫힘

• 입력 필드 값

• 탭 선택

• 폼 유효성

💡 애플리케이션 메모리에만 존재

서버 상태

• 데이터베이스의 데이터

• 사용자 정보

• 게시물 목록

• 비동기 데이터

💡 백엔드에서 관리하며 캐싱 필요

// UI 상태 예제
function Modal() {
  const [isOpen, setIsOpen] = useState(false);
  // isOpen은 UI 상태
}

// 서버 상태 예제
function UserList() {
  const [users, setUsers] = useState([]);
  // users는 서버에서 가져온 상태
}

6.2 React Context API

Context는 깊은 계층 구조에서 Props Drilling을 피하기 위해 사용합니다. 전역 상태를 여러 컴포넌트에 제공합니다.

Context 사용 패턴

// 1. Context 생성
const ThemeContext = React.createContext();

// 2. Provider로 감싸기
function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <MainContent />
    </ThemeContext.Provider>
  );
}

// 3. 자식 컴포넌트에서 사용
function Button() {
  const { theme } = useContext(ThemeContext);
  return <button style={{ background: theme }}>버튼</button>;
}

Context API의 장단점

✅ 장점

  • Props Drilling 제거
  • 내장 API (추가 라이브러리 불필요)
  • 간단한 전역 상태
  • 학습 곡선 완만

❌ 단점

  • 성능 최적화 어려움
  • DevTools 부족
  • 복잡한 상태 관리 시 코드 증가
  • 미들웨어 미지원

6.3 Redux 패턴

Redux는 예측 가능한 상태 관리를 위한 라이브러리입니다. Action → Reducer → Store → View의 단방향 사이클을 따릅니다.

Redux 데이터 흐름

💡 버튼을 클릭하여 Redux 사이클을 시뮬레이션하세요.

Redux의 3가지 원칙

Single Source of Truth

전체 상태가 하나의 Store에 저장

State is Read-Only

상태 변경은 Action을 통해서만 가능

Changes are Made with Pure Reducers

순수 함수인 Reducer로만 상태 업데이트

Redux 구현 예제

// 1. Action 정의
const INCREMENT = 'INCREMENT';
const increment = () => ({ type: INCREMENT });

// 2. Reducer 작성
function counterReducer(state = 0, action) {
  if (action.type === INCREMENT) {
    return state + 1;
  }
  return state;
}

// 3. Store 생성
const store = createStore(counterReducer);

// 4. 컴포넌트에서 사용
store.dispatch(increment());
const state = store.getState();

6.5 서버 상태 관리

API 데이터, 데이터베이스 상태 같은 비동기 데이터 관리를 위한 라이브러리들입니다.

React Query

  • 자동 캐싱
  • 백그라운드 동기화
  • 재시도 로직
  • 상태 머신

TanStack Query

SWR

  • 간단한 API
  • 자동 리페칭
  • 초점 재개 시 갱신
  • 번들 매우 작음

Vercel 개발

React Query 기본 사용법

import { useQuery } from 'react-query';

function UserList() {
  const { data, isLoading, error } = useQuery('users',
    () => fetch('/api/users').then(r => r.json())
  );
  
  if (isLoading) return <div>로딩...</div>;
  if (error) return <div>에러: {error.message}</div>;
  return <div>{data.map(u => <p key={u.id}>{u.name}</p>)}</div>;
}

상태 관리 방식 비교

Context API vs Redux 데모

같은 카운터를 두 가지 방식으로 구현했을 때의 차이를 확인하세요.

Context API

더 간단하지만 성능 최적화 어려움

Count: 0
const [count, setCount] = useState(0)

Redux

더 복잡하지만 성능 최적화 용이

Count: 0
dispatch({type: 'INCREMENT'})

상태 관리 선택 가이드

어떤 것을 선택할까?

  • 📍 useState + Props: 작은 프로젝트, 간단한 상태
  • 🌍 Context API: 중간 규모, 테마/언어 같은 전역 설정
  • 🎯 Zustand/Jotai: 현대적이고 간단한 상태 관리
  • ⚙️ Redux: 복잡한 상태 로직, 시간여행 디버깅 필요
  • 🔄 React Query: 서버 상태, 데이터 페칭
  • ♾️ 조합: UI 상태(Zustand) + 서버 상태(React Query)

핵심 요약

  • ✅ 상태는 UI 상태와 서버 상태로 나뉘며 다르게 관리합니다.
  • ✅ Context API는 Props Drilling을 제거하지만 성능 최적화가 어렵습니다.
  • ✅ Redux는 예측 가능한 상태 관리를 위해 단방향 데이터 흐름을 강제합니다.
  • ✅ Zustand, Jotai 같은 현대적 대안들은 더 간단하고 가볍습니다.
  • ✅ React Query, SWR은 서버 상태 관리를 위한 전문 라이브러리입니다.
  • ✅ 프로젝트 규모와 복잡도에 따라 적절한 도구를 선택해야 합니다.

하 · 입문

한 줄로 이해하기

상태관리는 "여러 컴포넌트가 함께 보는 데이터를 어디에 두고 어떻게 갱신할지"의 약속이며, 앱이 커질수록 중요해집니다.

왜 배워야 하나?

실무 사례로 (1) 로그인한 사용자 정보를 헤더·마이페이지·결제창이 모두 알아야 하는 경우, (2) 장바구니 항목 수를 사이트 전역 배지에 동기화, (3) 다크 모드 설정을 페이지 이동 후에도 유지해야 하는 경우 — 모두 상태관리 도구가 해결합니다. 잘못 설계하면 "내 코드는 분명 맞는데 왜 화면이 안 바뀌지?" 같은 디버그 지옥을 겪습니다.

🌱 핵심 용어

  • 로컬 상태: 한 컴포넌트만 쓰는 데이터(useState)
  • 전역 상태: 여러 컴포넌트가 공유하는 데이터
  • Props Drilling: 깊은 자식까지 props로 내려보내는 안티패턴
  • 스토어(Store): 전역 상태를 보관하는 단일 저장소
  • 서버 상태: API에서 받아와 캐싱·동기화하는 데이터

개념 풀어쓰기

상태관리를 도서관 시스템에 비유해봅시다. 책 한 권은 한 사람만 보면 책상 서랍(useState)에 두면 됩니다. 그런데 여러 학생이 같은 책을 봐야 한다면 학교 도서관(Context·Zustand·Redux)에 둬야 합니다. 도서관에는 사서(Reducer)가 있어서 "대출/반납"이라는 정해진 절차로만 책을 빌릴 수 있게 합니다. 그래야 책이 어디 갔는지 추적할 수 있고, 두 사람이 동시에 가져가는 사고를 막을 수 있죠. Redux는 사서가 매우 엄격해서 모든 변경 사유(Action)를 기록합니다. Zustand는 사서가 친절하게 "그냥 set하세요" 식으로 절차를 줄였습니다. React Query는 도서관과 별개로 "도서 검색 기록 캐시"를 자동 관리해, 같은 검색을 또 하지 않도록 도와주는 사서 보조원입니다. 처음엔 useState만으로 시작하고, props가 3단계 이상 내려가면 그때 도구를 도입하세요.

💡 학습 TIP: 작은 앱은 useState/useReducer면 충분합니다. 도구를 미리 도입할수록 코드가 무거워지므로, 필요해질 때 도입하세요.

한 페이지 요약

  • 로컬 상태 우선, 전역은 정말 필요할 때만
  • Context는 데이터 공유 도구이지 성능 도구가 아님
  • 서버 상태는 React Query·SWR로 분리 관리
  • Redux는 큰 팀·복잡한 흐름에 적합
  • Zustand·Jotai는 작은 보일러플레이트로 강력함
중 · 응용

이론 깊이

전역 상태는 크게 세 종류로 분류됩니다. (1) UI 상태(모달 열림 여부, 다크 모드)는 작고 단순하니 Context나 Zustand로 충분합니다. (2) 클라이언트 도메인 상태(장바구니, 폼 위저드 진행 단계)는 변화가 잦고 비즈니스 로직을 포함해 Redux/Zustand가 적합합니다. (3) 서버 상태(상품 목록, 사용자 프로필)는 캐싱·재시도·낙관적 갱신이 필요해 React Query·SWR이 압도적으로 유리합니다. Context는 모든 소비자가 리렌더되기 때문에 자주 바뀌는 값(예: 입력값)을 넣으면 안 됩니다. Zustand는 selector로 구독 부분만 정확히 가져오므로 Context의 성능 문제를 해결합니다. Redux Toolkit은 createSlice로 보일러플레이트를 90% 줄였고, RTK Query를 함께 쓰면 서버 상태까지 일원화됩니다.

핵심 코드 패턴

// Zustand 스토어 + 셀렉터
import { create } from 'zustand';

const useCart = create((set) => ({
  items: [],
  addItem: (p) => set((s) => ({ items: [...s.items, p] })),
  removeItem: (id) => set((s) => ({ items: s.items.filter(i => i.id !== id) })),
}));

// 컴포넌트에서 필요한 부분만 구독
function CartBadge() {
  const count = useCart((s) => s.items.length);
  return <span>🛒 {count}</span>;
}

대표 예제

💻 예제 1: React Query로 서버 상태

useQuery(['posts'], fetchPosts) 한 줄로 캐시·로딩·에러·재검증을 자동 처리하고, useMutation으로 낙관적 업데이트까지 30줄 안에 구현됩니다.

💻 예제 2: Atoms 단위 상태(Jotai)

각 atom은 독립적으로 구독되어 Redux의 슬라이스보다 더 잘게 쪼갤 수 있고, 의존성 그래프를 자동으로 추적해 파생 atom을 만들면 효율적입니다.

실무 응용

실무에서는 (1) Redux Toolkit + RTK Query로 대형 앱 일원화, (2) Zustand + persist 미들웨어로 localStorage 자동 동기화, (3) URL을 진실의 원천(useSearchParams)으로 두어 공유·뒤로가기를 자연스럽게 만들기 등이 자주 쓰입니다.

⚠️ 자주 하는 실수: 모든 상태를 Redux에 몰아넣어 보일러플레이트 폭발, Context에 자주 바뀌는 객체를 그대로 넣어 전체 트리 리렌더, 서버 데이터를 useState로 직접 캐싱하다 stale 데이터 표시.
상 · 심화

전문가 레벨

상태관리 라이브러리 내부 작동을 이해하면 디버깅이 빨라집니다. Redux는 단방향 흐름(Action → Reducer → Store → View) 위에 미들웨어 체인을 둬 사이드 이펙트(thunk·saga)를 분리합니다. Zustand는 단순한 Pub/Sub과 selector + shallow 비교를, Jotai는 atom-deps DAG를, Recoil은 atom과 selector 그래프를 활용합니다. Context API의 핵심 한계는 "구독 단위가 Provider 전체"라는 점이고, useSyncExternalStore(React 18)가 외부 스토어와 React의 동시성 렌더링을 안전하게 결합해주는 공식 다리입니다. 서버 상태 라이브러리는 Stale-While-Revalidate, Request Deduplication, Window Focus Refetch, Optimistic Update 등의 전략을 내장합니다. 또한 CRDT(Yjs, Automerge)는 협업 앱에서 충돌 없는 분산 상태 동기화를 가능케 하는 차세대 패러다임입니다.

고급 패턴

// useSyncExternalStore로 커스텀 외부 스토어를 React에 안전 연결
function createStore(initial) {
  let state = initial;
  const listeners = new Set();
  return {
    getState: () => state,
    setState: (updater) => {
      state = typeof updater === 'function' ? updater(state) : updater;
      listeners.forEach(l => l());
    },
    subscribe: (l) => { listeners.add(l); return () => listeners.delete(l); }
  };
}

const store = createStore({ count: 0 });

function useStore(selector) {
  return useSyncExternalStore(
    store.subscribe,
    () => selector(store.getState()),
    () => selector(store.getState())
  );
}

최신 트렌드

(1) Server Actions + useOptimistic(React 19): 서버와 클라이언트 상태 경계를 흐리고 낙관적 UI를 표준화, (2) Signals 기반 상태관리(Preact Signals, Zustand v5 검토 중): 미세 단위 반응성으로 불필요한 리렌더 제거, (3) Local-First Apps(Replicache, ElectricSQL): 오프라인 우선 + 서버 동기화로 새로운 UX 패러다임.
