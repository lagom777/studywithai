6장. 상태관리

React 애플리케이션의 상태를 효율적으로 관리하는 방법들

6.1 상태란?

상태(State)는 애플리케이션이 어느 순간의 조건을 나타내는 데이터입니다. 시간에 따라 변할 수 있으며, 이 변화가 UI 업데이트를 트리거합니다.

UI 상태 vs 서버 상태

UI 상태

• 모달 열림/닫힘

• 입력 필드 값

• 탭 선택

• 폼 유효성

💡 애플리케이션 메모리에만 존재

서버 상태

• 데이터베이스의 데이터

• 사용자 정보

• 게시물 목록

• 비동기 데이터

💡 백엔드에서 관리하며 캐싱 필요

// UI 상태 예제
function Modal() {
  const [isOpen, setIsOpen] = useState(false);
  // isOpen은 UI 상태
}

// 서버 상태 예제
function UserList() {
  const [users, setUsers] = useState([]);
  // users는 서버에서 가져온 상태
}

6.2 React Context API

Context는 깊은 계층 구조에서 Props Drilling을 피하기 위해 사용합니다. 전역 상태를 여러 컴포넌트에 제공합니다.

Context 사용 패턴

// 1. Context 생성
const ThemeContext = React.createContext();

// 2. Provider로 감싸기
function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <MainContent />
    </ThemeContext.Provider>
  );
}

// 3. 자식 컴포넌트에서 사용
function Button() {
  const { theme } = useContext(ThemeContext);
  return <button style={{ background: theme }}>버튼</button>;
}

Context API의 장단점

✅ 장점

  • Props Drilling 제거
  • 내장 API (추가 라이브러리 불필요)
  • 간단한 전역 상태
  • 학습 곡선 완만

❌ 단점

  • 성능 최적화 어려움
  • DevTools 부족
  • 복잡한 상태 관리 시 코드 증가
  • 미들웨어 미지원

6.3 Redux 패턴

Redux는 예측 가능한 상태 관리를 위한 라이브러리입니다. Action → Reducer → Store → View의 단방향 사이클을 따릅니다.

Redux 데이터 흐름

💡 버튼을 클릭하여 Redux 사이클을 시뮬레이션하세요.

Redux의 3가지 원칙

Single Source of Truth

전체 상태가 하나의 Store에 저장

State is Read-Only

상태 변경은 Action을 통해서만 가능

Changes are Made with Pure Reducers

순수 함수인 Reducer로만 상태 업데이트

Redux 구현 예제

// 1. Action 정의
const INCREMENT = 'INCREMENT';
const increment = () => ({ type: INCREMENT });

// 2. Reducer 작성
function counterReducer(state = 0, action) {
  if (action.type === INCREMENT) {
    return state + 1;
  }
  return state;
}

// 3. Store 생성
const store = createStore(counterReducer);

// 4. 컴포넌트에서 사용
store.dispatch(increment());
const state = store.getState();

6.4 현대적 대안들

Redux 외에도 상태 관리를 위한 더 가볍고 간단한 라이브러리들이 있습니다.

📦 Zustand

  • 미니멀한 API
  • 번들 크기 매우 작음
  • DevTools 지원
  • 미들웨어 지원
const useStore = create(set => ({
  count: 0,
  inc: () => set(s => ({count: s.count+1}))
}))

⚛️ Recoil

  • Facebook 개발
  • 선언적 상태
  • 비동기 지원
  • 세밀한 구독
const countAtom = atom({
  key: 'count',
  default: 0
})

⚙️ Jotai

  • Recoil 영감
  • 더 간단한 API
  • TypeScript 친화적
  • 번들 최소화
const countAtom = atom(0);
const [count, setCount] = useAtom(
  countAtom
)

비교 차트

라이브러리 번들 크기 학습 곡선 기능성
Redux 중간 높음 매우 풍부
Zustand ⭐⭐⭐ 낮음 충분
Recoil 중간 중간 매우 풍부
Jotai ⭐⭐⭐ 낮음 충분

6.5 서버 상태 관리

API 데이터, 데이터베이스 상태 같은 비동기 데이터 관리를 위한 라이브러리들입니다.

React Query

  • 자동 캐싱
  • 백그라운드 동기화
  • 재시도 로직
  • 상태 머신

TanStack Query

SWR

  • 간단한 API
  • 자동 리페칭
  • 초점 재개 시 갱신
  • 번들 매우 작음

Vercel 개발

React Query 기본 사용법

import { useQuery } from 'react-query';

function UserList() {
  const { data, isLoading, error } = useQuery('users',
    () => fetch('/api/users').then(r => r.json())
  );
  
  if (isLoading) return <div>로딩...</div>;
  if (error) return <div>에러: {error.message}</div>;
  return <div>{data.map(u => <p key={u.id}>{u.name}</p>)}</div>;
}

상태 관리 방식 비교

Context API vs Redux 데모

같은 카운터를 두 가지 방식으로 구현했을 때의 차이를 확인하세요.

Context API

더 간단하지만 성능 최적화 어려움

Count: 0
const [count, setCount] = useState(0)

Redux

더 복잡하지만 성능 최적화 용이

Count: 0
dispatch({type: 'INCREMENT'})

상태 관리 선택 가이드

어떤 것을 선택할까?

  • 📍 useState + Props: 작은 프로젝트, 간단한 상태
  • 🌍 Context API: 중간 규모, 테마/언어 같은 전역 설정
  • 🎯 Zustand/Jotai: 현대적이고 간단한 상태 관리
  • ⚙️ Redux: 복잡한 상태 로직, 시간여행 디버깅 필요
  • 🔄 React Query: 서버 상태, 데이터 페칭
  • ♾️ 조합: UI 상태(Zustand) + 서버 상태(React Query)

핵심 요약

  • ✅ 상태는 UI 상태와 서버 상태로 나뉘며 다르게 관리합니다.
  • ✅ Context API는 Props Drilling을 제거하지만 성능 최적화가 어렵습니다.
  • ✅ Redux는 예측 가능한 상태 관리를 위해 단방향 데이터 흐름을 강제합니다.
  • ✅ Zustand, Jotai 같은 현대적 대안들은 더 간단하고 가볍습니다.
  • ✅ React Query, SWR은 서버 상태 관리를 위한 전문 라이브러리입니다.
  • ✅ 프로젝트 규모와 복잡도에 따라 적절한 도구를 선택해야 합니다.
